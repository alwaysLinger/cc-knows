# Scaling Managed Agents: Decoupling the Brain from the Hands — 详细摘要

## 引言：线束假设会过时

文章开篇指出一个反复出现的主题：线束（harness）编码了关于 Claude 自身无法完成哪些事的假设，而这些假设需要不断质疑，因为它们会随模型改进而过时。例如，先前研究发现 Claude Sonnet 4.5 在感知上下文限制逼近时会过早结束任务（"上下文焦虑"），为此在线束中加入了上下文重置机制。但同一线束用于 Claude Opus 4.5 时，该行为已消失，重置机制成了冗余代码。

因此，Managed Agents 作为托管服务被设计出来，通过少量**旨在比任何具体实现更长寿命的接口**来运行长期智能体任务。

## 不要养"宠物"

早期方案将所有智能体组件（会话、线束、沙箱）放入单一容器。优点是文件编辑直接走系统调用，无需设计服务边界。但代价是容器变成了不可丢失的"宠物"（pets vs. cattle 类比中的 pet）：容器故障意味着会话丢失，容器无响应需要人工修复。调试时只能通过 WebSocket 事件流观察，无法定位故障根源；而进入容器调试又涉及用户数据，实际上丧失了调试能力。

此外，线束假设所有资源都在容器内部，当客户要求连接其 VPC 时，要么网络对等连接，要么在客户环境运行线束——耦合假设变成了扩展障碍。

## 解耦大脑与双手

解决方案是将"大脑"（Claude 及其线束）与"双手"（沙箱和工具）以及"会话"（事件日志）解耦，各自成为最少假设的接口，可独立故障或替换。

- **线束离开容器**：线束通过 `execute(name, input) → string` 调用容器，容器变为"牲畜"。容器死亡时，线束将其捕获为工具调用错误返回给 Claude；若 Claude 决定重试，可通过 `provision({resources})` 重建新容器。
- **线束自身也是"牲畜"**：由于会话日志位于线束外部，线束崩溃后可通过 `wake(sessionId)` 重启，用 `getSession(id)` 获取事件日志并从上次事件恢复。线束在循环中通过 `emitEvent(id, event)` 写入会话以保持事件持久记录。
- **安全边界**：耦合设计中，Claude 生成的不可信代码与凭据在同一容器运行，提示注入只需说服 Claude 读取自身环境变量即可获取令牌。结构性修复是确保令牌永远无法从沙箱触及。具体采用两种模式：Git 令牌在沙箱初始化时用于克隆仓库并写入本地 git remote，此后 `push`/`pull` 无需令牌；自定义工具通过 MCP 代理访问，代理从安全金库获取凭据，线束永远不接触凭据。

## 会话不是 Claude 的上下文窗口

长期任务常超出上下文窗口长度，标准应对方法都涉及保留/丢弃内容的不可逆决策。压缩（compaction）让 Claude 保存上下文摘要，记忆工具让 Claude 将上下文写入文件，上下文裁剪选择性移除旧工具结果或思考块。但不可逆决策可能导致失败——很难预知未来轮次需要哪些 token。

Managed Agents 中，会话作为独立于上下文窗口的持久上下文对象。`getEvents()` 接口允许"大脑"选择性地检索事件流的位置切片：从上次停止处继续、回退几步查看特定时刻的前因、重读特定动作前的上下文。获取的事件可在传给 Claude 前由线束进行变换（包括上下文组织和上下文工程），将会话中的可恢复上下文存储与线束中的任意上下文管理分离——因为无法预测未来模型需要何种上下文工程。

## 多个大脑，多双手

- **多个大脑**：解耦解决了客户将 Claude 连接到自身 VPC 的需求——线束不再假设资源在身旁。性能方面，旧架构中每个大脑需要一个容器，推理开始前必须完成容器配置（克隆仓库、启动进程、获取事件），即使不使用沙箱也需等待。解耦后，容器仅在需要时通过工具调用配置，p50 TTFT 降低约 60%，p95 降低超过 90%。扩展到多个大脑只需启动多个无状态线束并按需连接双手。

- **多双手**：每个手都是 `execute(name, input) → string` 接口——输入名称和参数，返回字符串。该接口支持任何自定义工具、MCP 服务器和自有工具。线束不知道沙箱是容器、手机还是 Pokémon 模拟器。由于没有手与任何大脑耦合，大脑之间可以传递手。

## 结论

文章回到核心问题：如何为"尚未想象的程序"设计系统。操作系统通过将硬件虚拟化为足够通用的抽象（进程、文件）延续了数十年。Managed Agents 以同样精神设计——对 Claude 未来需要的具体线束无偏见，而是提供通用接口容纳多种线束。Meta-harness 设计意味着对接口形状有主见（Claude 需要操作状态、执行计算、扩展到多个大脑和双手），但对大脑或双手的数量和位置不做假设。
