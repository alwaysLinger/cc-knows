# 扩展托管智能体：将大脑与双手解耦

发布于 2026 年 4 月 8 日

线束（harness）编码的假设会随着模型改进而过时。托管智能体（Managed Agents）——我们面向长期智能体工作的托管服务——围绕那些在线束变化时仍能保持稳定的接口而构建。

_通过我们的[文档](https://platform.claude.com/docs/en/managed-agents/overview)开始使用 Claude Managed Agents。_

工程博客上一个反复出现的主题是如何[构建有效的智能体](https://www.anthropic.com/engineering/building-effective-agents)以及如何为[长期运行的工作](https://www.anthropic.com/engineering/harness-design-long-running-apps)[设计线束](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents)。这些工作的一个共同线索是：线束编码了关于 Claude 自身无法完成哪些事情的假设。然而，这些假设需要经常被质疑，因为它们可能会随着模型的改进而[过时](http://www.incompleteideas.net/IncIdeas/BitterLesson.html)。

仅举一例，在先前的工作中[我们发现](https://www.anthropic.com/engineering/harness-design-long-running-apps) Claude Sonnet 4.5 在感知到上下文限制逼近时会过早结束任务——这种行为有时被称为"上下文焦虑"（context anxiety）。我们通过在线束中添加上下文重置来解决这个问题。但当我们在线束上使用 Claude Opus 4.5 时，发现这种行为已经消失了。重置机制变成了死代码。

我们预期线束会持续演进。因此我们构建了 Managed Agents：Claude 平台中的一个托管服务，通过少量旨在比任何具体实现（包括我们当前运行的实现）更长寿命的接口，代表你运行长期智能体。

构建 Managed Agents 意味着解决计算领域的一个老问题：如何为"[尚未想象的程序](http://www.catb.org/esr/writings/taoup/html/ch03s01.html)"设计系统。几十年前，操作系统通过将硬件虚拟化为足够通用的抽象——*进程（process）、文件（file）*——解决了这个问题，这些抽象足够通用以容纳当时还不存在的程序。抽象比硬件更持久。`read()` 命令不关心它访问的是 1970 年代的磁盘组还是现代的 SSD。上层的抽象保持稳定，而下层的实现可以自由变化。

Managed Agents 遵循同样的模式。我们将智能体的组件虚拟化：会话（session，即所有已发生事件的仅追加日志）、线束（harness，即调用 Claude 并将 Claude 的工具调用路由到相关基础设施的循环）和沙箱（sandbox，即 Claude 可以运行代码和编辑文件的执行环境）。这使得每个组件的实现可以被替换而不干扰其他组件。我们对这些接口的形状有主见，但对它们背后运行的内容没有偏见。

## 不要养"宠物"

我们最初将所有智能体组件放入单一容器，这意味着会话、智能体线束和沙箱共享一个环境。这种方法有其好处，包括文件编辑是直接的系统调用，而且不需要设计服务边界。

但是，将所有东西耦合到一个容器中，我们遇到了一个古老的基础设施问题：我们养了一只[_宠物_](https://cloudscaling.com/blog/cloud-computing/the-history-of-pets-vs-cattle/)。在宠物与牲畜（pets vs. cattle）的类比中，宠物是一个有名字的、精心照料的个体，你承受不起失去它，而牲畜是可以互换的。在我们的案例中，服务器成为了那个宠物；如果容器失败，会话就丢失了。如果容器无响应，我们不得不悉心照料它恢复健康。

照料容器意味着调试无响应的卡住会话。我们唯一的窗口是 WebSocket 事件流，但它无法告诉我们故障*在哪里*出现，这意味着线束中的 bug、事件流中的丢包或容器下线看起来都一样。要弄清楚出了什么问题，工程师必须进入容器内部打开一个 shell，但由于该容器通常也持有用户数据，这种方法本质上意味着我们缺乏调试能力。

第二个问题是，线束假设 Claude 处理的一切都与它在同一个容器中。当客户要求我们将 Claude 连接到他们的虚拟私有云（VPC）时，他们要么必须将其网络与我们对等连接，要么在他们自己的环境中运行我们的线束。嵌入线束中的假设在我们想将其连接到不同基础设施时变成了问题。

## 将大脑与双手解耦

我们得出的解决方案是将"大脑"（Claude 及其线束）与"双手"（沙箱和执行操作的工具）以及"会话"（会话事件日志）解耦。每一方都成为对另一方几乎不做假设的接口，每一方都可以独立故障或被替换。

**线束离开容器。** 将大脑与双手解耦意味着线束不再存在于容器内部。它调用容器的方式与调用任何其他工具相同：`execute(name, input) → string`。容器变成了牲畜。如果容器死亡，线束将失败捕获为工具调用错误并传回给 Claude。如果 Claude 决定重试，可以用标准配方重新初始化一个新容器：`provision({resources})`。我们不再需要悉心照料失败的容器恢复健康。

**从线束故障中恢复。** 线束本身也变成了牲畜。因为会话日志位于线束外部，线束中不需要有任何东西在崩溃后幸存。当一个线束失败时，可以启动一个新的线束：通过 `wake(sessionId)` 唤醒，使用 `getSession(id)` 取回事件日志，并从最后一个事件恢复。在智能体循环期间，线束通过 `emitEvent(id, event)` 写入会话，以保持事件的持久记录。

**安全边界。** 在耦合设计中，Claude 生成的任何不可信代码都与凭据运行在同一个容器中——因此提示注入（prompt injection）只需说服 Claude 读取自己的环境变量即可。一旦攻击者获取了这些令牌（token），他们就可以生成新的、不受限制的会话并将工作委托给它们。窄范围授权是明显的缓解措施，但这编码了一个关于 Claude 在有限令牌下无法做到什么的假设——而 Claude 正变得越来越聪明。结构性修复是确保令牌永远无法从 Claude 生成代码运行的沙箱中触及。

我们使用了两种模式来确保这一点。凭据可以与资源捆绑，或保存在沙箱外的金库（vault）中。对于 Git，我们使用每个仓库的访问令牌在沙箱初始化期间克隆仓库，并将其配置到本地 git remote 中。Git 的 `push` 和 `pull` 可以从沙箱内部工作，而智能体永远不需要处理令牌本身。对于自定义工具，我们支持 MCP（Model Context Protocol）并将 OAuth 令牌存储在安全金库中。Claude 通过专用代理调用 MCP 工具；该代理接收与会话关联的令牌，然后可以从金库获取相应的凭据并对外部服务发起调用。线束永远不会被告知任何凭据。

## 会话不是 Claude 的上下文窗口

长期任务通常超出 Claude 上下文窗口的长度，而解决此问题的标准方法都涉及关于保留什么内容的不可逆决策。我们在关于上下文工程（context engineering）的[先前工作](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)中探讨了这些技术。例如，压缩（compaction）让 Claude 保存其上下文窗口的摘要，记忆工具（memory tool）让 Claude 将上下文写入文件，实现跨会话学习。这可以与上下文裁剪（context trimming）配合使用，后者选择性移除 token，如旧的工具结果或思考块。

但选择性保留或丢弃上下文的不可逆决策可能导致失败。很难知道未来的轮次将需要哪些 token。如果消息被压缩步骤变换，线束会从 Claude 的上下文窗口中移除被压缩的消息，而这些消息只有在被存储的情况下才可恢复。先前工作[已探索](https://arxiv.org/pdf/2512.24601)了通过将上下文存储为*上下文窗口之外*的对象来解决此问题的方法。例如，上下文可以是 REPL 中的一个对象，LLM 通过编写代码来过滤或切片它，从而程序化地访问上下文。

在 Managed Agents 中，会话提供了同样的好处，作为存在于 Claude 上下文窗口之外的上下文对象。但上下文不是存储在沙箱或 REPL 中，而是持久存储在会话日志中。接口 `getEvents()` 允许大脑通过选择事件流的位置切片来查询上下文。该接口可以灵活使用，允许大脑从上次停止阅读的地方继续、回退到特定时刻之前的几个事件以查看前因，或重读特定动作之前的上下文。

任何获取的事件也可以在传递给 Claude 的上下文窗口之前在线束中进行变换。这些变换可以是在线束中编码的任何内容，包括上下文组织以实现高提示缓存命中率（prompt cache hit rate）和上下文工程。我们将会话中可恢复的上下文存储与线束中任意的上下文管理分离，因为我们无法预测未来模型将需要什么样的特定上下文工程。接口将上下文管理推入线束，只保证会话是持久的且可供查询。

## 多个大脑，多双手

**多个大脑。** 将大脑与双手解耦解决了我们最早的客户投诉之一。当团队希望 Claude 在其自己的 VPC 中的资源上工作时，唯一的路径是将其网络与我们对等连接，因为容纳线束的容器假设每个资源都在它旁边。一旦线束不再位于容器中，该假设就不复存在了。同样的变化还带来了性能收益。当我们最初将大脑放入容器时，意味着多个大脑需要同样多的容器。对于每个大脑，在容器配置完成之前无法进行推理；每个会话都要支付完整的容器设置成本。每个会话——即使是从不触碰沙箱的会话——都必须克隆仓库、启动进程、从我们的服务器获取待处理事件。

那种死时间用首 token 时间（TTFT，Time-To-First-Token）来表达，它衡量一个会话在接受工作到产生第一个响应 token 之间等待多长时间。TTFT 是用户最敏锐*感受到*的延迟。

将大脑与双手解耦意味着容器仅在大脑需要时通过工具调用 `(execute(name, input) → string)` 来配置。因此，不需要立即使用容器的会话不必等待。推理可以在编排层从会话日志中拉取待处理事件后立即开始。使用这种架构，我们的 p50 TTFT 降低了约 60%，p95 降低了超过 90%。扩展到多个大脑只需启动多个无状态线束，并仅在需要时连接双手。

**多双手。** 我们还希望将每个大脑连接到多双手的能力。在实践中，这意味着 Claude 必须对多个执行环境进行推理，并决定将工作发送到哪里——这是比在单个 shell 中操作更难的认知任务。我们最初将大脑放在单个容器中，是因为早期模型没有能力做到这一点。随着智能的扩展，单一容器反而变成了限制：当该容器失败时，我们失去了大脑正在触及的每一双手的状态。

将大脑与双手解耦使每只手成为一个工具，`execute(name, input) → string`：输入名称和参数，返回一个字符串。该接口支持任何自定义工具、任何 MCP 服务器以及我们自己的工具。线束不知道沙箱是容器、手机还是 Pokémon 模拟器。而且由于没有手与任何大脑耦合，大脑之间可以互相传递手。

## 结论

我们面临的挑战是一个古老的挑战：如何为"尚未想象的程序"设计系统。操作系统通过将硬件虚拟化为足够通用的抽象延续了数十年，这些抽象足以容纳当时还不存在的程序。借助 Managed Agents，我们旨在设计一个容纳未来线束、沙箱或 Claude 周围其他组件的系统。

Managed Agents 是同样精神下的元线束（meta-harness），对 Claude 未来需要的*具体*线束没有偏见。相反，它是一个具有通用接口的系统，允许许多不同的线束。例如，Claude Code 是一个优秀的线束，我们在各种任务中广泛使用。我们还展示了特定任务的智能体线束在狭窄领域中的卓越表现。Managed Agents 可以容纳其中任何一种，随时间匹配 Claude 的智能。

元线束设计意味着对 Claude 周围的接口有主见：我们预期 Claude 将需要操作状态（会话）和执行计算（沙箱）的能力。我们还预期 Claude 将需要扩展到多个大脑和多双手的能力。我们设计接口以便这些可以在长时间跨度上可靠且安全地运行。但我们对 Claude 将需要的大脑或双手的数量和位置不做任何假设。
