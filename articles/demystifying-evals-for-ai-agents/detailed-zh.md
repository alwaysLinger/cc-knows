# AI Agent 评估解密 — 详细摘要

**原文**: [Demystifying Evals for AI Agents](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents)（Anthropic, 2026-01-09）

## 引言

好的评估帮助团队更有信心地交付 AI Agent。没有评估，团队容易陷入被动循环——只在生产环境中发现问题，修一个又坏另一个。评估让问题和行为变化在影响用户前变得可见，其价值在 Agent 生命周期中复利增长。Agent 的自主性、智能性和灵活性使其难以评估，本文基于 Anthropic 内部实践和前沿客户经验，总结出跨 Agent 架构和用例的有效评估策略。

## 评估的结构

评估（eval）是对 AI 系统的测试：给定输入，用评分逻辑衡量输出的成功与否。本文聚焦开发阶段可自动运行的评估。单轮评估较简单（提示-响应-评分），多轮评估更复杂——Agent 执行"Agent 循环"（工具调用与推理），在环境中修改状态并动态适应。Agent 评估的特殊挑战在于：错误会跨轮次传播叠加，且前沿模型可能找到超越静态评估的创造性方案（如 Opus 4.5 在 τ2-bench 中发现策略漏洞，虽"未通过"评估，却给出了更优解）。

关键定义：任务（task）、试验（trial）、评分器（grader）、轨迹（transcript）、结果（outcome）、评估框架（evaluation harness）、Agent 框架（agent harness/scaffold）、评估套件（evaluation suite）。

## 为什么要构建评估？

团队初期靠手动测试和直觉可以走很远，但突破点之后，没有评估就会"盲目飞行"——用户投诉后才发现问题，手动复现、修复、祈祷不回归。Claude Code 的演进是典型案例：从快速反馈迭代开始，逐步增加简洁性和文件编辑的评估，再到复杂行为（如过度工程）的评估。评估还影响模型升级速度——有评估的团队可在数天内完成新模型验证和提示调优，没有的则需数周。此外，评估还免费提供基线和回归测试，可追踪延迟、token 用量、单位成本和错误率，并能成为产品与研究团队间最高带宽的沟通渠道。

## 如何评估 AI Agent

### 三类评分器

- **代码评分器**：字符串匹配、二值测试、静态分析、结果验证等。优点：快速、便宜、客观、可复现；缺点：对合理变化脆弱、缺乏细微度。
- **模型评分器**：评分规则、自然语言断言、成对比较、多评判共识等。优点：灵活、可扩展、捕获细微差别；缺点：非确定性、比代码评分器贵、需人工校准。
- **人工评分器**：专家评审、众包判断、抽查采样等。优点：金标准、匹配专家判断；缺点：昂贵、缓慢、需大规模专家。

评分方式可加权（组合分数达阈值）、二值（全部通过）或混合。

### 能力评估 vs 回归评估

能力评估（"Agent 擅长什么"）从低通过率起步；回归评估（"Agent 还能做以前能做的吗"）应接近 100%。高通过率的能力评估可"毕业"为回归评估套件。

### 编码 Agent 评估

编码 Agent 导航代码库编写、测试和调试代码。确定性评分器是天然选择——代码能否运行、测试是否通过？常用基准包括 SWE-bench Verified 和 Terminal-Bench。除结果测试外，还可用启发式规则评估代码质量、用模型评分器评估工具调用行为。文中给出了 YAML 格式的评估设计示例，涵盖确定性测试、LLM 评分规则、静态分析、状态检查、工具调用验证和追踪指标。

### 对话 Agent 评估

对话 Agent 在支持、销售、辅导等领域交互，需维护状态、使用工具、中途执行操作。成功是多维的：工单是否解决、是否在 10 轮内完成、语气是否恰当。常用基准包括 τ-Bench 和 τ2-Bench。通常需要第二个 LLM 模拟用户。文中示例展示了评分规则断言、状态检查、工具调用验证和轮次限制。

### 研究 Agent 评估

研究 Agent 收集、综合和分析信息，产出答案或报告。其独特挑战：专家可能意见不一、真值随时间变化、更长输出意味着更多出错空间。BrowseComp 基准测试 Agent 在开放网络中寻找特定信息的能力。策略是组合评分器：依据性检查验证主张有来源支撑，覆盖度检查定义关键事实，来源质量检查确认权威性。LLM 评分规则需频繁与人工专家判断校准。

### 计算机使用 Agent 评估

计算机使用 Agent 通过截图、鼠标点击、键盘输入等与软件交互。评估需在真实或沙盒环境中运行并检查是否达成目标。基准包括 WebArena（浏览器任务）和 OSWorld（完整操作系统控制）。浏览器使用 Agent 需平衡 token 效率与延迟。

### 非确定性的处理

Agent 行为在运行间变化。两个关键指标：**pass@k** 衡量 k 次尝试中至少一次成功的概率（适用于一次成功即可的工具）；**pass^k** 衡量 k 次试验全部成功的概率（适用于一致性至关重要的 Agent）。

## 从零到一：评估路线图

1. **尽早收集任务**：20-50 个来自真实失败的简单任务是良好起点。
2. **从已有手动测试转化**：将手动检查、Bug 追踪项、支持队列问题转为测试用例。
3. **编写无歧义任务和参考解**：两位领域专家应独立得出相同结论。
4. **构建平衡的问题集**：同时测试行为应发生和不应发生的情况。
5. **构建稳定的评估框架**：每次试验从干净环境开始。
6. **审慎设计评分器**：尽可能用确定性评分器，必要时用 LLM 评分器，谨慎使用人工评分器；为多组件任务设计部分得分。
7. **检查轨迹**：阅读轨迹验证评估是否测量了真正重要的东西。
8. **监控能力评估饱和**：100% 通过率的评估只追踪回归，不提供改进信号。
9. **长期维护评估集健康**：建立专职评估团队负责核心基础设施，领域专家和产品团队贡献任务。

## 评估与其他方法的配合

自动化评估是理解 Agent 性能的一种方式。完整图景还包括生产监控、用户反馈、A/B 测试、手动轨迹审查和系统性人工研究。不同方法对应开发不同阶段：评估用于上线前，生产监控用于上线后，人工审查用于校准 LLM 评分器。

## 结论

没有评估的团队深陷被动循环；投入评估的团队将失败转化为测试用例、加速开发。评估价值复利增长，前提是将其视为核心组件。不同 Agent 类型的模式各异，但基础原则不变：尽早开始、任务真实、标准明确、评分器审慎设计、难度足够、持续迭代、阅读轨迹。
