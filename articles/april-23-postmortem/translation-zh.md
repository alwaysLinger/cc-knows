# 近期 Claude Code 质量报告更新

发布于 2026年4月23日

我们将近期关于 Claude Code 质量问题的报告追溯至三个独立的变更。以下是发生的情况以及我们将做出的改变。

过去一个月，我们一直在调查用户反馈的 Claude 回答质量下降的问题。我们将这些报告追溯至三个分别影响 Claude Code、Claude Agent SDK 和 Claude Cowork 的独立变更。API 未受影响。

所有三个问题现已于4月20日修复（v2.1.116）。

在这篇文章中，我们将解释我们发现了什么、修复了什么，以及我们将采取哪些不同的做法，以确保类似问题发生的可能性大大降低。

我们非常重视关于质量退化的报告。我们从不会有意降低我们的模型质量，并且我们能够立即确认我们的 API 和推理层未受影响。

经过调查，我们确定了三个不同的问题：

1. 3月4日，我们将 Claude Code 的默认推理努力（reasoning effort）从 `high` 改为 `medium`，以减少部分用户在 `high` 模式下遇到的极长延迟——长到足以让 UI 看起来像冻结了一样。这是一个错误的权衡。在用户告诉我们他们更倾向于默认使用更高的智能，并为简单任务选择更低的努力等级后，我们于4月7日回退了这一变更。这影响了 Sonnet 4.6 和 Opus 4.6。
2. 3月26日，我们发布了一项变更，清除闲置超过一小时的会话中 Claude 的旧思考内容，以减少用户恢复这些会话时的延迟。一个 bug 导致这一操作在会话剩余的每一轮都持续执行，而不是仅执行一次，这使得 Claude 看起来健忘且重复。我们于4月10日修复了此问题。这影响了 Sonnet 4.6 和 Opus 4.6。
3. 4月16日，我们添加了一条系统提示词（system prompt）指令以减少冗长。与其他提示词变更叠加后，它损害了编码质量，并于4月20日被回退。这影响了 Sonnet 4.6、Opus 4.6 和 Opus 4.7。

由于每个变更在不同的时间表上影响了不同的流量片段，叠加效应看起来像是广泛的、不一致的质量退化。虽然我们在3月初开始调查报告，但起初很难将它们与用户反馈的正常波动区分开来，而且我们的内部使用和评估最初都无法复现已发现的问题。

这不是用户应该从 Claude Code 获得的体验。自4月23日起，我们将为所有订阅用户重置用量限额。

## Claude Code 默认推理努力等级的变更

当我们在2月于 Claude Code 中发布 Opus 4.6 时，我们将默认推理努力设为 `high`。

此后不久，我们收到用户反馈，Claude Opus 4.6 在高努力模式下偶尔会思考过久，导致 UI 看起来像冻结了一样，并给这些用户带来不成比例的延迟和 token 用量。

一般来说，模型思考时间越长，输出越好。努力等级（Effort levels）是 Claude Code 让用户设置这一权衡的方式——更多思考与更低延迟和更少用量限额消耗之间的权衡。在校准模型的努力等级时，我们考虑这一权衡，以便在测试时计算（test-time-compute）曲线上选择为人们提供最佳选项范围的点。然后在产品层面，我们选择将曲线上的哪个点设为默认值，该值就是我们发送给 Messages API 的 effort 参数；然后我们通过 `/effort` 提供其他选项。

![Image](https://www-cdn.anthropic.com/images/4zrzovbb/website/de3bcf9733b61f57234d8c45e663b1bd48677ea1-3840x2160.png)

在我们的内部评估和测试中，medium 努力在大多数任务上实现了略低的智能，但延迟显著减少。它也没有出现高努力模式下偶发的极长尾延迟问题，并且有助于最大化用户的用量限额。因此，我们推出了将 medium 设为默认努力等级的变更，并通过产品内对话框解释了理由。

![Image](https://www-cdn.anthropic.com/images/4zrzovbb/website/459b2a8a0baa88937eebcbe4566dde4d6cc7f185-3794x2260.png)

上线后不久，用户开始报告 Claude Code 感觉不那么智能了。我们进行了多次设计迭代，使当前努力等级设置更加清晰，以提醒用户他们可以更改默认值（启动时的通知、内联努力选择器，以及恢复 ultrathink），但大多数用户仍保持 medium 努力默认值。

在听取更多客户的反馈后，我们于4月7日撤销了这一决定。所有用户现在默认使用 Opus 4.7 的 `xhigh` 努力，以及其他所有模型的 `high` 努力。

## 一个丢失先前推理的缓存优化

当 Claude 推理完成一项任务时，该推理通常会保留在对话历史中，以便在后续每一轮中，Claude 都能看到它为什么做出了那些编辑和工具调用。

3月26日，我们发布了一项旨在提高此功能效率的改进。我们使用提示缓存（prompt caching）使连续的 API 调用对用户来说更便宜、更快速。Claude 在发出 API 请求时将输入 token 写入缓存，然后在一段时间不活动后，提示从缓存中被驱逐，为其他提示腾出空间。缓存利用率是我们精心管理的事项（更多关于我们的[方法](https://claude.com/blog/lessons-from-building-claude-code-prompt-caching-is-everything)）。

设计本应很简单：如果会话已闲置超过一小时，我们可以通过清除旧的思考部分来降低用户恢复该会话的成本。由于请求无论如何都会是缓存未命中（cache miss），我们可以从请求中修剪不必要的消息，以减少发送到 API 的未缓存 token 数量。之后我们将恢复发送完整的推理历史。为此，我们使用了 `clear_thinking_20251015` API 头以及 `keep:1`。

实现中存在一个 bug。它没有只清除一次思考历史，而是在会话剩余的每一轮都清除它。一旦会话跨越了一次空闲阈值，该进程剩余的每个请求都告诉 API 只保留最近的推理块并丢弃之前的所有内容。这会产生复合效应：如果你在 Claude 使用工具的过程中发送了一条后续消息，这会在损坏的标志下开始一个新轮次，因此即使是当前轮次的推理也被丢弃了。Claude 会继续执行，但越来越不记得它为什么选择这样做。这表现为人们报告的健忘、重复和奇怪的工具选择。

由于这会持续从后续请求中丢弃思考块，这些请求也导致了缓存未命中。我们认为这是导致用量限额消耗速度超出预期的独立报告的原因。

![Image](https://www-cdn.anthropic.com/images/4zrzovbb/website/332d9c487bb73c8078686068dcbe1b616720a8dd-3016x1198.png)

两个不相关的实验使我们起初难以复现该问题：一个仅限内部的服务端消息队列实验；以及一个在思考显示方式上的无关变更，抑制了该 bug 在大多数 CLI 会话中的表现，因此即使在外部构建测试中我们也没有发现它。

这个 bug 处于 Claude Code 的上下文管理、Anthropic API 和扩展思考（extended thinking）的交汇点。它引入的变更通过了多轮人工和自动化代码审查，以及单元测试、端到端测试、自动化验证和内部试用（dogfooding）。再加上这仅发生在一个边缘情况（陈旧会话）中以及复现问题的困难性，我们花了超过一周的时间才发现并确认根本原因。

作为调查的一部分，我们使用 Opus 4.7 对有问题的拉取请求进行了回溯 [Code Review](https://code.claude.com/docs/en/code-review)。当提供了收集完整上下文所需的代码仓库时，Opus 4.7 发现了这个 bug，而 Opus 4.6 没有。为防止此类情况再次发生，我们正在落地对代码审查的额外仓库上下文支持。

我们于4月10日在 v2.1.101 中修复了此 bug。

## 一个减少冗长的系统提示词变更

我们最新的模型 Claude Opus 4.7 相较于前代有一个显著的行为特点：正如我们在[发布时](https://www.anthropic.com/news/claude-opus-4-7)所写，它往往相当冗长。这使得它在难题上更聪明，但也产生了更多的输出 token。

在发布 Opus 4.7 前几周，我们开始调优 Claude Code 以做准备。每个模型的行为略有不同，我们在每次发布前都会花时间为其优化工具链（harness）和产品。

我们有许多减少冗长的工具：模型训练、提示词，以及改进产品中的思考用户体验。最终我们使用了所有这些方法，但对系统提示词的一个添加对 Claude Code 的智能产生了过大的影响：

> "长度限制：工具调用间的文本保持在25词以内。最终回复保持在100词以内，除非任务需要更多细节。"

经过数周的内部测试以及我们运行的评估集中没有回归，我们对这一变更感到有信心，并于4月16日与 Opus 4.7 一同发布。

作为此次调查的一部分，我们使用更广泛的评估集进行了更多消融实验（ablation，即从系统提示词中移除行以理解每行的影响）。其中一项评估显示 Opus 4.6 和 4.7 均下降了3%。我们立即在4月20日的版本中回退了该提示词。

## 展望未来

我们将采取几项不同的做法来避免这些问题：我们将确保更大比例的内部员工使用与公众完全相同的 Claude Code 构建（而非用于测试新功能的版本）；我们将改进我们在内部使用的 [Code Review](https://code.claude.com/docs/en/code-review) 工具，并将这个改进版本发布给客户。

我们还在系统提示词变更上增加了更严格的管控。我们将在每次 Claude Code 的系统提示词变更时运行针对各模型的完整评估套件，持续进行消融实验以理解每行的影响，并且我们构建了新工具使提示词变更更容易审查和审计。我们还在 CLAUDE.md 中增加了指引，确保模型特定的变更仅对目标模型生效。对于任何可能牺牲智能的变更，我们将增加缓冲期（soak period）、更广泛的评估套件和渐进式发布，以便更早发现问题。

我们最近在 X 上创建了 @ClaudeDevs，为我们提供空间来深入解释产品决策及其背后的理由。我们将在 GitHub 的集中讨论帖中分享同样的更新。

最后，我们想感谢我们的用户：那些使用 `/feedback` 命令与我们分享问题（或在网上发布具体、可复现示例）的人，才是最终使我们能够识别和修复这些问题的人。今天我们正在为所有订阅用户重置用量限额。

我们对您的反馈和耐心深表感激。
