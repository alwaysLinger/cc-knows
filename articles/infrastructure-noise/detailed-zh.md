# 量化智能体编程评估中的基础设施噪声

## 详细摘要

### 背景

智能体编程基准测试（如 SWE-bench、Terminal-Bench）常用于比较前沿模型的软件工程能力，排行榜顶位之间往往仅差几个百分点。然而，研究发现基础设施配置本身就能产生超过这些差距的差异。在内部实验中，Terminal-Bench 2.0 上资源最充足与最紧张配置之间的差距达 6 个百分点（p < 0.01）。

与静态基准测试不同，智能体编程评估给予模型一个完整环境，模型在其中编写程序、运行测试、安装依赖并多次迭代。运行时不再是被动容器，而是问题解决过程的有机组成部分。资源预算和时间限制不同的两个智能体，实际上并非在参加同一场考试。

### 问题发现

研究团队在 Google Kubernetes Engine（GKE）集群上运行 Terminal-Bench 2.0 时，发现得分与官方排行榜不一致，基础设施错误率异常高：多达 6% 的任务因 Pod 错误而失败，且多数与模型解题能力无关。

根本原因在于资源执行方式。Kubernetes 将每任务资源规格同时作为保证分配和硬性上限，容器一旦超出即被终止。容器运行时实际通过两个独立参数执行资源限制：保证分配（预留资源）和硬性上限（终止阈值）。当两者设为相同值时，瞬态内存峰值就会 OOM 终止本可成功的容器。Terminal-Bench 排行榜使用的是更宽松的沙箱方案，允许临时超额分配而不终止容器。

### 实验设计

为量化脚手架效应，团队在六种资源配置下运行 Terminal-Bench 2.0，从严格执行每任务规格（1x）到完全无上限，其他条件完全一致（同一 Claude 模型、同一测试框架、同一任务集）。

### 关键发现

**1x 到 3x 区间**：成功率随资源余量增加而提升，主要由基础设施错误率单调下降驱动（从 5.8% 降至 2.1%，p < 0.001）。但成功率波动在噪声范围内（p=0.40），说明 1x 下崩溃的任务大多本就会失败。

**3x 到无上限区间**：趋势发生变化——基础设施错误仅额外下降 1.6 个百分点，但成功率跃升近 4 个百分点。额外资源使智能体能尝试需要大量分配的策略，如拉取大型依赖、生成高开销子进程、运行内存密集型测试套件。无上限时总提升达 +6 个百分点（p < 0.01）。`rstan-to-pystan` 和 `compile-compcert` 等任务在获得内存余量时成功率显著改善。

### 对测量本质的影响

约 3x 规格以内，额外资源修复的是基础设施可靠性问题；超过 3x 后，额外资源开始主动帮助智能体解决此前无法解决的问题。紧限制无意中奖励高效策略，宽限制则更宽容，奖励能充分利用资源的智能体。两者都是合理的测试对象，但合并为单一得分而不标明资源配置，使差异和现实泛化能力难以解释。

以 `bn-fit-modify` 任务为例：该任务需要贝叶斯网络拟合，部分模型首先安装标准 Python 数据科学栈（pandas、networkx、scikit-learn），在宽限制下可行，在紧限制下安装阶段即 OOM。更精简的策略（仅用标准库从头实现数学运算）同样存在，但不同模型有不同的默认策略，资源配置决定了哪种策略恰好能成功。

### 跨评估验证

该效应在 Anthropic 不同模型上一致（方向一致、幅度不同）。在 SWE-bench 上的交叉实验中，将可用内存从 1x 变化至 5x（227 个问题、每题 10 个样本），同样趋势成立但幅度较小：5x 比 1x 仅高 1.54 个百分点。SWE-bench 任务资源需求较低，效应更小是合理的。

### 其他方差来源

时间限制在某些配置下也起作用。评估设置的每个元素都可能影响最终得分：集群健康、硬件规格、并发级别、甚至出站带宽。智能体评估本质上是端到端系统测试，任何组件都可能成为混淆因素。团队观察到通过率随一天中的时间波动，可能与 API 延迟随流量模式变化有关。

### 建议

理想方案是在完全相同的硬件条件下运行评估，但实际中不一定可行。鉴于容器运行时通过保证分配和硬性上限两个参数执行资源，建议评估为每个任务分别指定这两个参数，而非单一值。两者之间的区间应校准至使得分差异落在噪声范围内。以 Terminal-Bench 2.0 为例，3x 上限将基础设施错误率降低约三分之二（5.8%→2.1%，p < 0.001），同时得分提升在噪声范围内（p=0.40）。

### 核心结论

基准测试得分越来越多地被用作决策依据，但相应的运行和报告严谨性并未同步提升。排行榜上 2 分的领先可能反映真正的能力差异，也可能只是因为评估运行在更强的硬件上，或碰巧在一天中更幸运的时间。在资源方法论标准化之前，低于 3 个百分点的排行榜差距值得怀疑。"几分的领先可能代表真正的能力差距——也可能只是一台更大的虚拟机。"
