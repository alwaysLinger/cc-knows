# 量化智能体编程评估中的基础设施噪声

发布于 2026 年 2 月 5 日

基础设施配置可以使智能体编程基准测试的得分波动数个百分点——有时甚至超过排行榜上顶级模型之间的差距。

智能体编程基准测试（如 SWE-bench 和 Terminal-Bench）常用于比较前沿模型的软件工程能力——排行榜上的顶级位置往往仅差几个百分点。这些得分常被视为模型相对能力的精确测量，并越来越多地用于指导部署决策。然而，我们发现仅基础设施配置就能产生超过这些差距的差异。在内部实验中，Terminal-Bench 2.0 上资源最充足与最紧张配置之间的差距为 6 个百分点（p < 0.01）。

静态基准测试直接对模型输出评分——运行时环境不影响结果。智能体编程评估则不同：模型获得一个完整的环境，在其中编写程序、运行测试、安装依赖，并经过多轮迭代。运行时不再是一个被动容器，而是问题解决过程的有机组成部分。资源预算和时间限制不同的两个智能体，实际上并非在参加同一场考试。

评估开发者已经开始考虑这一点。例如，Terminal-Bench 2.0 在其最新的 2.0 版本中按任务指定了推荐的 CPU 和 RAM。然而，指定资源并不等于一致地执行资源限制。此外，我们发现执行方法本身也会改变基准测试最终实际测量的内容。

---

### 我们是如何发现这个问题的

我们在 Google Kubernetes Engine（GKE）集群上运行 Terminal-Bench 2.0。在校准设置时，我们注意到得分与基准测试的官方排行榜不一致，而且基础设施错误率异常高：多达 6% 的任务因 Pod 错误而失败，其中大部分与模型解决任务的能力无关。

得分差异归根结底在于资源执行方式。我们的 Kubernetes 实现将每任务资源规格同时作为保证分配和硬性上限：每个容器被保证获得指定资源，但一旦超出即被终止。容器运行时通过两个独立参数执行资源限制：保证分配（guaranteed allocation）——预先预留的资源——以及硬性上限（hard limit）——容器被终止的阈值。当这两个值设为相同时，瞬态峰值没有任何余量：短暂的内存波动就可能 OOM 终止一个本可成功的容器。为了解决这个问题，Terminal-Bench 的排行榜使用了不同的沙箱方案，其实现更加宽松，允许临时超额分配而不终止容器，以优先保证基础设施稳定性。

这一发现引出了一个更大的问题：资源配置对评估得分的影响到底有多大？

为了量化脚手架（scaffold）的影响，我们在六种资源配置下运行了 Terminal-Bench 2.0，从严格执行每任务规格（1x，即同时作为下限和上限），到完全无上限。其他一切保持不变：相同的 Claude 模型、相同的测试框架、相同的任务集。

在我们的实验中，成功率随资源余量增加而提高。这主要由基础设施错误率在每个步骤单调下降所驱动，从严格执行时的 5.8% 下降到无上限时的 0.5%。从严格执行到 3x 余量的下降（5.8% 到 2.1%）在 p < 0.001 水平上显著。余量越多，因超出分配而被终止的容器越少。

从 1x 到 3x，成功率在噪声范围内波动（p=0.40）。大多数在 1x 下崩溃的任务本就会失败——这是我们在数据中观察到的。智能体进行探索，撞上资源墙，被抢占，但它从未走在通往正确解的路上。

然而，从 3x 左右开始，这一趋势发生了变化：成功率的上升速度超过了基础设施错误的下降速度。

从 3x 到无上限，基础设施错误额外下降 1.6 个百分点，而成功率跃升近 4 个百分点。额外的资源使智能体能够尝试只有在大额分配下才能奏效的方法，例如拉取大型依赖、生成高开销的子进程，以及运行内存密集型的测试套件。在无限制资源下，相对于 1x 的总提升为 +6 个百分点（p < 0.01）。在边际上，像 `rstan-to-pystan` 和 `compile-compcert` 这样的任务在获得内存余量时成功率显著改善。

---

### 这如何影响测量

在约 3x Terminal-Bench 规格以内，额外资源修复的是基础设施可靠性问题，即瞬态资源峰值。Terminal-Bench 维护者使用的沙箱方案在后台隐式地做到了这一点；评估变得更稳定，但并没有变得更简单。

然而，超过 3x 标记后，额外资源开始主动帮助智能体解决此前无法解决的问题，这表明资源限制实际上可以改变评估所测量的内容。紧限制无意中奖励非常高效的策略，而宽限制更宽容，奖励能更好利用所有可用资源的智能体。

一个编写精简、高效代码且速度很快的智能体在紧约束下表现良好。一个用重型工具暴力求解的智能体在宽约束下表现良好。两者都是合理的测试对象，但将它们合并为单一得分而不标明资源配置，使得差异——以及现实泛化能力——难以解释。

在 `bn-fit-modify` 这个需要贝叶斯网络拟合的 Terminal-Bench 任务上，某些模型的第一步是安装标准 Python 数据科学栈：`pandas`、`networkx`、`scikit-learn` 及其所有工具链。在宽限制下，这行得通。在紧限制下，Pod 在安装阶段就耗尽内存，而智能体还未写下一行解决方案代码。更精简的策略是存在的（仅使用标准库从头实现数学运算），而且有些模型确实默认采用它。其他模型则不会。不同的模型有不同的默认策略，而资源配置决定了哪些策略恰好能成功。

我们在不同的 Anthropic 模型上复现了这一核心发现。效应的方向一致，幅度有所不同。同样的趋势似乎在 Claude 以外的模型上也成立，但我们尚未进行严格测试。

我们还通过在 SWE-bench 上运行交叉实验，测试了这种模式是否在 Terminal-Bench 以外的评估上成立。我们在 227 个问题（每题 10 个样本）上将可用总 RAM 从基线的 1x 变化到 5x。同样的效应成立，但幅度较小：得分同样随 RAM 单调增加，但 5x 仅比 1x 高 1.54 个百分点。SWE-bench 任务的资源需求较低，因此效应更小是预期的，但这表明资源配置在那里也不是中性的。

### 其他方差来源

资源配置并非唯一的隐藏变量。在某些配置下，时间限制也开始发挥作用。

原则上，评估设置的每个元素都可能影响最终得分，从集群健康状况到硬件规格，从并发级别甚至到出站带宽。智能体评估本质上是端到端的系统测试，该系统的任何组件都可能成为混淆因素。例如，我们观察到一个有趣的现象：通过率随一天中的时间波动，这可能是因为 API 延迟随流量模式和事件而变化。我们尚未正式量化这一效应，但它说明了一个更大的问题："模型能力"与"基础设施行为"之间的边界比单一基准测试得分所暗示的更加模糊。模型提供商可以通过专用硬件来屏蔽其评估基础设施免受此影响，但外部评估者很难做到同样的事情。

公共基准测试通常旨在测量纯模型能力，但在实践中，它们有将其与基础设施怪癖混淆的风险。有时这可能是可取的，因为它能实现整个栈的端到端测试，但更多时候并非如此。对于旨在公开共享的编程评估，在多个时间和多天运行有助于平均掉噪声。

### 我们的建议

理想的方案是在完全相同的硬件条件下运行每个评估——包括运行评估的脚手架和推理栈——因为这能确保全面的完美可复现性。然而，这并不总是实际的。

鉴于容器运行时实际执行资源的方式——通过保证分配和单独的硬性终止阈值——我们建议评估为每个任务指定这两个参数，而非单一固定值。单一精确规格将保证分配等同于终止阈值，不留任何余量：我们在 1x 下记录到的瞬态内存峰值就足以破坏评估的稳定性。将两个参数分开，可以让容器获得足够的喘息空间以避免虚假的 OOM 终止，同时仍然执行防止得分膨胀的硬性上限。

两者之间的区间应校准至使下限和上限处的得分落在噪声范围内。例如，在 Terminal-Bench 2.0 中，比每任务规格高 3x 的上限将基础设施错误率降低了约三分之二（5.8% 到 2.1%，p < 0.001），同时得分提升温和且完全在噪声范围内（p = 0.40）。这是一个合理的权衡：基础设施混淆因素在很大程度上被中和，而没有移除有意义的资源压力。确切倍数因基准测试和任务分布而异，因此应予以报告，但经验校准原则是通用的。

### 为什么我们关注此事

这些发现对评估基础设施之外也有实际影响。基准测试得分越来越多地被用作决策输入，但这种增加的关注度（和依赖度）并未总是伴随着相应的运行和报告严谨性。就目前而言，排行榜上 2 分的领先可能反映真正的能力差异，也可能反映某个评估运行在更强的硬件上，甚至是在一天中更幸运的时间，或两者兼有。如果没有公开（或标准化）的设置配置，外部很难判断，除非有兴趣的各方付出额外努力在相同条件下复现客观结果。

对于像 Anthropic 这样的实验室，其含义是智能体评估的资源配置应被视为一等实验变量，以与提示格式（prompt format）或采样温度（sampling temperature）同等的严谨性进行记录和控制。对于基准测试维护者，发布推荐资源规格（如 Terminal-Bench 2.0 所做的）可以起到很大作用，而指定执行方法将弥合我们发现的那道差距。对于任何消费基准测试结果的人，核心要点是：智能体评估上的小分数差异比所报告数字的精度所暗示的具有更多不确定性——特别是因为某些混淆因素简直太难控制了。

在资源方法论标准化之前，我们的数据表明排行榜上低于 3 个百分点的差距值得怀疑，直到评估配置被记录和匹配。我们在 Terminal-Bench 中等资源配置范围内观察到的差距略低于 2 个百分点。朴素的二项式置信区间已经跨越 1-2 个百分点；我们在此记录的基础设施混淆因素叠加在其之上，而非包含其中。在分配范围的两端，差距达到 6。

"几分的领先可能代表真正的能力差距——也可能只是一台更大的虚拟机。"
