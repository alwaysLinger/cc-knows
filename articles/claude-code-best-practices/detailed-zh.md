# Claude Code 最佳实践：详细摘要

## 核心约束：上下文窗口管理

文章开篇指出，大多数最佳实践都源于一个根本约束：**Claude 的上下文窗口会快速填满，且性能随填充而下降。** 上下文窗口包含整个对话历史、每次文件读取和命令输出。一次调试或代码探索就可能消耗数万 token。当上下文接近饱和时，Claude 开始遗忘早期指令，错误率上升。因此，上下文窗口是需要管理的最重要资源。可通过自定义状态栏持续跟踪上下文使用量。

## 给 Claude 验证自身工作的方式

这是**单一最高杠杆**的做法。Claude 能自行验证（运行测试、对比截图、校验输出）时，表现会显著提升。没有验证标准，Claude 可能产出看似正确但实际不工作的结果，用户成为唯一的反馈环，每个错误都需要人工介入。

具体策略包括：
- **提供验证标准**：给出具体测试用例，实现后运行测试
- **视觉验证 UI**：粘贴截图让 Claude 对比实现与设计
- **定位根因**：粘贴错误信息，修复后验证构建成功，而非压制错误

UI 变更可借助 Claude Chrome 扩展在浏览器中测试。验证也可以是测试套件、linter 或检查输出的 Bash 命令。

## 先探索，再规划，再编码

让 Claude 直接编码容易解决错误的问题。推荐四阶段工作流：

1. **探索**：进入 Plan 模式，Claude 只读文件、回答问题，不做修改
2. **规划**：让 Claude 创建详细实现方案，可用 Ctrl+G 在编辑器中直接修改
3. **实现**：退出 Plan 模式，让 Claude 按方案编码并验证
4. **提交**：让 Claude 提交并创建 PR

注意：Plan 模式有开销。小任务（修错别字、加日志、重命名变量）直接做即可。规划在不确定方案、修改多文件、不熟悉代码时最有价值。

## 在提示中提供具体上下文

Claude 能推断意图但不会读心。具体策略：

- **界定范围**：指定文件、场景和测试偏好
- **指向来源**：让 Claude 查 git 历史等具体信息源
- **引用已有模式**：指向代码库中的现有实现作为参考
- **描述症状**：提供症状、可能位置和"修复"的定义

丰富内容可通过 `@` 引用文件、粘贴图片、给出 URL、管道输入数据、让 Claude 自行获取上下文等方式提供。

## 配置环境

### 写好 CLAUDE.md

CLAUDE.md 是 Claude 每次对话开始时读取的特殊文件。用 `/init` 生成初始版本，然后持续精炼。关键原则：
- **应包含**：Claude 猜不到的 Bash 命令、偏离默认的代码风格、测试指令、仓库规范、架构决策、开发环境怪癖、常见陷阱
- **不应包含**：Claude 能从代码推断的内容、标准语言惯例、详细 API 文档（改为链接）、频繁变化的信息、长篇教程、逐文件描述、"写干净代码"等自明规则

文件过长会导致重要规则被淹没。如果 Claude 不断违反某条规则，可能是文件太长；如果 Claude 问的问题在 CLAUDE.md 中已有答案，可能是措辞模糊。用强调标记（如 "IMPORTANT"）提升遵从度。

CLAUDE.md 支持用 `@path/to/import` 语法导入其他文件，可放在主目录、项目根目录、父目录和子目录等位置。

### 配置权限

三种减少中断的方式：**Auto 模式**（分类器模型审批命令）、**权限白名单**（允许已知安全命令）、**沙箱**（OS 级隔离）。按信任程度选择。

### 使用 CLI 工具

CLI 工具是与外部服务交互最节省上下文的方式。安装 `gh`、`aws`、`gcloud` 等工具。Claude 也能学习不熟悉的 CLI 工具。

### 连接 MCP 服务器

用 `claude mcp add` 连接 Notion、Figma、数据库等外部工具，扩展 Claude 的能力范围。

### 设置 Hooks

Hooks 在 Claude 工作流的特定节点自动运行脚本。与 CLAUDE.md 指令（建议性）不同，Hooks 是确定性的，保证动作一定执行。可让 Claude 帮你写 Hooks。

### 创建 Skills

在 `.claude/skills/` 中创建 `SKILL.md` 文件，为 Claude 添加领域知识和可复用工作流。Skills 按需加载，不会膨胀每次对话。可用 `disable-model-invocation: true` 控制手动触发。

### 创建自定义子代理

在 `.claude/agents/` 中定义专业助手，拥有独立上下文和工具集。适用于需要读取大量文件或需要专注的任务，不会污染主对话。

### 安装插件

运行 `/plugin` 浏览市场，插件捆绑了 Skills、Hooks、子代理和 MCP 服务器。

## 有效沟通

### 像问资深工程师一样提问

Claude 适合用于代码库学习和探索，如"日志怎么工作的？""怎么新建 API 端点？"这是有效的入职流程。

### 让 Claude 采访你

对大型功能，先让 Claude 用 AskUserQuestion 工具采访你，涵盖技术实现、UI/UX、边界情况和权衡。采访完成后写完整规格，然后在新会话中执行。

## 管理会话

### 频繁纠偏

- **Esc**：中途停止，保留上下文可重定向
- **Esc+Esc 或 /rewind**：打开回退菜单
- **"Undo that"**：让 Claude 撤回更改
- **/clear**：不相关任务间重置上下文

同一问题修正超过两次后，用 `/clear` 清空，写更好的初始提示重新开始。

### 激进管理上下文

频繁使用 `/clear`；自动压缩触发时保留重要内容；用 `/compact <指令>` 控制压缩方向；用 Esc+Esc 选择检查点做部分压缩；在 CLAUDE.md 中自定义压缩行为；用 `/btw` 提问不进入历史的问题。

### 用子代理做调研

子代理在独立上下文中探索，返回摘要，不污染主对话。也可用于实现后的代码审查。

### 检查点回退

Claude 在变更前自动创建检查点。可恢复对话、代码或两者。这允许大胆尝试，失败了就回退。

### 恢复对话

用 `claude --continue` 继续最近会话，`claude --resume` 从列表选择。用 `/rename` 命名会话便于查找。

## 自动化与扩展

### 非交互模式

`claude -p "prompt"` 运行非交互会话，可集成 CI、pre-commit 钩子和脚本。支持纯文本、JSON 和流式 JSON 输出格式。

### 多会话并行

- **Worktrees**：隔离的 git 检出
- **桌面应用**：可视化多会话管理
- **Web 版 Claude Code**：云端隔离 VM
- **Agent Teams**：自动协调多会话

推荐 Writer/Reviewer 模式：一个会话写代码，另一个审查，避免自我偏见。

### 批量任务

循环调用 `claude -p` 处理文件列表，用 `--allowedTools` 限定权限。先在少量文件测试，优化提示后大规模运行。

### Auto 模式

用 `--permission-mode auto` 实现无人值守执行，分类器模型在后台做安全检查，阻断越权、未知基础设施和恶意内容驱动的动作。

## 常见失败模式

1. **大杂烩会话**：不相关任务混在一起 → `/clear` 分隔
2. **反复修正**：两次修正失败后上下文被污染 → 清空重来写更好提示
3. **CLAUDE.md 过长**：重要规则被淹没 → 无情精简
4. **信任-验证断层**：缺测试/校验 → 始终提供验证
5. **无限探索**：不设边界的调研 → 限定范围或用子代理

## 发展直觉

这些模式是起点而非教条。有时应让上下文积累（复杂问题中历史有价值），有时应跳过规划（探索性任务），有时模糊提示恰好在观察 Claude 如何理解问题。注意什么有效什么无效，逐步发展出指南无法捕捉的直觉。
