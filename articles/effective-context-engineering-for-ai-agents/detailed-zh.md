# AI 智能体的有效上下文工程 — 详细摘要

**原文**: [Effective Context Engineering for AI Agents](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents) (Anthropic, 2025-09-29)

---

## 一、上下文工程 vs 提示词工程

在 Anthropic 看来，上下文工程是提示词工程的自然进阶。提示词工程聚焦于编写和组织 LLM 指令以获得最优输出，而**上下文工程**涵盖在 LLM 推理过程中策展和维护最优 token 集合的全部策略，包括系统指令、工具、MCP、外部数据、消息历史等所有进入上下文窗口的信息。

随着 AI 应用从单轮分类/生成任务转向多轮、长时间运行的代理（Agent），工程师需要管理整个上下文状态。代理在循环中不断产生新数据，这些信息必须被循环精炼——上下文工程就是从不断演化的信息宇宙中，策展进入有限上下文窗口的内容。

## 二、为什么上下文工程对构建能力强大的代理至关重要

研究表明，随着上下文窗口中 token 数量增加，模型准确回忆信息的能力会下降——这被称为**上下文腐化（Context Rot）**。LLM 和人类一样拥有有限的"注意力预算"（Working Memory），每个新 token 都会消耗部分预算。

这源于 Transformer 架构的特性：n 个 token 之间形成 n² 的成对注意力关系，上下文越长，注意力关系越稀薄。此外，模型训练数据中短序列远多于长序列，导致模型对长上下文依赖的经验和专用参数更少。位置编码插值等技术虽允许模型处理更长序列，但会带来位置理解精度下降。这些因素造成的是性能梯度而非断崖：模型在长上下文下仍能工作，但精度可能下降。

## 三、有效上下文的剖析

核心原则：**找到最小的高信号 token 集合，最大化期望结果的概率。**

### 系统提示词

应使用清晰、直接的语言，找到"恰当高度"——在两个常见失败模式之间取得平衡：
- **过高硬编码**：在提示词中写死复杂逻辑，导致脆弱性和维护困难
- **过于笼统**：提供模糊的高层指导，缺乏具体信号

推荐将提示词组织为独立段落（如 `<background_information>`、`<instructions>` 等），使用 XML 标签或 Markdown 标题分隔。从最小提示词开始测试，根据失败模式逐步添加指令和示例。

### 工具

工具定义了代理与信息/行动空间之间的契约。工具应自包含、健壮、用途明确，输入参数应描述性强且无歧义。最常见的失败模式是**臃肿的工具集**——功能重叠过多或导致工具选择歧义。如果人类工程师都无法确定该用哪个工具，AI 代理更不可能做好。

### 示例

提供示例（少样本提示）是已知最佳实践，但不应堆砌边界情况清单。应精选一组**多样且典型的示例**来有效展示期望行为——对 LLM 而言，示例就是"一图胜千言"。

## 四、上下文检索与代理式搜索

代理的定义：LLM 在循环中自主使用工具。

当前趋势从嵌入式的预推理检索转向**"即时"（Just-in-Time）上下文策略**：代理维护轻量标识符（文件路径、存储查询、网页链接等），运行时通过工具动态加载数据。例如 Claude Code 使用 `head`、`tail` 等命令分析大数据，而无需将完整数据加载到上下文中。这种方式类似人类认知——我们不记忆整个信息库，而是建立索引系统按需检索。

这些引用的元数据本身也提供行为信号：文件名 `test_utils.py` 在 `tests/` 文件夹中与在 `src/core_logic/` 中暗示不同用途。文件夹层级、命名规范、时间戳都是重要信号。

自主导航还支持**渐进式发现（Progressive Disclosure）**：代理通过探索逐步发现相关上下文，每次交互产出指导下一步决策的上下文，维持工作记忆中仅保留必要内容。

当然也有权衡：运行时探索比预取慢，且需要精心工程设计以确保代理拥有正确的工具和启发式方法。**混合策略**可能是最有效的——部分数据预取加速，部分自主探索补充。Claude Code 即采用此模式：CLAUDE.md 文件预加载，`glob`/`grep` 等原语用于即时导航。

## 五、长时任务的上下文工程

### 压缩（Compaction）

当对话接近上下文窗口上限时，摘要关键内容并重启新窗口。在 Claude Code 中，模型保留架构决策、未解决 bug 和实现细节，丢弃冗余工具输出。压缩的艺术在于取舍——过于激进的压缩可能丢失关键上下文。建议先最大化召回率确保不遗漏，再迭代提高精度消除冗余。最安全的轻量压缩形式是**工具结果清除（Tool Result Clearing）**。

### 结构化笔记（Structured Note-taking）

代理定期将笔记持久化到上下文窗口之外的内存中，后续按需调回。Claude Code 的待办列表就是此模式。Claude 玩 Pokémon 的案例展示了代理如何在没有记忆结构提示的情况下，自主维护精确计数、区域地图、战斗策略等笔记，在上下文重置后读取笔记继续多小时的训练或探索。Anthropic 已在 Claude 开发者平台发布文件式记忆工具公测版。

### 子代理架构（Sub-agent Architectures）

主代理协调高层计划，专用子代理以独立上下文执行聚焦任务。子代理可能使用数万 token 广泛探索，但仅返回 1,000-2,000 token 的精炼摘要。这在复杂研究任务上比单代理系统表现显著提升。

三种技术的适用场景：
- **压缩**：需要大量来回交互的任务
- **笔记**：有明确里程碑的迭代开发
- **多代理**：并行探索有收益的复杂研究分析

## 六、结论

上下文工程代表了构建 LLM 应用的根本性转变。核心原则始终不变：**找到最小的高信号 token 集合，最大化期望结果的概率**。随着模型能力提升，所需工程指令会减少，但将上下文视为珍贵有限资源将始终是构建可靠代理的核心。
