# AI 智能体的有效上下文工程 — 简要提炼

**原文**: [Effective Context Engineering for AI Agents](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents) (Anthropic, 2025-09-29)

## 核心观点

**上下文工程（Context Engineering）** 是提示词工程（Prompt Engineering）的自然演进。它不再只关注如何写出好的提示词，而是回答一个更根本的问题：什么样的上下文配置最可能让模型产生期望的行为？上下文是有限资源——随着 token 数量增长，模型准确回忆信息的能力会下降（即"上下文腐化"现象），因此必须精心管理每一轮推理中进入模型注意力预算的 token 集合。

## 主要论据

- LLM 基于 Transformer 架构，token 间存在 n² 的注意力关系，上下文越长，注意力越稀薄
- 模型在训练数据中更常见短序列，对长上下文依赖的经验和参数较少
- 位置编码插值等技术虽能扩展上下文窗口，但会带来位置理解精度下降

## 关键实践

1. **系统提示词**：找到"恰当高度"——既不过度硬编码脆弱逻辑，也不过于笼统
2. **工具设计**：保持精简、无歧义，避免臃肿工具集导致决策模糊
3. **示例选择**：精选少量多样且典型的示例，而非堆砌边界情况
4. **即时检索（Just-in-Time）**：代理维护轻量标识符，运行时动态加载数据，而非预先加载全部信息
5. **混合策略**：部分数据预取加速，部分自主探索补充

## 长时任务的三种技术

- **压缩（Compaction）**：接近上下文窗口上限时，摘要关键内容并重启新窗口
- **结构化笔记（Structured Note-taking）**：代理将关键信息持久化到上下文窗口之外，后续按需调回
- **子代理架构（Sub-agent Architectures）**：主代理协调，子代理以独立上下文执行专项任务，仅返回精炼摘要

## 结论

上下文工程的核心原则始终如一：**找到最小的高信号 token 集合，最大化期望结果的概率**。随着模型能力提升，所需的工程指令会减少，但将上下文视为珍贵有限资源这一理念将始终是构建可靠代理的核心。
