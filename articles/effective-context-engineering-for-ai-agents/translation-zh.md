# AI 智能体的有效上下文工程

发布于 2025 年 9 月 29 日

上下文是 AI 智能体（AI Agent）的关键但有限的资源。在这篇文章中，我们探索有效策展和管理驱动智能体的上下文的策略。

经过几年提示词工程（Prompt Engineering）成为应用 AI 的焦点之后，一个新术语开始崭露头角：**上下文工程（Context Engineering）**。用语言模型构建应用，越来越不在于为提示词找到正确的措辞，而更多在于回答一个更广泛的问题——"什么样的上下文配置最可能让模型产生我们期望的行为？"

**上下文**是指从大语言模型（LLM）采样时包含的 token 集合。当前的**工程**问题在于，在 LLM 固有约束下优化这些 token 的效用，以持续达成期望结果。有效驾驭 LLM 通常需要*以上下文的方式思考*——换言之：考虑 LLM 在任意时刻可用的整体状态，以及该状态可能产生的潜在行为。

在这篇文章中，我们将探索上下文工程这一新兴技艺，并提供一个构建可操控、高效智能体的精炼心智模型。

---

## 上下文工程 vs 提示词工程

在 Anthropic，我们将上下文工程视为提示词工程的自然进阶。提示词工程指的是编写和组织 LLM 指令以获得最优结果的方法（参见[我们的文档](https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/overview)获取概述和有用的提示词工程策略）。**上下文工程**指的是在 LLM 推理过程中策展和维护最优 token（信息）集合的一整套策略，包括提示词之外可能出现在上下文中的所有其他信息。

在用 LLM 进行工程的早期，提示是 AI 工程工作中最大的组成部分，因为日常聊天交互之外的大多数用例都需要针对单次分类或文本生成任务优化的提示。顾名思义，提示词工程的主要焦点是如何编写有效的提示，尤其是系统提示（System Prompt）。然而，随着我们转向构建能在多轮推理和更长时间跨度上运行的更强大智能体，我们需要管理整个上下文状态（系统指令、工具、[模型上下文协议](https://modelcontextprotocol.io/docs/getting-started/intro)（Model Context Protocol, MCP）、外部数据、消息历史等）的策略。

在循环中运行的智能体会生成越来越多的可能对下一轮推理相关的数据，这些信息必须被循环精炼。上下文工程就是从不断演化的可能信息宇宙中，策选进入有限上下文窗口的内容的[艺术与科学](https://x.com/karpathy/status/1937902205765607626?lang=en)。

_与编写提示这一离散任务不同，上下文工程是迭代的，策选阶段发生在我们每次决定传递什么给模型时。_

## 为什么上下文工程对构建强大智能体至关重要

尽管 LLM 速度越来越快、能管理越来越大的数据量，我们观察到 LLM 和人类一样，在某个点上会失去焦点或产生困惑。对"大海捞针"式基准测试的研究揭示了[上下文腐化（Context Rot）](https://research.trychroma.com/context-rot)的概念：随着上下文窗口中 token 数量增加，模型从上下文中准确回忆信息的能力会下降。

虽然一些模型展现出比其他模型更温和的性能退化，但这一特征在所有模型中都会出现。因此，上下文必须被视为边际收益递减的有限资源。像人类拥有[有限的工作记忆容量](https://journals.sagepub.com/doi/abs/10.1177/0963721409359277)一样，LLM 在解析大量上下文时也拥有一个"注意力预算"。每个新引入的 token 都会在一定程度上消耗这个预算，这增加了仔细策选 LLM 可用 token 的必要性。

这种注意力稀缺性源于 LLM 的架构约束。LLM 基于 [Transformer 架构](https://arxiv.org/abs/1706.03762)，该架构使每个 token 能够[关注（Attend to）](https://huggingface.co/blog/Esmail-AGumaan/attention-is-all-you-need)整个上下文中的所有其他 token。这产生了 n 个 token 之间 n² 的成对关系。

随着上下文长度增加，模型捕捉这些成对关系的能力被拉薄，在上下文大小和注意力聚焦之间产生了天然张力。此外，模型从训练数据分布中发展出注意力模式，其中短序列通常比长序列更常见。这意味着模型对长上下文依赖的经验更少，专用参数也更少。

[位置编码插值（Position Encoding Interpolation）](https://arxiv.org/pdf/2306.15595)等技术允许模型通过适配到原本训练的更小上下文来处理更长序列，但会带来一些 token 位置理解上的退化。这些因素创造的是性能梯度而非硬性断崖：模型在更长上下文下仍然高度能干，但在信息检索和长程推理方面可能表现出相比更短上下文时降低的精度。

这些现实意味着，深思熟虑的上下文工程对构建强大智能体至关重要。

## 有效上下文的剖析

鉴于 LLM 受限于有限的注意力预算，*好的*上下文工程意味着找到*最小*的高信号 token 集合，以最大化某种期望结果的概率。实施这一实践说起来容易做起来难，但在下文中，我们将概述这一指导原则在上下文不同组件中的实际含义。

**系统提示词**应极其清晰，使用简单直接的语言，以*恰当的高度*呈现想法。恰当的高度是两个常见失败模式之间的金发姑娘区（Goldilocks Zone）。在一个极端，我们看到工程师在提示词中硬编码复杂、脆弱的逻辑来激发精确的智能体行为。这种方法造成了脆弱性并增加了随时间推移的维护复杂性。在另一个极端，工程师有时提供模糊的高层指导，未能给 LLM 提供期望输出的具体信号，或错误假设了共享上下文。最优高度在两者之间取得平衡：足够具体以有效引导行为，又足够灵活以为模型提供强有力的启发式方法来指导行为。

![在上下文工程过程中校准系统提示词](/_next/image?url=https%3A%2F%2Fwww-cdn.anthropic.com%2Fimages%2F4zrzovbb%2Fwebsite%2F0442fe138158e84ffce92bed1624dd09s37ac46f-2292x1288.png&w=3840&q=75)

_在光谱的一端，我们看到脆弱的 if-else 硬编码提示，在另一端是过于笼统或错误假设共享上下文的提示。_

我们推荐将提示词组织为独立段落（如 `<background_information>`、`<instructions>`、`## 工具指导`、`## 输出描述` 等），并使用 XML 标签或 Markdown 标题来划分这些段落，尽管随着模型变得更强大，提示词的确切格式可能变得不那么重要。

无论你如何决定构建系统提示词，你都应该追求完全描述期望行为的最小信息集。（注意最小并不一定意味着简短；你仍然需要预先给智能体足够的信息以确保它遵循期望行为。）最好从用可用最佳模型测试最小提示词开始，看它在你的任务上表现如何，然后根据初始测试中发现的失败模式添加清晰的指令和示例来改善性能。

**工具**允许智能体与其环境交互，并在工作时获取新的额外上下文。因为工具定义了智能体与其信息/行动空间之间的契约，工具促进效率极其重要——既通过返回 token 高效的信息，也通过鼓励高效的智能体行为。

在[为 AI 智能体编写工具——用 AI 智能体](https://www.anthropic.com/engineering/writing-tools-for-agents)中，我们讨论了构建能被 LLM 良好理解且功能重叠最少的工具。与设计良好的代码库的函数类似，工具应当自包含、对错误健壮，且在预期用途方面极其明确。输入参数同样应当描述性强、无歧义，并发挥模型的固有优势。

我们看到最常见的失败模式之一是臃肿的工具集——覆盖过多功能或导致关于使用哪个工具的歧义决策点。如果人类工程师都无法确定在给定情况下应该使用哪个工具，就不能期望 AI 智能体做得更好。正如我们稍后将讨论的，为智能体策选最小可行工具集也可以在长时间交互中带来更可靠的上下文维护和修剪。

提供示例，又称少样本提示（Few-shot Prompting），是我们继续强烈建议的知名最佳实践。然而，团队常常在提示词中塞入一长串边界情况，试图阐述 LLM 在特定任务中应遵循的每一条规则。我们不推荐这种做法。相反，我们推荐致力于策选一组多样且典型的示例，有效展示智能体的期望行为。对 LLM 来说，示例就是"一图胜千言"的"图片"。

我们对上下文不同组件（系统提示词、工具、示例、消息历史等）的总体指导是：深思熟虑，让你的上下文信息丰富但紧凑。现在让我们深入运行时动态检索上下文。

## 上下文检索与代理式搜索

在[构建有效的 AI 智能体](https://www.anthropic.com/research/building-effective-agents)中，我们强调了基于 LLM 的工作流和智能体之间的区别。自那篇文章以来，我们倾向于智能体的一个[简单定义](https://simonwillison.net/2025/Sep/18/agents/)：LLM 在循环中自主使用工具。

与客户合作中，我们看到领域正在收敛到这个简单范式。随着底层模型变得更强大，智能体的自主水平可以扩展：更聪明的模型允许智能体独立驾驭微妙的问题空间并从错误中恢复。

我们现在看到工程师在设计智能体上下文方面的思维方式正在转变。今天，许多 AI 原生应用采用某种形式的基于嵌入的预推理时检索，来为智能体推理提供重要上下文。随着领域向更智能体的方法转变，我们越来越多地看到团队用"即时"（Just-in-Time）上下文策略来增强这些检索系统。

与预先处理所有相关数据不同，采用"即时"方法构建的智能体维护轻量标识符（文件路径、存储查询、网页链接等），并使用这些引用在运行时通过工具将数据动态加载到上下文中。Anthropic 的智能体编码方案 [Claude Code](https://www.anthropic.com/claude-code) 使用这种方法对大型数据库执行复杂数据分析。模型可以编写针对性查询、存储结果，并利用 Bash 命令如 `head` 和 `tail` 来分析大量数据，而无需将完整数据对象加载到上下文中。这种方法类似人类认知：我们通常不会记忆整个信息语料库，而是引入外部组织和索引系统，如文件系统、收件箱和书签，以按需检索相关信息。

除了存储效率，这些引用的元数据还提供了一种高效精炼行为的机制——无论是显式提供还是直观暗示的。对于在文件系统中操作的智能体，名为 `test_utils.py` 的文件出现在 `tests` 文件夹中暗示着与同名文件位于 `src/core_logic/` 中不同的用途。文件夹层级、命名规范和时间戳都提供了重要信号，帮助人类和智能体理解如何以及何时使用信息。

让智能体自主导航和检索数据还支持渐进式发现（Progressive Disclosure）——换言之，允许智能体通过探索逐步发现相关上下文。每次交互产出指导下一步决策的上下文：文件大小暗示复杂度；命名规范提示用途；时间戳可以是相关性的代理。智能体可以逐层组装理解，仅在工作记忆中保留必要内容，并利用笔记策略实现额外持久化。这种自我管理的上下文窗口使智能体聚焦于相关子集，而非淹没在详尽但可能无关的信息中。

当然，存在权衡：运行时探索比检索预计算数据更慢。不仅如此，还需要有主见且深思熟虑的工程设计，以确保 LLM 拥有正确导航其信息景观的工具和启发式方法。没有适当指导，智能体可能因误用工具、追踪死胡同或未能识别关键信息而浪费上下文。

在某些场景中，最有效的智能体可能采用混合策略——部分数据预取以提速，并根据自主判断进一步探索。"正确"自主水平的决策边界取决于任务。Claude Code 就是一个采用混合模型的智能体：[CLAUDE.md](http://claude.md) 文件被预先放入上下文，而 `glob` 和 `grep` 等原语允许它导航环境并即时检索文件，有效绕过了陈旧索引和复杂语法树的问题。

混合策略可能更适合内容动态性较低的上下文，如法律或金融工作。随着模型能力提升，智能体设计将趋向于让智能模型智能地行动，逐步减少人工策选。鉴于该领域快速进步的步伐，"做最简单可行的事"可能仍然是我们对在 Claude 上构建智能体的团队的最佳建议。

### 长时任务的上下文工程

长时任务（Long-horizon Task）要求智能体在 token 数量超过 LLM 上下文窗口的动作序列中保持连贯性、上下文和目标导向行为。对于跨越数十分钟到数小时连续工作的任务，如大型代码库迁移或综合研究项目，智能体需要专门技术来解决上下文窗口大小限制。

等待更大的上下文窗口可能看似是显而易见的策略。但在可预见的未来，所有大小的上下文窗口都可能受到上下文污染（Context Pollution）和信息相关性问题的困扰——至少在需要最强智能体性能的情况下。为使智能体能够在扩展时间跨度上有效工作，我们开发了几种直接应对上下文污染约束的技术：压缩（Compaction）、结构化笔记（Structured Note-taking）和多智能体架构（Multi-agent Architecture）。

#### 压缩

压缩（Compaction）是在对话接近上下文窗口限制时，摘要其内容并用摘要重新启动新上下文窗口的做法。压缩通常作为上下文工程中驱动更好长期连贯性的第一个杠杆。压缩的核心是以高保真方式蒸馏上下文窗口的内容，使智能体能以最小的性能退化继续工作。

例如在 Claude Code 中，我们通过将消息历史传递给模型来摘要和压缩最关键的细节来实现这一点。模型保留架构决策、未解决的 bug 和实现细节，同时丢弃冗余的工具输出或消息。智能体然后可以用这个压缩的上下文加上最近访问的五个文件继续。用户获得了连续性，无需担心上下文窗口限制。

压缩的艺术在于保留什么与丢弃什么的选择，因为过于激进的压缩可能导致丢失微妙但关键的上下文，其重要性可能只在后来才显现。对于实现压缩系统的工程师，我们推荐在复杂智能体轨迹上仔细调优提示词。从最大化召回率开始，确保压缩提示词捕获轨迹中每一条相关信息，然后迭代提高精度，消除多余内容。

低挂的多余内容的一个例子是清除工具调用和结果——一旦工具在消息历史深处被调用过，智能体为什么还需要再次看到原始结果？最安全的轻量压缩形式之一是工具结果清除（Tool Result Clearing），最近作为 [Claude 开发者平台上的一个功能](https://www.anthropic.com/news/context-management)发布。

#### 结构化笔记

结构化笔记（Structured Note-taking），或称智能体记忆（Agentic Memory），是一种智能体定期将笔记持久化到上下文窗口之外的内存中的技术。这些笔记在后续时间被调回上下文窗口。

这种策略以最小的开销提供持久记忆。像 Claude Code 创建待办列表，或你的自定义智能体维护 NOTES.md 文件，这个简单模式允许智能体在复杂任务中跟踪进度，维护关键上下文和依赖关系，否则这些信息会在数十次工具调用中丢失。

[Claude 玩 Pokémon](https://www.twitch.tv/claudeplayspokemon) 展示了记忆如何在非编码领域变革智能体能力。智能体在数千个游戏步骤中维护精确计数——跟踪诸如"在过去 1,234 步中我一直在 1 号路线训练 Pokémon，皮卡丘已获得 8 级，目标是 10 级"这样的目标。在没有任何关于记忆结构的提示下，它绘制了探索区域的地图，记住了已解锁的关键成就，并维护了战斗策略笔记，帮助它学习哪种攻击对不同对手最有效。

上下文重置后，智能体读取自己的笔记并继续多小时训练序列或地下城探索。这种跨越摘要步骤的连贯性使长时策略成为可能，而仅将所有信息保留在 LLM 上下文窗口中是不可能的。

作为我们 [Sonnet 4.5 发布](https://www.anthropic.com/effective-context-engineering-for-ai-agents)的一部分，我们在 Claude 开发者平台上以公测版发布了[记忆工具](http://anthropic.com/news/context-management)，通过基于文件的系统使存储和查询上下文窗口之外的信息变得更加容易。这允许智能体随时间建立知识库，跨会话维护项目状态，并引用先前工作，而无需将所有内容保留在上下文中。

#### 子智能体架构

子智能体架构（Sub-agent Architecture）提供了另一种绕过上下文限制的方式。不是一个智能体试图在整个项目中维护状态，专门的子智能体可以用干净的上下文窗口处理聚焦任务。主智能体用高层计划进行协调，而子智能体执行深层技术工作或使用工具查找相关信息。每个子智能体可能广泛探索，使用数万或更多 token，但仅返回其工作的浓缩摘要（通常 1,000-2,000 token）。

这种方法实现了关注点的清晰分离——详细的搜索上下文保留在子智能体内部，而主导智能体聚焦于综合和分析结果。这一模式在[我们如何构建多智能体研究系统](https://www.anthropic.com/engineering/multi-agent-research-system)中讨论过，在复杂研究任务上比单智能体系统表现出了显著提升。

这些方法的选择取决于任务特征。例如：
* 压缩为需要大量来回交互的任务维持对话流畅性
* 笔记在有明确里程碑的迭代开发中表现出色
* 多智能体架构处理并行探索有收益的复杂研究和分析

即使模型持续改进，在扩展交互中保持连贯性的挑战仍将是构建更有效智能体的核心。

## 结论

上下文工程代表了我们用 LLM 构建应用方式的根本性转变。随着模型变得更强大，挑战不仅仅是精心制作完美的提示——而是深思熟虑地策选在每一步中进入模型有限注意力预算的信息。无论你是在为长时任务实现压缩、设计 token 高效的工具，还是让智能体即时探索其环境，指导原则始终如一：**找到最小的高信号 token 集合，最大化期望结果的概率**。

我们概述的技术将随着模型改进而持续演进。我们已经看到更聪明的模型需要更少的规范性工程，允许智能体以更大自主性运行。但即使能力扩展，将上下文视为珍贵、有限资源仍将是构建可靠、有效智能体的核心。

今天就在 Claude 开发者平台上开始上下文工程，并通过我们的[记忆和上下文管理](https://platform.claude.com/cookbook/tool-use-memory-cookbook)手册获取有用的提示和最佳实践。
