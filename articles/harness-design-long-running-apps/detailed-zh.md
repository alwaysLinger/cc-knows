# Harness Design for Long-Running Application Development — 详细摘要

## 引言与背景

Anthropic Labs 团队的 Prithvi Rajasekaran 长期致力于两个互联问题：让 Claude 产出高质量前端设计，以及让它在无人干预下构建完整应用。此前团队已通过提示工程和 harness 设计将 Claude 性能提升至基线之上，但均触及天花板。本文介绍借鉴生成对抗网络（GAN）思想设计的多智能体架构，以突破这些瓶颈。

## 朴素实现为何失效

团队此前验证了 harness 设计对长时序智能体编码的显著影响，并观察到两个持续存在的失败模式：

**上下文退化与焦虑**：模型在长任务中逐渐丧失连贯性，且出现"上下文焦虑"——接近上下文限制时过早收尾。压缩（compaction）虽能保留连续性，但无法提供干净起点，焦虑仍会残留。上下文重置（完全清除上下文窗口并结构化交接状态）是更有效的解法，但引入了编排复杂度和额外开销。在 Claude Sonnet 4.5 上，压缩不足以解决焦虑问题，重置成为必要手段。

**自我评估失灵**：模型评估自身产出时倾向于过度正面评价，尤其在设计等主观领域。即使存在可验证结果的任务中，模型也会表现出不良判断。将工作智能体与评估智能体分离是有效的杠杆——单独调优一个持怀疑态度的评估器，远比让生成器批判自身工作更可行。

## 前端设计：让主观质量可评分

针对前端设计中自我评估问题最突出的场景，团队制定了四项评分标准：

- **设计质量**：设计是否呈现统一整体感，色彩、排版、布局等是否创造独特氛围与身份
- **原创性**：是否有定制决策的痕迹，而非模板布局、库默认值和 AI 生成模式（如紫色渐变白卡片）
- **工艺**：技术执行水准——排版层次、间距一致性、色彩和谐度、对比度
- **功能性**：独立于美学的可用性

标准有意加重设计与原创性权重，因 Claude 默认在工艺和功能性上已表现良好，但设计与原创性常产出平庸结果。评估器使用少样本示例校准，确保评分与人类偏好对齐。

技术实现基于 Claude Agent SDK。生成器产出 HTML/CSS/JS，评估器通过 Playwright MCP 主动浏览页面、截图分析后逐项评分并撰写详细评论。每轮迭代 5-15 次，每次推动生成器向更有特色的方向演进，完整运行最长达 4 小时。生成器还被告知在每轮评估后做战略决策：分数趋势良好则精炼当前方向，方向不对则转向全新美学。

关键发现：评分措辞直接影响输出特征——"最佳设计应达到博物馆水准"等表述推动了特定视觉收敛；分数虽整体提升但并非严格线性，中间迭代有时优于最终版本；实现复杂度跨轮次递增；即便首轮迭代，有评分标准的输出也明显优于无干预基线。一个引人注目的案例：荷兰艺术博物馆网站在第 10 轮迭代时放弃了此前干净暗色主题，转而创造了一个 3D 空间体验——CSS 透视渲染的棋盘地板、墙上自由挂画、门洞式画廊导航。

## 扩展至全栈编码

### 三智能体架构

将 GAN 模式扩展至全栈开发，构建了三智能体系统：

**规划器**：将 1-4 句简短提示扩展为完整产品规格。被要求在范围上雄心勃勃，聚焦产品语境和高层技术设计而非详细实现——因预判若规划器过早指定粒度过细的技术细节且出错，错误将级联传导至下游实现。同时被要求在规格中融入 AI 功能。

**生成器**：采用逐一功能实现模式，每个冲刺基于 React/Vite/FastAPI/SQLite（后改为 PostgreSQL）技术栈实现。冲刺结束后先自评，再交接给 QA。配备 git 做版本控制。

**评估器**：使用 Playwright MCP 像用户一样点击运行中的应用，测试 UI 功能、API 端点和数据库状态。对每个冲刺按产品深度、功能性、视觉设计和代码质量等标准打分，每项设硬阈值，任一不达标则冲刺失败并附详细反馈。

**冲刺合约机制**：因产品规格有意保持高层级，生成器与评估器在每个冲刺前协商"完成标准"——生成器提出构建内容和验证方式，评估器审核确保方向正确，双方迭代至达成一致。通信通过文件进行。

### 运行结果

以"复古 2D 游戏制作器"为测试用例：

| Harness | 时长 | 费用 |
|---------|------|------|
| 单智能体 | 20 分钟 | $9 |
| 完整 harness | 6 小时 | $200 |

单智能体产出看似符合预期，但深入使用后发现布局浪费空间、工作流僵化、核心游戏功能损坏（实体定义与游戏运行时之间的连接断裂）。

完整 harness 中，规划器将一行提示扩展为 16 功能、10 冲刺的规格，远超单智能体尝试范围。应用界面更精致，画布使用全视口，面板尺寸合理，视觉身份一致。精灵编辑器更丰富。因规划器被要求融入 AI 功能，应用内建了 Claude 集成。最关键的差异在游戏模式——实际可操控角色，物理虽有粗糙之处但核心功能可用。

评估器通过逐项检查冲刺合约的测试标准发现了大量具体 bug，例如矩形填充工具未正确触发、删除键处理器条件逻辑错误、FastAPI 路由顺序导致 422 错误等。但将评估器调优至此水平需要多轮迭代——早期运行中评估器会识别问题后又说服自己问题不大而批准工作，也倾向于浅层测试而非探测边缘情况。

## Harness 的迭代演进

### 简化原则

每个 harness 组件都编码了一个假设：模型无法独自完成某事。这些假设值得压力测试，因为它们可能不正确，也可能随模型改进而过时。团队采用逐一移除组件的方法评估影响。

### 移除冲刺结构

升级至 Opus 4.6 后（其计划更周密、维持智能体任务更久、长上下文检索更优），移除了冲刺分解。评估器改为结束时单次评审。

关键洞察：评估器的价值并非固定的是非决策，而是取决于任务是否超出当前模型独立能力边界。在 4.5 上边界很近，评估器对整个构建都有意义；在 4.6 上边界外推，原需评估器检查的任务现可由生成器独立完成，评估器在边界内成为不必要开销，但在生成器能力边缘处仍提供真实提升。

### V2 Harness 结果

以"浏览器端 DAW"为测试用例：

| 阶段 | 时长 | 费用 |
|------|------|------|
| 规划器 | 4.7 分钟 | $0.46 |
| 构建（第 1 轮） | 2 小时 7 分钟 | $71.08 |
| QA（第 1 轮） | 8.8 分钟 | $3.24 |
| 构建（第 2 轮） | 1 小时 2 分钟 | $36.89 |
| QA（第 2 轮） | 6.8 分钟 | $3.09 |
| 构建（第 3 轮） | 10.9 分钟 | $5.88 |
| QA（第 3 轮） | 9.6 分钟 | $4.06 |
| **合计** | **3 小时 50 分钟** | **$124.70** |

构建器在无冲刺分解的情况下连贯运行超过 2 小时。QA 仍捕获了真实缺陷：首轮发现核心 DAW 功能仅有展示层而缺乏交互深度（音轨无法拖拽、无乐器 UI 面板、无可视化效果编辑器）；次轮发现音频录制仍为桩代码、音轨缩放和拆分未实现、效果可视化仅为数字滑块。最终应用具备音乐制作程序核心功能，集成的 AI 代理可通过提示自主完成简单制作流程。

## 展望

随着模型持续改进，可期望其在更复杂任务上工作更长时间。在某些场景下，模型周围的脚手架重要性会降低；但另一方面，模型越强，开发能实现超越基线能力的复杂任务的 harness 空间就越大。

核心教训：始终对目标模型进行实验、阅读其在真实问题上的轨迹并调优性能；复杂任务中分解任务并对各子问题应用专门智能体仍有空间；新模型发布时应重新审视 harness，剥离不再承载性能的组件并添加实现新能力所需的组件。

作者的信念：有趣的 harness 组合空间不会随模型改进而缩小，而是迁移——AI 工程师的核心工作是持续找到下一个新颖组合。
