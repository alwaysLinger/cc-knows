# 用并行 Claude 团队构建 C 编译器 — 详细摘要

## 引言

Anthropic 安全团队研究员 Nicholas Carlini 介绍了一种名为"智能体团队"(Agent Teams) 的新方法：多个 Claude 实例并行工作于共享代码库，无需人工主动干预即可持续推进。为压力测试这一方法，他让 16 个智能体从零编写一个基于 Rust 的 C 编译器，目标是能编译 Linux 内核。历经近 2,000 次 Claude Code 会话和约 20,000 美元 API 费用，团队产出了一个 10 万行编译器，可在 x86、ARM 和 RISC-V 上编译 Linux 6.9。

## 实现长时运行的 Claude

现有的智能体框架（如 Claude Code）需要操作者在线协同。为让 Claude 持续自主推进，作者构建了一个简单的无限循环框架：Claude 完成一个任务后立即接取下一个。核心是一个 bash 脚本，在循环中反复调用 `claude --dangerously-skip-permissions`，将每次会话日志按 git commit 哈希存储。智能体提示中要求 Claude 将问题拆为小片、追踪进度、不断推进。循环无限运行——作者提到有一次 Claude 意外执行了 `pkill -9 bash` 杀死了自己，终止了循环。

## 并行运行 Claude

单智能体框架有两个弱点：(1) 一个会话同一时间只能做一件事；(2) 无法实现专业化分工。并行可以同时解决这两个问题。

实现方式极为精简：创建一个裸 git 仓库，每个智能体在独立 Docker 容器中运行，将仓库挂载到 `/upstream`，智能体克隆到本地 `/workspace` 完成后推送回上游。

同步算法：智能体通过在 `current_tasks/` 目录写文件来"锁定"任务。若两个智能体抢同一任务，git 同步机制会迫使后者选择不同任务。完成后拉取合并、推送变更、删除锁。合并冲突频繁但 Claude 能自行解决。新会话在新容器中启动，循环往复。

作者未实现其他跨智能体通信方式，也未使用编排智能体 (orchestration agent)，而是让每个 Claude 自行决定如何行动——通常选择"最明显的下一个问题"。

## 编程中的经验教训

### 编写极高质量的测试

Claude 会自主求解给定问题，因此任务验证器必须近乎完美，否则它就解错问题。改进测试框架需要：寻找高质量编译器测试套件、为开源软件编写验证器和构建脚本、观察智能体的错误模式并设计新测试。例如，项目后期 Claude 频繁在实现新功能时破坏已有功能，作者因此搭建了 CI 流水线并实施更严格的执行策略。

### 站在 Claude 的角度思考

作者不断提醒自己：测试框架是给 Claude 用的，不是给自己用的。这意味着要重新思考测试如何传达结果：

- **上下文窗口污染**：测试框架不应打印数千字节无用输出。最多打印几行，重要信息记入文件。日志应便于自动处理——有错误就写 `ERROR` 并在同一行写明原因以便 grep。预计算汇总统计，避免 Claude 重复计算。
- **时间盲区**：Claude 无法感知时间，会花数小时跑测试而非推进进度。框架少量打印增量进度，并提供 `--fast` 选项运行 1% 或 10% 的随机子样本——子样本对每个智能体确定性但跨 VM 随机，确保覆盖全部文件的同时每个智能体能精确定位回归。

此外，每个智能体被放入全新容器，缺乏上下文。为此，作者在提示中要求 Claude 频繁更新 README 和进度文件。

### 让并行变得容易

当存在大量独立失败测试时，并行化轻而易举——每个智能体选不同测试修复。测试套件达 99% 通过率后，各智能体分别攻克不同小型开源项目（SQLite、Redis、libjpeg、MQuickJS、Lua）。

但当开始编译 Linux 内核时陷入困境：编译内核是一个巨大整体任务，每个智能体都撞上同样的 bug，修复后互相覆盖。16 个智能体同时运行毫无帮助。

解决方案是利用 GCC 作为在线参照编译器 (oracle)：随机用 GCC 编译大部分内核文件，仅用 Claude 的编译器编译剩余文件。若内核能运行则问题不在 Claude 编译的部分；若失败则将部分文件换回 GCC 编译进一步缩小范围。这让每个智能体能并行修复不同文件中的不同 bug。之后还需用增量调试 (delta debugging) 技术找到单独可编译但组合会失败的文件对。

### 多智能体角色

并行也使专业化成为可能：一个智能体负责合并重复代码，一个负责编译器自身性能，一个负责生成高效编译输出，一个从 Rust 开发者角度审查项目设计并做结构性改进，另一个负责文档。

## 压力测试智能体团队的极限

此项目本质是能力基准测试 (capability benchmark)，旨在探索 LLM 当前勉强能达成的极限，从而为未来模型将可靠达成的目标做准备。

作者在 Claude 4 全系列模型上使用此项目做基准测试。之前 Opus 4 勉强能产出功能编译器；Opus 4.5 是首个跨过门槛能通过大型测试套件的版本，但仍无法编译真正的大型项目。Opus 4.6 再次测试极限。

### 评估

近 2,000 次 Claude Code 会话、两周时间、20 亿输入 token、1.4 亿输出 token，总花费略低于 20,000 美元——虽远超最贵的 Claude Max 订阅计划，但与人工团队成本相比微不足道。

编译器为纯净室实现 (clean-room implementation)，开发期间 Claude 无互联网访问，仅依赖 Rust 标准库。10 万行代码，可在 x86/ARM/RISC-V 上编译可启动的 Linux 6.9，也能编译 QEMU、FFmpeg、SQLite、PostgreSQL、Redis，GCC torture 测试套件通过率 99%，且能编译运行 Doom。

**局限**：缺少 16 位 x86 编译器（调用 GCC 代替）、无自研汇编器和链接器（仍使用 GCC 的）、并非所有项目都能编译、生成代码效率远低于 GCC 关闭优化时的水平、Rust 代码质量不及专业开发者。

编译器已接近 Opus 能力极限。作者曾努力修复上述局限但未完全成功——新功能和修复频繁破坏已有功能。一个特别棘手的例子：Opus 无法实现 16 位 x86 代码生成器以满足 Linux 的 32k 代码限制（Claude 生成的输出超 60kb）。

## 展望

每一代语言模型都开启新的协作方式——从 IDE 自动补全到函数体补全，再到与 Claude 配对编程。但这些产品都假设：用户定义任务、LLM 运行数秒或数分钟返回结果、用户提供后续输入。

智能体团队展示了自主实现整个复杂项目的可能性。然而全自动开发仍有真实风险——无人监督时，很容易看到测试通过就认定工作完成，而这通常并非如此。作者曾有渗透测试经验，对程序员部署自己从未验证过的软件深感忧虑。

作者对进展之快既兴奋又不安，认为需要新的策略来安全地驾驭这一新范式。
