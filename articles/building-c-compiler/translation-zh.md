# 用并行 Claude 团队构建 C 编译器

发布于 2026 年 2 月 5 日
作者：Nicholas Carlini，Anthropic 安全团队研究员。

我一直在实验一种监管语言模型的新方法，我们称之为"智能体团队"(agent teams)。

在智能体团队中，多个 Claude 实例并行工作于共享代码库，无需人工主动干预。这种方法极大地扩展了 LLM 智能体所能达成的范围。

为了进行压力测试，我让 16 个智能体从零编写一个基于 Rust 的 C 编译器，目标是能够编译 Linux 内核。历经近 2,000 次 Claude Code 会话和 20,000 美元的 API 费用，智能体团队产出了一个 100,000 行的编译器，能够在 x86、ARM 和 RISC-V 上编译 Linux 6.9。

[这个编译器本身是一个有趣的作品](https://github.com/anthropics/claudes-c-compiler)，但我在此重点关注的是我在设计长时间运行自主智能体团队的框架 (harness) 方面学到的经验：如何编写测试来让智能体在无人监督下保持在正轨上，如何组织工作使多个智能体能够并行推进，以及这种方法的天花板在哪里。

---

## 实现长时运行的 Claude

现有的智能体框架 (agent scaffolds) 如 Claude Code 需要操作者在线并可供协作。如果你要求解决一个长期且复杂的问题，模型可能解决其中一部分，但最终会停下来等待后续输入——一个问题、一个状态更新，或一个澄清请求。

为了激发持续的自主进展，我构建了一个将 Claude 置于简单循环中的框架（如果你见过 Ralph-loop，这应该看起来很熟悉）。当它完成一个任务后，立即接取下一个。*（请在容器中运行此脚本，而不是在你的实际机器上）。*

```bash
#!/bin/bash

while true; do
    COMMIT=$(git rev-parse --short=6 HEAD)
    LOGFILE="agent_logs/agent_${COMMIT}.log"

    claude --dangerously-skip-permissions \
           -p "$(cat AGENT_PROMPT.md)" \
           --model claude-opus-X-Y &> "$LOGFILE"
done
```

在智能体提示 (agent prompt) 中，我告诉 Claude 要解决什么问题，并要求它通过将问题拆分为小片段、追踪正在做的工作、确定下一步该做什么的方式来推进，并有效地持续前进直到完美。（关于最后这一点，Claude 没有选择。循环永远运行——尽管有一次，我确实看到 Claude 意外执行了 `pkill -9 bash`，从而杀死了自己并终止了循环。哎呀！）

## 并行运行 Claude

并行运行多个实例可以解决单智能体框架的两个弱点：

1. 一个 Claude Code 会话同一时间只能做一件事。特别是随着项目范围扩大，并行调试多个问题效率要高得多。
2. 运行多个 Claude 智能体可以实现专业化 (specialization)。当几个智能体被指派解决当前的实际问题时，其他专业化智能体可以被调用来（例如）维护文档、关注代码质量，或解决专业化子任务。

我的并行 Claude 实现是非常基础的。创建一个新的裸 git 仓库，为每个智能体启动一个 Docker 容器，仓库挂载到 `/upstream`。每个智能体将本地副本克隆到 `/workspace`，完成后从自己的本地容器推送到上游。

为了防止两个智能体同时尝试解决同一问题，框架使用了一个简单的同步算法：

1. Claude 通过在 `current_tasks/` 目录中写入文本文件来对任务"加锁"（例如，一个智能体可能锁定 `current_tasks/parse_if_statement.txt`，而另一个锁定 `current_tasks/codegen_function_definition.txt`）。如果两个智能体试图认领同一任务，git 的同步机制会迫使第二个智能体选择不同的任务。
2. Claude 完成任务后，从上游拉取、合并其他智能体的变更、推送自己的变更，并移除锁。合并冲突很频繁，但 Claude 足够聪明能够自行解决。
3. 无限智能体生成循环在新的容器中生成新的 Claude Code 会话，循环重复。

这是一个非常早期的研究原型。我还没有实现任何其他智能体间通信方法，也没有强制执行任何管理高层目标的流程。我没有使用编排智能体 (orchestration agent)。

相反，我把如何行动的决定权留给每个 Claude 智能体。在大多数情况下，Claude 会选择"最明显的下一个"问题。当卡在某个 bug 上时，Claude 经常会维护一个记录失败方法和剩余任务的运行文档。在项目的 [git 仓库](https://github.com/anthropics/claudes-c-compiler)中，你可以阅读历史记录，观察它如何对各种任务加锁。

## 与 Claude 智能体团队编程的经验教训

框架将 Claude 放入循环中运行，但只有当 Claude 能够判断如何取得进展时，这个循环才有用。我的大部分精力都花在设计 Claude 周围的环境——测试、环境、反馈——以便它能够在没有我的情况下自行定位。这些是我在编排多个 Claude 实例时发现最有帮助的方法。

### 编写极高质量的测试

Claude 会自主工作来解决我给它的任何问题。因此任务验证器 (task verifier) 必须近乎完美，否则 Claude 会解决错误的问题。改进测试框架需要寻找高质量的编译器测试套件、为开源软件包编写验证器和构建脚本，以及观察 Claude 正在犯的错误，然后在识别出那些失败模式后设计新的测试。

例如，在项目接近尾声时，Claude 每次实现新功能时都频繁破坏已有功能。为了解决这个问题，我构建了持续集成 (CI) 流水线，并实施了更严格的执行策略，使 Claude 能够更好地测试自己的工作，从而让新提交不能破坏已有代码。

### 站在 Claude 的角度思考

我必须不断提醒自己，我是在为 Claude 而不是为自己编写这个测试框架，这意味着要重新思考许多关于测试应该如何传达结果的假设。

例如，每个智能体被放入一个没有上下文的全新容器中，会花大量时间来定位自己，特别是在大型项目上。在我们甚至还没到测试之前，为了帮助 Claude 自助，我在提示中包含了维护大量 README 和进度文件的说明，这些文件应该随着当前状态频繁更新。

我也牢记语言模型存在固有限制这一事实，在这种情况下，需要围绕这些限制进行设计。这些限制包括：

- **上下文窗口污染 (Context window pollution)：** 测试框架不应打印数千字节的无用输出。最多，它应该打印几行输出，并将所有重要信息记录到文件中，以便 Claude 在需要时能找到。日志文件应该便于自动处理：如果有错误，Claude 应该写 `ERROR` 并在同一行写明原因，以便 grep 能找到它。预计算汇总统计信息很有帮助，这样 Claude 就不必重新计算。
- **时间盲区 (Time blindness)：** Claude 无法感知时间，如果放任不管，它会愉快地花数小时运行测试而不是推进进度。框架不频繁地打印增量进度（以避免污染上下文），并包含一个默认的 `--fast` 选项，运行 1% 或 10% 的随机样本。这个子样本对每个智能体是确定性的，但跨 VM 是随机的，因此 Claude 仍然覆盖所有文件，但每个智能体都能完美识别回归。

### 让并行变得容易

当存在许多不同的失败测试时，并行化是轻而易举的：每个智能体选择一个不同的失败测试来修复。在测试套件达到 99% 通过率后，每个智能体致力于让不同的小型开源项目（例如 SQLite、Redis、libjpeg、MQuickJS、Lua）能够编译。

但是当智能体开始编译 Linux 内核时，它们卡住了。与拥有数百个独立测试的测试套件不同，编译 Linux 内核是一个巨大的单一任务。每个智能体都会撞上同一个 bug，修复那个 bug，然后互相覆盖对方的更改。运行 16 个智能体没有帮助，因为每个都卡在解决同一任务上。

解决方案是使用 [GCC](https://gcc.gnu.org/) 作为在线已知正确编译器参照 (online known-good compiler oracle) 来进行比较。我编写了一个新的测试框架，随机用 GCC 编译大部分内核，仅用 Claude 的 C 编译器编译剩余的文件。如果内核能工作，那么问题不在 Claude 编译的那部分文件中。如果出问题，它可以进一步细化，用 GCC 重新编译其中一些文件。这让每个智能体能够并行工作，修复不同文件中的不同 bug，直到 Claude 的编译器最终能编译所有文件。（在此方法生效后，还需要应用增量调试 (delta debugging) 技术来找到单独可编译但组合会失败的文件对。）

### 多智能体角色

并行也使专业化成为可能。LLM 编写的代码经常重新实现已有功能，所以我指派一个智能体来合并它发现的任何重复代码。我让另一个智能体负责提高编译器本身的性能，第三个智能体负责输出高效的编译代码。我让另一个智能体从 Rust 开发者的角度审查项目设计，并做结构性更改以提高整体代码质量，还有一个负责文档工作。

## 压力测试智能体团队的极限

此项目被设计为能力基准测试 (capability benchmark)。我感兴趣的是压力测试 LLM 今天*勉强*能达成的极限，以帮助我们为未来模型将可靠达成的目标做准备。

我一直在使用 C 编译器项目作为整个 Claude 4 模型系列的基准测试。正如我在之前的项目中所做的，我首先起草了我想要的：一个从零开始的优化编译器，无依赖，GCC 兼容，能够编译 Linux 内核，并设计为支持多个后端。虽然我指定了设计的某些方面（例如，它应该有 SSA 中间表示 (IR) 以支持多个优化遍），但我没有详细说明如何实现。

之前的 Opus 4 模型勉强能够产出功能编译器。Opus 4.5 是第一个跨过门槛的版本，能够产出通过大型测试套件的功能编译器，但仍然无法编译任何真正的大型项目。我在 Opus 4.6 上的目标再次是测试极限。

### 评估

在近 2,000 次 Claude Code 会话、两周的时间里，Opus 4.6 消耗了 20 亿输入 token 和 1.4 亿输出 token，总成本略低于 20,000 美元。与即使最昂贵的 Claude Max 计划相比，这也是一个极其昂贵的项目。但这个总额只是我自己产出这个编译器所需成本的一小部分——更不用说整个团队了。

这是一个纯净室实现 (clean-room implementation)（Claude 在开发期间的任何时候都没有互联网访问）；它仅依赖 Rust 标准库。这个 100,000 行的编译器可以在 x86、ARM 和 RISC-V 上编译可启动的 Linux 6.9。它还能编译 QEMU、FFmpeg、SQLite、PostgreSQL、Redis，并在大多数编译器测试套件上有 99% 的通过率，包括 [GCC torture 测试套件](https://gcc.gnu.org/onlinedocs/gccint/Torture-Tests.html)。它也通过了开发者的终极试金石：它能编译并运行 Doom。

然而，这个编译器并非没有局限性。这些局限性包括：

- 它缺少启动 Linux 从实模式 (real mode) 引导所需的 16 位 x86 编译器。为此它调用 GCC（x86_32 和 x86_64 编译器是它自己的）。
- 它没有自己的汇编器 (assembler) 和链接器 (linker)；这些是 Claude 最后开始自动化的部分，目前仍有一些 bug。演示视频是使用 GCC 汇编器和链接器制作的。
- 编译器成功构建了许多项目，但并非全部。它还不是一个可以即插即用替代真正编译器的方案。
- 生成的代码效率不高。即使启用了所有优化，它输出的代码效率也不如 GCC 在*禁用*所有优化时的输出。
- Rust 代码质量是合理的，但远不及专业 Rust 程序员可能产出的质量。

生成的编译器几乎已经达到了 Opus 能力的极限。我（非常努力地！）尝试修复上述几个局限性但没有完全成功。新功能和错误修复频繁破坏已有功能。

作为一个特别具有挑战性的例子，Opus 无法实现启动进入 16 位实模式所需的 16 位 x86 代码生成器。虽然编译器可以通过 66/67 操作码前缀 (opcode prefixes) 输出正确的 16 位 x86 代码，但生成的编译输出超过 60kb，远远超过 Linux 强制执行的 32k 代码限制。因此 Claude 在这里只是取巧，在这个阶段调用 GCC（这仅适用于 x86。对于 ARM 或 RISC-V，Claude 的编译器可以完全自行编译。）

[编译器的源代码已公开](https://github.com/anthropics/claudes-c-compiler)。下载它，阅读代码，并在你喜欢的 C 项目上试用它。我一直发现理解语言模型能做什么的最好方法是把它们推向极限，然后研究它们开始崩溃的地方。在接下来的几天里，如果你想跟随 Claude 继续尝试解决这些局限性的进展，我会继续让 Claude 推送新的更改。

## 展望未来

每一代语言模型都开启了与它们协作的新方式。早期模型在 IDE 中用于 tab 补全 (tab-completion) 很有用。不久之后，模型可以从文档字符串 (docstring) 补全函数体。Claude Code 的推出将智能体 (agents) 带入主流，使开发者能够与 Claude 配对编程。但这些产品都运行在一个假设之下：用户定义任务，LLM 运行几秒或几分钟后返回答案，然后用户提供后续输入。

智能体团队展示了自主实现整个复杂项目的可能性。这使我们作为这些工具的用户，能够对目标更加雄心勃勃。

我们仍处于早期阶段，完全自主的开发伴随着真实的风险。当人类在开发过程中与 Claude 并肩工作时，他们可以确保一致的质量并实时捕获错误。对于自主系统，很容易看到测试通过就认为工作完成了，而这通常并非如此。我以前从事渗透测试 (penetration testing) 工作，利用大型公司产品中的漏洞，程序员部署他们从未亲自验证过的软件这种想法是一个真正的担忧。

所以，虽然这个实验让我兴奋，但也让我感到不安。构建这个编译器是我最近最有趣的经历之一，但我没有预料到在 2026 年初就能达到如此接近可能的地步。语言模型和我们与之交互的框架的快速进步，为编写大量新代码打开了大门。我期望积极的应用将超过消极的应用，但我们正在进入一个需要新策略来安全导航的新世界。
