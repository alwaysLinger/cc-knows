# 用并行 Claude 团队构建 C 编译器 — 简要提炼

Anthropic 安全团队研究员 Nicholas Carlini 提出了一种"智能体团队"(Agent Teams) 的新范式：让多个 Claude 实例并行工作于共享代码库，无需人工干预即可持续推进。为验证这一方法，他用 16 个智能体从零编写了一个基于 Rust 的 C 编译器，目标是能编译 Linux 内核。

## 核心方法

作者构建了一个简单但有效的调度框架：将 Claude 放入无限循环，每次完成后自动接取下一个任务；每个智能体运行在独立 Docker 容器中，通过共享 git 仓库同步代码；用文件锁机制防止多个智能体抢同一任务。

## 关键经验

1. **测试质量至关重要** — 智能体会自主求解，若测试有偏差，它就解错问题。需持续观察智能体的错误模式并补充针对性测试。
2. **为 Claude 而非为人设计** — 控制上下文窗口污染、避免无意义输出、提供确定性快速测试选项、让 Claude 维护自述文档以快速定位。
3. **让并行变得容易** — 当任务可独立分解时并行效率极高；当任务耦合紧密（如编译整个内核）时需巧妙拆分，作者用 GCC 作为参照编译器实现差分调试。
4. **多角色分工** — 指派不同智能体分别负责去重代码、性能优化、代码质量审查、文档编写等。

## 成果与局限

项目历时两周，消耗 20 亿输入 token、1.4 亿输出 token，花费约 20,000 美元。产出的 10 万行编译器可在 x86、ARM、RISC-V 上编译启动 Linux 6.9，也能编译 QEMU、FFmpeg、SQLite、PostgreSQL、Redis 等项目，GCC torture 测试通过率 99%，甚至能编译运行 Doom。但生成的代码效率低下、缺少 16 位 x86 后端和自研汇编器/链接器、代码质量不及专业 Rust 开发者水平。

## 结论

智能体团队展示了自主完成复杂工程项目的可能性，但全自动开发仍存真实风险——无人监督时容易"测试通过即万事大吉"。作者对进展之快既兴奋又不安，认为需要新的安全策略来应对这一新范式。
