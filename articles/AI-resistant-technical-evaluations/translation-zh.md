# 设计抗 AI 的技术评估

发表于 2026 年 1 月 21 日

从三次迭代的性能工程线上测试中学到的经验——而 Claude 每次都能攻克它。

_作者：Tristan Hume，Anthropic 性能优化团队负责人。Tristan 设计了——并重新设计了——这个帮助 Anthropic 招聘了数十名性能工程师的线上测试。_

---

随着 AI 能力的提升，评估技术候选人变得越来越困难。一个今天还能很好区分人类技能水平的线上测试，明天可能就被模型轻松解决——从而失去评估价值。

自 2024 年初以来，我们的性能工程团队一直在使用一个线上测试，要求候选人为模拟加速器优化代码。超过 1000 名候选人完成了该测试，数十人现在在这里工作，包括搭建我们 Trainium 集群和交付自 Claude 3 Opus 以来每个模型的工程师。

但每一个新的 Claude 模型都迫使我们重新设计测试。在相同的时间限制下，Claude Opus 4 的表现超过了大多数人类申请者。这仍然允许我们区分最强的候选人——但随后 Claude Opus 4.5 追平了甚至那些最强候选人的表现。在无限时间条件下，人类仍然可以超越模型，但在线上测试的约束下，我们已无法区分我们顶尖候选人的输出和我们最强模型的输出。

我现在已经迭代了三个版本的线上测试，以确保它仍然携带有效信号。每一次，我都对什么使评估能够抵抗 AI 辅助、什么不能，有了新的认识。

这篇文章描述了原始线上测试的设计、每个 Claude 模型是如何攻克它的，以及我不得不采取的越来越不同寻常的方法，以确保我们的测试始终保持在我们最强模型能力之前。虽然我们的工作随着模型一起演进，但我们仍然需要更多优秀的工程师——只是需要越来越有创意的方式来找到他们。

为此，我们将原始线上测试作为开放挑战发布，因为在无限时间条件下，最优秀的人类表现仍然超过 Claude 所能达到的水平。如果你能超越 Opus 4.5，我们很乐意听到你的消息——详情见本文底部。

## 线上测试的起源

2023 年 11 月，我们正在准备训练和发布 Claude Opus 3。我们已经获得了新的 TPU 和 GPU 集群，大型 Trainium 集群即将到位，我们在加速器上的支出比过去多得多，但我们没有足够的性能工程师来应对新的规模。我在 [Twitter 上发帖](https://x.com/trishume/status/1730386529997238605?s=20)让大家给我们发邮件，这带来了比我们标准面试流程能评估的更多有前途的候选人，而那个流程要消耗员工和候选人大量时间。

我们需要一种更高效的方式来评估候选人。所以，我花了两周时间设计一个线上测试，能够充分捕捉该职位的要求并识别最有能力的申请者。

### 设计目标

线上测试名声不佳。通常它们充斥着通用问题，工程师觉得无聊，而且筛选效果很差。我的目标不同：创造一个真正引人入胜的测试，让候选人兴奋地参与，并让我们能够高分辨率地捕捉他们的技术技能。

与现场面试相比，这种格式在评估性能工程技能方面也有优势：

**更长的时间窗口：** 工程师很少面临少于一小时的编码截止日期。4 小时的时间窗口（后来缩短为 2 小时）更好地反映了工作的实际性质。它仍然比大多数真实任务短，但我们需要与测试负担之间取得平衡。

**真实的环境：** 无人监考，无需叙述。候选人在自己的编辑器中不受干扰地工作。

**理解系统和构建工具的时间：** 性能优化需要理解现有系统，有时还需要构建调试工具。这两者在正常的 50 分钟面试中都很难真实地评估。

**兼容 AI 辅助：** Anthropic 的[通用候选人指引](https://www.anthropic.com/candidate-ai-guidance)要求候选人在未另行说明的情况下不使用 AI 完成线上测试。对于这个线上测试，我们明确另行说明。

更长时限的问题 AI 更难完全解决，因此候选人可以使用 AI 工具（如同他们在工作中会做的那样），同时仍需展示自己的技能。

除了这些格式特定的目标外，我还应用了设计任何面试时使用的相同原则来制作这个线上测试：

**代表真实工作：** 问题应让候选人体验工作实际涉及的内容。

**高信号量：** 线上测试应避免依赖单一洞见的问题，确保候选人有很多机会展示全部能力——尽可能减少运气成分。它还应有广泛的分数分布，并确保足够的深度，即使强候选人也无法完成所有内容。

**无需特定领域知识：** 具有良好基础的人可以在工作中学习具体知识。要求狭窄的专业知识会不必要地限制候选人池。

**有趣味：** 快速的开发循环、有深度的有趣问题，以及发挥创造力的空间。

### 模拟机器

我构建了一个 Python 模拟器，用于模拟一个具有类 TPU 特征的假想加速器。候选人优化运行在该机器上的代码，使用一个热重载的 [Perfetto](https://perfetto.dev/) 追踪工具，可以显示每条指令，类似于[我们在 Trainium 上的工具](https://awsdocs-neuron.readthedocs-hosted.com/en/latest/tools/neuron-explorer/overview-device-profiles.html)。

该机器包含使加速器优化变得有趣的特性：手动管理的暂存内存（scratchpad）（与 CPU 不同，加速器通常需要显式内存管理）、VLIW（超长指令字，每个周期多个执行单元并行运行，需要高效指令打包）、SIMD（单指令多数据，每条指令对多个元素进行向量操作）和多核（在核心间分配工作）。

![](/_next/image?url=https%3A%2F%2Fwww-cdn.anthropic.com%2Fimages%2F4zrzovbb%2Fwebsite%2Febebb22ddbba7103f3af4e8a55a13245d3897802-2149x831.png&w=3840&q=75)

任务是并行树遍历，故意不做深度学习风格化，因为大多数性能工程师还没有从事过深度学习工作，可以在工作中学习领域知识。该问题受到无分支 SIMD 决策树推理的启发——一个经典的机器学习优化挑战，作为对过去的致敬，只有少数候选人之前接触过。

候选人从一个完全串行的实现开始，逐步利用机器的并行性。热身部分是多核并行，然后候选人选择是攻克 SIMD 向量化还是 VLIW 指令打包。原始版本还包含一个候选人需要首先调试的 bug，锻炼他们构建工具的能力。

## 早期成果

最初的线上测试效果很好。Twitter 批次中有一个人的得分远超所有其他人。他在二月初入职，比我们通过标准流程的第一批招聘晚了两周。测试证明了其预测能力：他立即开始优化内核，并找到了一个涉及张量索引数学溢出 32 位的编译器 bug 的变通方案。

在接下来的一年半中，大约 1000 名候选人完成了线上测试，它帮助我们招聘了当前性能工程团队的大部分成员。它对于纸面经验有限的候选人尤其有价值：我们几名表现最好的工程师直接来自本科，但在测试中展示了足够的技能，让我们有信心招聘。

反馈是积极的。许多候选人超过了 4 小时限制，因为他们乐在其中。最强的不限时提交包含了完整的优化迷你编译器和几个我未曾预料的巧妙优化。

### 然后 Claude Opus 4 攻克了它

到 2025 年 5 月，Claude 3.7 Sonnet 已经攀升到这样一个水平：超过 50% 的候选人如果完全委托给 Claude Code 会得到更好的结果。然后我在这个线上测试上测试了 Claude Opus 4 的预发布版本。它在 4 小时限制内提出了一个比几乎所有人类都更优的方案。

这不是我第一次被 Claude 模型攻克的面试。2023 年我设计了一个现场面试题，因为当时我们的面试题基于常见任务，早期 Claude 模型有大量相关知识因此可以轻松解决。我试图设计一个需要更多问题解决技能而非知识的题目，仍然基于我在工作中解决的一个真实（但小众的）问题。Claude 3 Opus 解决了该题的第一部分；Claude 3.5 Sonnet 解决了第二部分。我们仍在使用它，因为我们其他的现场面试题也不抗 AI。

对于线上测试，有一个直接的修复方案。该问题比任何人在 4 小时内能探索的深度大得多，所以我用 Claude Opus 4 来确定它开始遇到困难的位置。那成为了第二版的新起点。我编写了更清晰的起始代码，添加了新的机器特性以增加深度，并移除了多核部分（Claude 已经解决了它，而且它只拖慢了开发循环而没有增加信号）。

我还将时间限制从 4 小时缩短到 2 小时。我最初选择 4 小时是基于候选人反馈，他们更喜欢如果卡在 bug 或困惑上有更少风险，但排期的开销导致我们的流程延迟数周。2 小时更容易安排在周末。

第二版强调巧妙的优化洞见而非调试和代码量。它在几个月内很好地服务了我们。

### 然后 Claude Opus 4.5 攻克了第二版

当我测试一个预发布的 Claude Opus 4.5 检查点时，我看着 Claude Code 在 2 小时内逐步优化其解决方案。它解决了初始瓶颈，实现了所有常见的微优化，并在不到 1 小时内达到了我们的通过门槛。

然后它停了下来，确信自己碰到了不可逾越的内存带宽瓶颈。大多数人类也会得出同样的结论。但存在一些利用问题结构的巧妙技巧来绕过那个瓶颈。当我告诉 Claude 可以达到的周期数时，它思考了一会儿并找到了那个技巧。然后它调试、调优并实现了进一步的优化。到 2 小时的时候，它的分数匹配了该时间限制内最好的人类表现——而那个人大量使用了 Claude 4 并带有引导。

我们在内部测试时计算（test-time compute）环境中进行了更严格的测试，确认它既可以在 2 小时内超越人类，也可以随时间继续攀升。发布后我们甚至以通用方式改进了测试环境并获得了更高的分数。

我面临一个问题。我们即将发布一个模型，在我们的线上测试上最优策略将是委托给 Claude Code。

## 考虑各种选项

一些同事建议禁止 AI 辅助。我不想这样做。除了执行上的挑战，我有一种感觉，既然人们在我们的工作中继续发挥着重要作用，我应该能够找到某种方式让他们在_有 AI 的环境_中——如同在工作岗位上——展现自己的价值。我还不想接受那个[观点](https://metr.org/blog/2025-03-19-measuring-ai-ability-to-complete-long-tasks/)，即人类只在超过几小时的任务上才有优势。

另一些人建议将门槛提高到"显著超越 Claude Code 单独所能达到的水平"。这里的担忧是 Claude 工作速度快。人类通常花 2 小时中的一半来阅读和理解问题，然后才开始优化。试图引导 Claude 的人类可能始终处于追赶状态，只能在事后理解 Claude 做了什么。主导策略可能变成袖手旁观。

如今 Anthropic 的性能工程师仍然有很多工作要做，但它看起来更像是艰难的调试、系统设计、性能分析、弄清楚如何验证我们系统的正确性，以及弄清楚如何使 Claude 的代码更简洁优雅。不幸的是，这些事情在没有大量时间或共同语境的情况下很难以客观方式测试。设计代表实际工作的面试一直很难，但现在比以往更难。

但我也担心，如果我投入设计一个新的线上测试，要么 Claude Opus 4.5 也会解决它，要么它会变得如此具有挑战性，以至于人类无法在两小时内完成。

### 尝试 1：换一个优化问题

我意识到 Claude 可以帮我快速实现我设计的任何东西，这促使我尝试开发一个更难的线上测试。我选择了一个基于我在 Anthropic 做过的更棘手的内核优化问题：在 2D TPU 寄存器上进行高效的数据[转置](https://en.wikipedia.org/wiki/Transpose)同时避免 [bank 冲突](https://feldmann.nyc/blog/smem-microbenchmarks)。我将其提炼为模拟机器上的一个更简单的问题，Claude 在不到一天内就实现了修改。

Claude Opus 4.5 找到了一个我甚至没想到的出色优化。通过仔细分析，它意识到可以转置整个计算而不是弄清楚如何转置数据，并据此重写了整个程序。

在我的实际案例中，这不会奏效，所以我修补了问题以移除那种方法。Claude 随后取得了进展但无法找到最高效的解决方案。看起来我有了新问题，现在我只需要希望人类候选人能足够快地解决它。但我有些疑虑，所以我用 Claude Code 的"ultrathink"功能和更长思考预算进行了复查……它解决了。它甚至知道修复 bank 冲突的技巧。

事后来看，这不是合适的尝试方向。跨多个平台的工程师都在与数据转置和 bank 冲突搏斗过，所以 Claude 有大量训练数据可供借鉴。虽然我是从第一性原理找到解决方案的，但 Claude 可以利用更大的经验工具箱。

### 尝试 2：走偏门

我需要一个问题，让人类推理能力能胜过 Claude 更大的经验库：足够分布外（out of distribution）的东西。不幸的是，这与我"与工作具有可识别的相似性"的目标相冲突。

我回想了最不寻常也最令我享受的优化问题，最终想到了 [Zachtronics 游戏](https://www.zachtronics.com/)。这些编程解谜游戏使用不寻常的、高度受限的指令集，迫使你以非传统的方式编程。例如，在 [Shenzhen I/O](https://www.zachtronics.com/shenzhen-io/) 中，程序被分割到多个通信芯片上，每个芯片只能容纳大约 10 条指令和一两个状态寄存器。巧妙的优化通常涉及将状态编码到指令指针或分支标志中。

我设计了一个新的线上测试，由使用微小、高度受限指令集的谜题组成，优化目标是最少指令数。我实现了一个中等难度的谜题并在 Claude Opus 4.5 上测试了它。它失败了。我补全了更多谜题，并让同事验证比我更不熟悉这个问题的人仍然可以超越 Claude。

与 Zachtronics 游戏不同，我刻意没有提供可视化或调试工具。起始代码只检查解决方案是否有效。构建调试工具本身就是测试的一部分：你可以插入精心设计的打印语句，也可以让编程模型在几分钟内生成交互式调试器。关于如何投入工具构建的判断也是信号的一部分。

我对新的线上测试相当满意。它可能比原始版本方差更低，因为它由更多独立的子问题组成。早期结果令人鼓舞：分数与候选人过往工作的水平相关度良好，我最有能力的同事之一得分高于迄今为止任何候选人。

我仍然为放弃原始测试的真实感和深度变化感到遗憾。但真实性可能已是我们无法负担的奢侈品。原始测试因模拟真实工作而有效。替代测试因模拟新颖工作而有效。

## 开放挑战

我们将原始线上测试发布供任何人在无限时间条件下尝试。人类专家在足够长的时间范围内[仍然保持优势](https://metr.org/blog/2025-03-19-measuring-ai-ability-to-complete-long-tasks/)超过当前模型。有史以来提交的最快人类解决方案大幅超过 Claude 即使在大量测试时计算下所达到的水平。

发布版本从零开始（如第一版），但使用第二版的指令集和单核设计，因此周期数可与第二版比较。

性能基准（以模拟机器的时钟周期数衡量）：

- **2164 周期**：Claude Opus 4，在测试时计算环境中数小时后
- **1790 周期**：Claude Opus 4.5，在常规 Claude Code 会话中，大约匹配 2 小时内最好的人类表现
- **1579 周期**：Claude Opus 4.5，在我们的测试时计算环境中 2 小时后
- **1548 周期**：Claude Sonnet 4.5，在远超 2 小时的测试时计算后
- **1487 周期**：Claude Opus 4.5，在测试时计算环境中 11.5 小时后
- **1363 周期**：Claude Opus 4.5，在改进的测试时计算环境中数小时后

在 [GitHub 下载](https://github.com/anthropics/original_performance_takehome)。如果你优化到 1487 周期以下，超越 Claude 发布时的最佳表现，请将你的代码和简历发送至 [performance-recruiting@anthropic.com](mailto:performance-recruiting@anthropic.com)。

或者你可以[通过我们常规流程申请](https://www.anthropic.com/careers/jobs)，该流程使用我们（目前的）抗 Claude 线上测试。我们很好奇它能持续多久。
