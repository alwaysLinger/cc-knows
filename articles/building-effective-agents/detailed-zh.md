# 构建有效智能体 — 详细摘要

## 什么是智能体？

"智能体"定义因人而异：有人指完全自主、长期运行、使用多种工具完成复杂任务的系统；有人指遵循预定义工作流的更规范实现。Anthropic 将所有变体统称为**智能体系统（agentic systems）**，但在架构上区分两个概念：

- **工作流（Workflows）**：LLM 和工具通过预定义代码路径编排。
- **智能体（Agents）**：LLM 动态掌控自身流程和工具使用，自主决定如何完成任务。

## 何时使用（何时不使用）智能体

核心建议：**寻找最简方案，仅在必要时增加复杂度**。这意味着有时根本不需要构建智能体系统。智能体系统通常以延迟和成本换取更好的任务表现，需审慎权衡此取舍。若需更多复杂度，工作流适合定义明确的任务（可预测、一致），智能体适合需要灵活性和模型驱动决策的大规模场景。对多数应用而言，优化单次 LLM 调用加上检索和上下文示例通常已足够。

## 框架的选择

文中列举了 Claude Agent SDK、AWS Strands Agents SDK、Rivet、Vellum 等框架。框架简化了调用 LLM、定义工具、链式调用等底层任务，但额外抽象层可能掩盖底层提示和响应、增加调试难度，也容易诱使开发者在简单方案即可时添加不必要的复杂度。建议：先用 LLM API 直接实现——许多模式几行代码即可完成；若用框架，务必理解底层代码，错误假设是客户常见问题根源。

## 基础构建块：增强型 LLM

智能体系统的基础构建块是经过增强的 LLM——加入检索、工具和记忆等能力。当前模型可主动生成搜索查询、选择工具、决定保留哪些信息。实现关键：将能力针对具体用例定制，确保为 LLM 提供简洁、文档完善的接口。Model Context Protocol（MCP）是一种实现方式，允许通过简单客户端集成日益增长的第三方工具生态。

## 工作流模式

### 提示链（Prompt Chaining）

将任务分解为顺序步骤，每个 LLM 调用处理上一步的输出，可在中间步骤添加程序检查（"gate"）确保流程正确。适用于任务可清晰分解为固定子任务的场景，以延迟换精度。示例：先生成营销文案再翻译；先写提纲、检查提纲是否达标、再据提纲写文档。

### 路由（Routing）

对输入分类后导向专门的后续处理，实现关注点分离。若无路由，为某类输入优化可能损害其他输入的表现。适用于有明确类别且分类可准确完成的复杂任务。示例：将客服查询（一般问题、退款、技术支持）导向不同流程；将简单问题路由至低成本模型（如 Claude Haiku 4.5），难题路由至高能力模型（如 Claude Sonnet 4.5）。

### 并行化（Parallelization）

LLM 同时处理任务并以程序聚合输出。两种变体：**分段（Sectioning）**——将任务拆为独立子任务并行执行；**投票（Voting）**——同一任务多次执行以获得多样输出。适用于子任务可并行加速，或需要多视角/多次尝试以提高结果可信度。分段示例：一个模型处理用户查询，另一个并行筛查不当内容；自动化评估中各 LLM 调用评估不同方面。投票示例：多提示审查代码漏洞；多提示从不同角度评估内容是否不当。

### 编排者-工作者（Orchestrator-Workers）

中央 LLM 动态分解任务、委派给工作 LLM、综合结果。与并行化的关键区别在于灵活性——子任务非预定义，由编排者根据输入动态决定。适用于无法预判所需子任务的复杂任务。示例：需修改多文件的编码产品；需从多源收集分析信息的搜索任务。

### 评估者-优化者（Evaluator-Optimizer）

一个 LLM 生成响应，另一个提供评估和反馈，循环迭代。适用于有明确评估标准且迭代改进可带来可衡量价值的场景。两个好迹象：人类表达反馈后 LLM 响应明显改善；LLM 自身能提供此类反馈。示例：文学翻译中评估 LLM 提供细致批评；复杂搜索任务中评估者决定是否需进一步搜索。

## 智能体

随着 LLM 在理解复杂输入、推理规划、可靠使用工具和错误恢复等关键能力上日趋成熟，智能体开始在产品中涌现。智能体从人类指令或交互讨论开始，明确任务后自主规划和执行，可能在检查点或遇阻时回到人类寻求反馈。执行中，从环境获取"真实信号"（如工具调用结果、代码执行输出）以评估进展至关重要。任务完成时终止，也可设停止条件（如最大迭代次数）以保持控制。

智能体可处理复杂任务，但实现往往简单——通常就是 LLM 基于环境反馈在循环中使用工具。因此工具集及其文档的清晰设计至关重要。适用于开放性问题——步骤数难以预测、无法硬编码固定路径。但自主性意味着更高成本和复合错误风险，建议在沙盒环境中充分测试并设置适当防护。示例：解决 SWE-bench 任务的编码智能体；Claude "computer use" 参考实现。

## 组合与定制

这些模式并非规定性的，开发者可根据用例塑造和组合。关键在于衡量性能并迭代实现——**仅在复杂度确能改善结果时才添加**。

## 总结

成功不在于构建最复杂的系统，而在于构建适合需求的系统。三条核心原则：保持设计**简洁**；通过明确展示规划步骤确保**透明性**；通过充分的工具**文档和测试**精心打造智能体-计算机接口（ACI）。框架帮助快速起步，但进入生产时不妨减少抽象、用基础组件构建。

**附录 2：工具的提示工程**——工具定义应获得与整体提示同等的提示工程关注。格式选择原则：给模型足够 token "思考"再动笔，避免让模型写进死角；格式贴近模型在互联网文本中自然见过的形式；避免格式开销（如精确计数代码行数、转义 JSON 中的代码）。投入与 HCI 同等精力打造 ACI：站在模型角度审视工具是否显而易见、优化参数命名和描述如同为初级开发者写优质文档、在 workbench 中测试模型使用工具的错误并迭代、防呆（Poka-yoke）设计工具以减少出错可能。SWE-bench 智能体开发中，优化工具的时间实际超过了优化整体提示的时间——例如将相对路径改为绝对路径后模型使用完美无缺。
