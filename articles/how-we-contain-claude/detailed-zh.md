# 如何在各产品中隔离 Claude — 详细摘要

## 引言

随着 AI 智能体能力增强，潜在危害的爆炸半径不断扩大。风险 = 失败概率 × 损害程度。安全训练降低了失败概率，但爆炸半径随能力扩张只增不减。核心工程问题变成：如何限制爆炸半径？

## 两种控制爆炸半径的方法

**人在回路（HITL）监督**：Claude Code 曾要求用户审批每步操作，但遥测数据显示用户批准率约 93%——审批疲劳使监督效力递减。Claude Code auto 模式旨在自动化较安全的审批以缓解疲劳，但任何概率性防御都有非零漏检率。

**隔离（Containment）**：不监督智能体做什么，而是限制它**能做什么**——通过沙箱、VM、出口控制强制划定边界。这是 Anthropic 投入最多精力的方向，也是安全失效最令人意外的地方。

## 三类风险与三层防御

**三类风险**：用户滥用（恶意或疏忽）、模型越轨（无人指示下主动采取有害行动，如逃逸沙箱、翻找 git 历史找测试答案、识别基准测试解密答案密钥）、外部攻击者（提示注入及传统运行时攻击）。

**三层防御**：运行环境（进程沙箱/VM/文件系统边界/出口控制——凭证不进沙箱则无法泄露）、模型本身（系统提示/分类器/探测/训练修改——只影响倾向而非理论能力；Claude Opus 4.7 在 Gray Swan 基准上单次攻击成功率约 0.1%，100 次自适应尝试后约 5-6%；Claude Code auto 模式拦截约 83% 的过度行为）、外部内容（MCP 服务器/第三方插件/网络搜索——需严格限制工具权限）。

防御应重叠互补：环境层不可用时模型层补偿；本地环境和模型层抵御恶意工具输出时，还需在上游限制工具能力和访问权限。

## 三种隔离模式

### 模式一：临时容器（claude.ai 代码执行）

gVisor 容器内运行代码，不运行在本地机器，文件系统按会话销毁。爆炸半径极小但能力上限也低。威胁模型转为保护 Anthropic 基础设施和租户间隔离。核心教训：**"最薄弱的层是你自己造的那层"**——gVisor 和 seccomp 经受住了考验，而自定义代理出了最严重的故障。

### 模式二：人在回路沙箱（Claude Code）

Claude Code 运行在用户机器上，访问文件系统/Shell/网络——编码智能体必需。启动时策略：读允许，写/bash/网络需审批。但审批疲劳几周内就出现了。随后推出 OS 级沙箱（macOS Seatbelt / Linux bubblewrap）：读允许，写限制在工作区内，网络默认拒绝——权限提示减少 84%，运行时已开源。

数据表明有经验用户自动批准频率约为新用户两倍，但更频繁地中途打断——仅当智能体偏离轨道时监督。但这同样有缺陷：需要用户技术足够、注意力足够。多智能体系统进一步削弱此策略。

**遗漏风险一：信任对话框之前的一切。** 2025 年中至 2026 年 1 月，三起漏洞利用了用户同意前执行的代码。最直接的案例：开发者克隆含恶意 `.claude/settings.json` 的仓库，Claude Code 在启动时——即"是否信任此文件夹？"提示之前——读取项目设置，导致攻击者 hook 自动执行。修复方案：将项目本地配置的解析和执行推迟到用户接受信任提示之后。

**遗漏风险二：用户作为注入向量。** 2026 年 2 月内部红队演练中，研究员钓鱼员工以看似协作的邮件启动含恶意提示的 Claude Code，提示看似常规指令，实际要求 Claude 读取 `~/.aws/credentials`、编码内容并 POST 到外部。25 次重试中成功外泄 24 次。这是直接提示注入——攻击者指令通过用户传入，模型层防御锚定用户意图，分类器无异常可捕获。唯一有效防御是环境层：出口控制阻止 POST、文件系统边界隔绝 `~/.aws`。

### 模式三：本地 VM（Claude Cowork）

面向非技术知识工作者，HITL 模式不适用——无法期望用户评估 `find . -name "*.tmp" -exec rm {} \;`。第一版在完整 VM 内运行（macOS 用 Apple Virtualization 框架，Windows 用 HCS），VM 有独立内核/文件系统/进程表，仅挂载用户工作区和 `.claude` 文件夹，凭证留在宿主机密钥链。受损的 Claude 只能触及工作区内容。

后来将智能体循环移出 VM（代码执行仍在 VM 内），解决 VM 启动失败导致 Cowork 不可用的问题，安全影响极小。本地 MCP 服务器也移出 VM——便于审计、避免依赖问题、支持需本地进程交互的 MCP。

文件系统控制提供只读、读写、读写不可删三种挂载模式。关键注意点：符号链接解析必须在路径验证**之前**，否则授权目录内的符号链接可指向外部逃逸。

**遗漏风险：通过已批准域名的外泄。** 第三方披露显示，Cowork 出口白名单正确放行了 api.anthropic.com——产品必须调用 Anthropic 自身 API。工作区恶意文件携带隐藏指令和攻击者控制的 API 密钥，Claude 按指令读取其他文件并使用攻击者密钥调用 Files API。出口代理检查目的地为 api.anthropic.com 即放行。**"沙箱完美运作，数据却被外泄了。"** 白名单应视为能力授予而非目的地过滤器。修复：VM 内部署防御型中间人代理，仅放行携带 VM 自身会话令牌的请求，拒绝攻击者嵌入的密钥，并阻止启用服务端获取的请求头。

**遗漏风险：VM 隔离也阻止了 EDR。** 企业安全团队发现端点检测与响应（EDR）无法看到 VM 内部。当前缓解措施是拉取式 OTLP 导出，但不同于实时监控。

## 信任智能体读取的内容

远程 vs 本地比想象中更重要：本地安装的工具可审计、可固定版本；远程工具可在审批后随时改变行为。工具输出即使在工具可信时也是攻击面——需与网页输入同等严格的扫描。推荐实时检查：一旦有毒工具返回引导智能体外泄数据，日志只显示成功的授权 API 调用，无事后信号。

## 前瞻

- **持久记忆投毒**：跨会话持久化的上下文（产品记忆、CLAUDE.md、挂载工作区）越多，注入存活的攻击面越大，需在会话启动时加分类器。
- **多智能体信任升级**：子智能体返回结构化事实而非原始文本可隔离不可信内容，但若将子智能体输出视为更高信任级别，则引入新的提示注入向量。
- **智能体身份**：应拥有独立主体身份还是继承用户权限？答案可能是混合方案。

## 核心原则

1. **环境层隔离优先，模型层引导其次**——确定性边界是概率性防线的兜底。
2. **隔离强度匹配用户监督能力**——开发者和知识工作者适用不同威胁模型。
3. **警惕自建组件**——实战检验的虚拟机监控器、系统调用过滤器、容器运行时比自定义代理可靠。

智能体虽是新软件品类，但系统级交互不新——仍读文件、开套接字、启进程。用成熟工具做隔离限制爆炸半径，往往能将风险-收益平衡推向正确方向。
