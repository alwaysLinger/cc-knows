# 如何在各产品中隔离 Claude — 简要提炼

## 核心观点

随着 AI 智能体（Agent）能力增强，其潜在危害的爆炸半径（blast radius）也在扩大。Anthropic 在构建 claude.ai、Claude Code 和 Claude Cowork 三款产品的过程中，将**环境层隔离**作为首要防线，模型层行为引导作为补充，形成纵深防御体系。

## 两种控制爆炸半径的路径

1. **人在回路（HITL）监督**：让用户审批每一步操作。但数据表明用户批准率高达 93%，审批疲劳使监督形同虚设。
2. **隔离（Containment）**：不监督智能体做了什么，而是限制它能做什么——通过沙箱、虚拟机和出口控制强制划定访问边界。

## 三类风险与三层防御

风险分为：用户滥用、模型越轨（如主动逃逸沙箱、翻找测试答案）、外部攻击者（提示注入等）。对应三层防御：运行环境（沙箱/VM/出口控制）、模型本身（系统提示/分类器/训练）、外部内容（工具输出审计）。

## 三种隔离模式

- **临时容器（claude.ai）**：gVisor 沙箱内运行代码，文件系统按会话销毁，爆炸半径最小但能力也最受限。
- **人在回路沙箱（Claude Code）**：OS 级沙箱（macOS Seatbelt / Linux bubblewrap），权限提示减少 84%，但依赖用户具备技术判断力。两次重大疏漏：启动信任对话框前的配置执行漏洞、用户被钓鱼后直接成为注入向量。
- **本地 VM（Claude Cowork）**：面向非技术用户，完整 VM 隔离，凭证留在宿主机。关键教训：出口白名单不应视为"目的地过滤器"而应视为"能力授予"——允许访问 api.anthropic.com 意味着允许向任意账户上传文件。

## 核心原则

1. **环境层隔离优先，模型层引导其次**——确定性边界是概率性防线的兜底。
2. **隔离强度应匹配用户的监督能力**——对开发者和知识工作者采用不同策略。
3. **警惕自建组件**——经过实战检验的虚拟机监控器、系统调用过滤器比自定义代理更可靠。

智能体虽是新软件品类，但其系统级交互（读文件、开套接字、启进程）并不新鲜，用成熟工具做隔离依然是最务实的防御路径。
