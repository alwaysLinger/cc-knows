# 为智能体编写有效工具——用智能体优化工具

发布于 2025 年 9 月 11 日

智能体的效果取决于我们赋予它的工具。我们分享如何编写高质量的工具和评估，以及如何利用 Claude 优化其自身工具以提升性能。

[模型上下文协议（Model Context Protocol, MCP）](https://modelcontextprotocol.io/docs/getting-started/intro)可以使 LLM 智能体（Agent）获得数以百计的工具来解决真实世界任务。但我们如何让这些工具发挥最大效用？

在这篇文章中，我们描述了在各种智能体 AI 系统中提升性能的最有效技术¹。

我们首先介绍如何：

* 构建和测试工具原型
* 创建并运行针对智能体工具的全面评估
* 与 Claude Code 等智能体协作，自动提升工具性能

最后，我们分享在此过程中总结的编写高质量工具的关键原则：

* 选择要实现的正确工具（以及不该实现的工具）
* 使用命名空间（Namespacing）界定工具功能边界
* 从工具中返回有意义的上下文信息
* 优化工具响应的 token 效率
* 提示工程化（Prompt-engineering）工具描述和规格

## 什么是工具？

在计算领域，确定性系统（Deterministic systems）在输入相同时总是产生相同的输出，而*非确定性*系统（Non-deterministic systems）——如智能体——即使在相同的起始条件下也可能生成不同的响应。

当我们传统地编写软件时，我们在确定性系统之间建立契约。例如，像 `getWeather("NYC")` 这样的函数调用，每次被调用时都将以完全相同的方式获取纽约市的天气。

工具是一种新型软件，它反映了确定性系统与非确定性智能体之间的契约。当用户问"今天要带伞吗？"时，智能体可能会调用天气工具、凭通用知识回答、甚至先追问关于位置的澄清问题。偶尔，智能体可能会产生幻觉（Hallucinate），甚至无法理解如何使用工具。

这意味着我们需要从根本上重新思考为智能体编写软件的方式：我们不能像为其他开发者或系统编写函数和 API 那样来编写工具和 [MCP 服务器](https://modelcontextprotocol.io/)，我们需要为智能体来设计它们。

我们的目标是通过让智能体使用工具追求各种成功策略，来扩大智能体在解决广泛任务时的有效策略空间。幸运的是，根据我们的经验，对智能体最"符合人体工程学"（Ergonomic）的工具，最终对人类来说也出奇地直观。

## 如何编写工具

在本节中，我们描述如何与智能体协作来编写和改进你给它们的工具。首先快速搭建工具原型并在本地测试。接下来，运行全面评估来衡量后续变更的效果。与智能体并肩工作，你可以重复评估和改进工具的过程，直到智能体在真实任务上取得出色表现。

### 搭建原型

如果不亲自动手，很难预判哪些工具对智能体来说是顺手的、哪些不是。先快速搭建工具原型。如果你使用 [Claude Code](https://www.anthropic.com/claude-code) 来编写工具（可能一次完成），最好给 Claude 提供工具所依赖的软件库、API 或 SDK 的文档（包括可能的 [MCP SDK](https://modelcontextprotocol.io/docs/sdk)）。对 LLM 友好的文档通常可以在官方文档站点的扁平 `llms.txt` 文件中找到（这是我们 [API 的](https://docs.anthropic.com/llms.txt)）。

将你的工具封装为[本地 MCP 服务器](https://modelcontextprotocol.io/docs/develop/connect-local-servers)或[桌面扩展](https://www.anthropic.com/engineering/desktop-extensions)（Desktop Extension, DXT），就可以在 Claude Code 或 Claude Desktop 应用中连接和测试你的工具。

要将本地 MCP 服务器连接到 Claude Code，运行 `claude mcp add <name> <command> [args...]`。

要将本地 MCP 服务器或 DXT 连接到 Claude Desktop 应用，分别导航到 `Settings > Developer` 或 `Settings > Extensions`。

工具也可以直接传入 [Anthropic API](https://docs.anthropic.com/en/docs/agents-and-tools/tool-use/overview) 调用进行程序化测试。

亲自测试工具以发现粗糙之处。收集用户反馈，建立对你期望工具支持的使用场景和提示的直觉。

### 运行评估

接下来，你需要通过运行评估来衡量 Claude 使用工具的效果。首先，生成大量基于真实世界使用的评估任务。我们建议与智能体协作分析结果并确定如何改进工具。端到端流程见我们的[工具评估 cookbook](https://platform.claude.com/cookbook/tool-evaluation-tool-evaluation)。

#### 生成评估任务

有了早期原型，Claude Code 可以快速探索你的工具并创建数十个提示-响应对。提示应受真实世界使用启发，并基于真实的数据源和服务（例如，内部知识库和微服务）。我们建议避免过于简单或肤浅的"沙盒"环境，它们无法以足够的复杂度对工具进行压力测试。强评估任务可能需要多次工具调用——可能多达数十次。

以下是强任务的示例：

* 安排下周与 Jane 的会议，讨论我们最新的 Acme Corp 项目。附上上次项目规划会议的笔记，并预留一间会议室。
* 客户 ID 9182 报告说他们在一次购买尝试中被收费三次。查找所有相关的日志条目，并确定是否有其他客户受到同一问题的影响。
* 客户 Sarah Chen 刚提交了取消请求。准备一个挽留方案。确定：(1) 他们离开的原因，(2) 什么样的挽留方案最有吸引力，(3) 在提供方案之前我们应该注意的任何风险因素。

以下是较弱的任务：

* 安排下周与 jane@acme.corp 的会议。
* 搜索支付日志中的 `purchase_complete` 和 `customer_id=9182`。
* 查找客户 ID 45892 的取消请求。

每个评估提示应配对一个可验证的响应或结果。你的验证器可以简单到真实答案与采样响应之间的精确字符串比较，也可以高级到委托 Claude 来评判响应。避免过于严格的验证器，它们会因为格式、标点或有效替代措辞等无关紧要的差异而拒绝正确响应。

对于每个提示-响应对，你还可以选择性地指定你期望智能体在解决任务时调用的工具，以衡量智能体在评估期间是否成功理解了每个工具的用途。然而，由于可能存在多条正确解决任务的路径，尽量避免过度指定或过度拟合到特定策略。

#### 运行评估

我们建议使用直接 LLM API 调用程序化运行评估。使用简单的智能体循环（包裹交替 LLM API 和工具调用的 `while` 循环）：每个评估任务一个循环。每个评估智能体应被赋予一个任务提示和你的工具。

在评估智能体的系统提示中，我们建议指示智能体不仅输出结构化响应块（用于验证），还要输出推理和反馈块。指示智能体在工具调用和响应块*之前*输出这些内容，可以通过触发思维链（Chain-of-Thought, CoT）行为来提升 LLM 的有效智能。

如果你使用 Claude 运行评估，可以开启[交错思考](https://docs.anthropic.com/en/docs/build-with-claude/extended-thinking#interleaved-thinking)（Interleaved Thinking）来获得类似功能的"开箱即用"版本。这将帮助你探究智能体为何调用或不调用某些工具，并突出工具描述和规格中需要改进的具体领域。

除顶层准确率外，我们建议收集其他指标，如单个工具调用和任务的总运行时间、工具调用总数、总 token 消耗以及工具错误。跟踪工具调用可以帮助揭示智能体追求的常见工作流，并提供工具合并的机会。

#### 分析结果

智能体是你发现问题的得力伙伴，能对从矛盾的工具描述到低效的工具实现和令人困惑的工具 schema 等一切问题提供反馈。然而，请记住，智能体在反馈和响应中省略的内容往往比包含的更重要。LLM 并不总是[说出它们的真实想法](https://www.anthropic.com/research/tracing-thoughts-language-model)。

观察智能体在何处受阻或困惑。阅读评估智能体的推理和反馈（或 CoT），识别粗糙之处。审查原始记录（包括工具调用和工具响应），以捕捉智能体 CoT 中未明确描述的任何行为。读懂言外之意；记住，你的评估智能体不一定知道正确答案和策略。

分析你的工具调用指标。大量冗余工具调用可能表明需要对分页或 token 限制参数进行调整；大量因无效参数导致的工具错误可能表明工具需要更清晰的描述或更好的示例。当我们推出 Claude 的[网页搜索工具](https://www.anthropic.com/news/web-search)时，我们发现 Claude 不必要地在工具的 `query` 参数后追加 `2025`，导致搜索结果偏差和性能下降（我们通过改进工具描述引导 Claude 走上了正确的方向）。

### 与智能体协作

你甚至可以让智能体分析你的结果并为你改进工具。只需将评估智能体的记录拼接后粘贴到 Claude Code 中。Claude 是分析记录和同时重构大量工具的专家——例如，确保在进行新更改时工具实现和描述保持自洽。

事实上，本文中的大部分建议都来自我们反复使用 Claude Code 优化内部工具实现的过程。我们的评估基于内部工作空间构建，模拟了我们内部工作流的复杂性，包括真实的项目、文档和消息。

我们依赖留出测试集（Held-out test sets）来确保我们没有过拟合到"训练"评估。这些测试集表明，即使超越了"专家"工具实现所达到的水平，我们仍能提取额外的性能提升——无论那些工具是由我们的研究人员手工编写还是由 Claude 自身生成的。

在下一节中，我们将分享我们从这一过程中学到的一些经验。

## 编写有效工具的原则

在本节中，我们将我们的学习成果提炼为编写有效工具的几条指导原则。

### 为智能体选择正确的工具

更多工具并不总是带来更好的结果。我们观察到的一个常见错误是工具仅仅包装现有软件功能或 API 端点——无论这些工具是否适合智能体。这是因为智能体与传统软件具有不同的"可供性"（Affordances）——即它们感知这些工具潜在行动的方式不同。

LLM 智能体的"上下文"有限（即它们一次能处理的信息量有限），而计算机内存便宜且充裕。考虑在地址簿中搜索联系人的任务。传统软件程序可以高效地存储并逐个处理联系人列表，逐个检查后再移到下一个。

然而，如果 LLM 智能体使用的工具返回所有联系人，然后必须逐个 token 地阅读每个联系人，它就在浪费有限的上下文空间来处理无关信息（想象一下从头到尾逐页阅读地址簿来搜索联系人——即暴力搜索）。更好、更自然的方法（对智能体和人类都一样）是先跳到相关页面（可能是按字母顺序查找）。

我们建议构建少量精心设计的、面向特定高影响力工作流的工具，与你的评估任务匹配，然后逐步扩展。在地址簿的例子中，你可能会选择实现 `search_contacts` 或 `message_contact` 工具，而不是 `list_contacts` 工具。

工具可以合并功能，在底层处理可能的*多个*离散操作（或 API 调用）。例如，工具可以在工具响应中丰富相关元数据，或在单次工具调用中处理频繁链式的多步任务。

以下是一些示例：

* 与其实现 `list_users`、`list_events` 和 `create_event` 工具，不如考虑实现一个 `schedule_event` 工具，它查找空闲时间并安排事件。
* 与其实现 `read_logs` 工具，不如考虑实现 `search_logs` 工具，它只返回相关的日志行和一些周围上下文。
* 与其实现 `get_customer_by_id`、`list_transactions` 和 `list_notes` 工具，不如实现 `get_customer_context` 工具，它一次性编译客户的所有近期和相关信息。

确保你构建的每个工具都有明确、独立的目的。工具应使智能体能够像人类一样——在拥有相同底层资源的情况下——分解和解决任务，同时减少原本会被中间输出消耗的上下文。

过多或功能重叠的工具也会分散智能体追求高效策略的注意力。精心、有选择地规划你要构建（或不构建）的工具，确实可以带来丰厚回报。

### 工具命名空间

你的 AI 智能体可能会获得数十个 MCP 服务器和数百种不同工具的访问权限——包括其他开发者提供的工具。当工具在功能上重叠或用途模糊时，智能体可能会混淆应该使用哪些工具。

命名空间（Namespacing，在公共前缀下分组相关工具）可以帮助界定大量工具之间的边界；MCP 客户端有时默认这样做。例如，按服务（如 `asana_search`、`jira_search`）和按资源（如 `asana_projects_search`、`asana_users_search`）对工具进行命名空间划分，可以帮助智能体在正确的时机选择正确的工具。

我们发现，在基于前缀和基于后缀的命名空间之间做选择，对我们的工具使用评估有非平凡的影响。效果因 LLM 而异，我们鼓励你根据自身的评估来选择命名方案。

智能体可能调用错误的工具、用错误的参数调用正确的工具、调用过少的工具，或不正确地处理工具响应。通过有选择地实现名称反映任务自然细分的工具，你同时减少了载入智能体上下文的工具和工具描述数量，并将智能体计算从其上下文卸载回工具调用本身。这降低了智能体犯错的总体风险。

### 从工具返回有意义的上下文

同理，工具实现在将信息返回给智能体时应注意只返回高信号信息。它们应优先考虑上下文相关性而非灵活性，摒弃低级技术标识符（例如：`uuid`、`256px_image_url`、`mime_type`）。像 `name`、`image_url` 和 `file_type` 这样的字段更有可能直接指导智能体的下游行动和响应。

智能体也倾向于更成功地处理自然语言名称、术语或标识符，而非晦涩的标识符。我们发现，仅将任意字母数字 UUID 解析为更具语义和可解释性的语言（甚至零索引 ID 方案），就能显著提升 Claude 在检索任务中的精度，减少幻觉。

在某些情况下，智能体可能需要灵活地与自然语言和技术标识符输出进行交互，哪怕只是为了触发下游工具调用（例如，`search_user(name='jane')` → `send_message(id=12345)`）。你可以通过在工具中暴露一个简单的 `response_format` 枚举参数来同时启用两者，允许智能体控制工具返回 `"concise"` 还是 `"detailed"` 响应（下图所示）。

你可以添加更多格式以获得更大的灵活性，类似于 GraphQL 中你可以精确选择要接收的信息片段。以下是一个控制工具响应详细程度的 ResponseFormat 枚举示例：

```python
enum ResponseFormat {
   DETAILED = "detailed",
   CONCISE = "concise"
}
```

### 优化工具响应的 token 效率

优化上下文质量很重要。但优化工具响应中返回给智能体的上下文*数量*同样重要。

我们建议对可能消耗大量上下文的任何工具响应，实现分页（Pagination）、范围选择、过滤和/或截断（Truncation）的某种组合，并设置合理的默认参数值。对于 Claude Code，我们默认将工具响应限制为 25,000 个 token。我们预计智能体的有效上下文长度会随时间增长，但对上下文高效工具的需求将持续存在。

如果你选择截断响应，请务必用有用的指引来引导智能体。你可以直接鼓励智能体追求更节省 token 的策略，比如对知识检索任务进行多次小型、有针对性的搜索，而不是一次大型搜索。同样，如果工具调用引发错误（例如，在输入验证期间），你可以对错误响应进行提示工程，清晰传达具体且可操作的改进建议，而非晦涩的错误码或堆栈跟踪。

### 提示工程化你的工具描述

现在我们来到改进工具最有效的方法之一：提示工程化你的工具描述和规格。因为这些内容被载入智能体的上下文，它们可以集体引导智能体走向有效的工具调用行为。

在编写工具描述和规格时，想象你如何向团队新成员描述你的工具。考虑你可能隐含带入的上下文——专用查询格式、小众术语的定义、底层资源之间的关系——并将其显式化。通过清晰描述（并用严格数据模型强制执行）预期输入和输出来消除歧义。特别是，输入参数应明确命名：与其用名为 `user` 的参数，不如用名为 `user_id` 的参数。

通过评估，你可以更有信心地衡量提示工程的影响。即使对工具描述的微小改进也能带来显著提升。Claude Sonnet 3.5 在 [SWE-bench Verified](https://www.anthropic.com/engineering/swe-bench-sonnet) 评估上达到了最先进性能，正是我们对工具描述做出精确改进之后——显著降低了错误率并提升了任务完成率。

你可以在我们的[开发者指南](https://docs.anthropic.com/en/docs/agents-and-tools/tool-use/implement-tool-use#best-practices-for-tool-definitions)中找到工具定义的其他最佳实践。如果你在为 Claude 构建工具，我们还建议阅读工具如何动态载入 Claude 的[系统提示](https://docs.anthropic.com/en/docs/agents-and-tools/tool-use/implement-tool-use#tool-use-system-prompt)。最后，如果你在为 MCP 服务器编写工具，[工具注解](https://modelcontextprotocol.io/specification/2025-06-18/server/tools)有助于披露哪些工具需要开放世界访问或执行破坏性变更。

## 展望

要为智能体构建有效工具，我们需要将软件开发实践从可预测的、确定性的模式重新调整为非确定性的模式。

通过我们在本文中描述的迭代、评估驱动的流程，我们发现了使工具成功的持续模式：有效的工具有意且清晰地定义、审慎使用智能体上下文、可以在多样工作流中组合，并使智能体能够直觉式地解决真实世界任务。

未来，我们预计智能体与世界交互的具体机制将不断演变——从 MCP 协议的更新到底层 LLM 本身的升级。通过系统化、评估驱动的方法来改进智能体工具，我们可以确保随着智能体变得更加有能力，它们使用的工具也将与之同步进化。
