# 为智能体编写有效工具 — 详细摘要

## 什么是工具？

传统软件中，确定性系统之间建立契约——如 `getWeather("NYC")` 每次调用行为完全一致。而工具（Tool）是确定性系统与非确定性智能体（Agent）之间的新型契约。当用户问"今天要带伞吗？"，智能体可能调用天气工具、凭知识回答、或先追问位置——行为不可预测。因此，不能沿用面向开发者的 API 设计方式来为智能体编写工具和 MCP 服务器，必须重新设计。

## 如何编写工具

### 搭建原型

不要纸上谈兵，应快速搭建原型并亲测。用 Claude Code 编写工具时，提供相关 SDK 文档（包括 `llms.txt` 文件），将工具封装为本地 MCP 服务器或桌面扩展（DXT），即可在 Claude Code 或 Claude Desktop 中连接测试。也可通过 Anthropic API 直接传入工具进行程序化测试。收集用户反馈，建立对使用场景的直觉。

### 运行评估

评估是系统性衡量工具性能的关键。

**生成评估任务**：与智能体协作，创建大量提示-响应对。任务应源于真实场景，避免过于简单的沙盒环境。优质任务需要多次工具调用，例如"安排与 Jane 下周的会议，附上上次项目规划会议的笔记，并预留会议室"；弱任务则过于直接，如"搜索客户 ID 45892 的取消请求"。每个提示应配对可验证的响应——可以是精确字符串比对，也可让 Claude 判定。验证器不宜过严，避免因格式或措辞差异拒绝正确答案。

**运行评估**：推荐用直接 LLM API 调用程序化运行。每个评估任务使用一个简单智能体循环（`while` 循环包裹交替的 LLM API 和工具调用）。在系统提示中要求智能体输出推理和反馈块（在工具调用之前），以触发思维链（CoT）行为。使用 Claude 时可开启交错思考（Interleaved Thinking）。除顶层准确率外，还应收集工具调用总时长、调用次数、token 消耗和工具错误等指标。

**分析结果**：智能体擅长发现矛盾描述、低效实现和混乱 schema，但它省略的信息往往更重要——LLM 并非总表达真实想法。观察智能体在何处困惑，阅读推理链和原始记录，分析指标（冗余调用暗示需调整分页参数；无效参数错误暗示需改进描述）。例如，Claude 网页搜索工具上线时，发现 Claude 不必要地在查询参数后追加 `2025`，通过改进描述解决了此问题。

### 与智能体协作优化

将评估记录拼接后交给 Claude Code 分析，它能批量重构工具以确保一致性。作者内部评估基于真实工作空间，使用留出测试集（held-out test set）防止过拟合。测试表明，Claude 优化的工具甚至超越了人类专家手工编写的版本。

## 编写有效工具的原则

### 1. 选择正确的工具

工具多不等于效果好。常见错误是简单包装 API 端点。智能体的"可供性"（Affordance）与传统软件不同——LLM 上下文有限，而计算机内存廉价。以地址簿搜索为例：传统程序可逐条遍历，但若工具返回所有联系人让智能体逐 token 阅读，则浪费上下文。正确做法是实现 `search_contacts` 而非 `list_contacts`。

工具可合并功能——在单次调用中处理多步操作。例如：用 `schedule_event` 替代 `list_users` + `list_events` + `create_event`；用 `search_logs` 替代 `read_logs`；用 `get_customer_context` 替代 `get_customer_by_id` + `list_transactions` + `list_notes`。每个工具应有明确独立的目的，使智能体能像人类一样分解任务，同时减少中间输出消耗的上下文。工具过多或重叠会分散智能体注意力。

### 2. 命名空间

智能体可能访问数十个 MCP 服务器和数百个工具。功能重叠或用途模糊时，智能体会困惑。命名空间（在公共前缀下分组相关工具）可界定边界。例如按服务（`asana_search`、`jira_search`）和资源（`asana_projects_search`、`asana_users_search`）命名。前缀式与后缀式命名对评估结果有非平凡影响，具体效果因 LLM 而异，应根据自身评估选择。

### 3. 返回有意义的上下文

工具应只返回高信号信息，优先上下文相关性而非灵活性，摒弃低级技术标识符（如 `uuid`、`256px_image_url`、`mime_type`），而返回 `name`、`image_url`、`file_type` 等直接指导下游行动的字段。将任意字母数字 UUID 解析为语义化语言（甚至零索引 ID）即可显著提升 Claude 在检索任务中的精度、降低幻觉。

若智能体需同时交互自然语言和技术标识符（如 `search_user(name='jane')` → `send_message(id=12345)`），可暴露 `response_format` 枚举参数，允许智能体控制返回"简洁"或"详细"响应。

### 4. Token 效率优化

优化上下文质量重要，优化返回数量同样重要。对可能消耗大量上下文的工具响应，实现分页、范围选择、过滤和/或截断，并设置合理默认值。Claude Code 默认限制工具响应 25,000 token。若截断响应，应提供有用指引——鼓励智能体采取高效策略（多次小范围搜索而非一次大范围搜索）；错误响应应明确具体可操作的改进建议，而非晦涩的错误码。

### 5. 提示工程化工具描述

这是最有效的方法之一。工具描述和规格被载入智能体上下文，可集体引导其工具调用行为。编写时想象给新员工描述——将隐含上下文（专用查询格式、术语定义、资源关系）显式化。消除歧义，用严格数据模型约束输入输出。参数命名须无歧义：用 `user_id` 而非 `user`。通过评估可精确衡量提示工程的影响。Claude Sonnet 3.5 在 SWE-bench Verified 上取得突破，正是源于对工具描述的精细改进——显著降低错误率、提升任务完成率。

## 展望

为智能体构建有效工具，需从确定性模式转向非确定性模式。通过迭代、评估驱动的流程，有效工具的共性已清晰：定义意图明确、审慎使用上下文、可组合于多样工作流、使智能体能直觉式解决真实任务。随着 MCP 协议更新和底层 LLM 升级，具体机制将演变，但系统化的评估驱动方法可确保工具与智能体能力同步进化。
