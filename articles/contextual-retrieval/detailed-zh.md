# Contextual Retrieval 详细摘要

## 引言

AI 模型在特定场景中需要背景知识才能发挥作用，如客服聊天机器人需要业务知识、法律分析机器人需要案例库。开发者通常使用 RAG（Retrieval-Augmented Generation，检索增强生成）来增强模型知识，但传统 RAG 在编码信息时会丢失上下文，导致检索失败。Anthropic 提出 **Contextual Retrieval**（上下文检索），包含两个子技术：Contextual Embeddings（上下文嵌入）和 Contextual BM25（上下文 BM25），可将检索失败率降低 49%，叠加重排序后降低 67%。

## 关于直接使用更长提示的说明

若知识库小于 200,000 token（约 500 页材料），可直接将全部知识库放入提示中，无需 RAG。Anthropic 的 Prompt Caching（提示缓存）使此方案更快更经济——延迟降低 2 倍以上，成本降低最多 90%。但知识库增长后，需要更可扩展的方案，即 Contextual Retrieval。

## RAG 基础：扩展到更大知识库

传统 RAG 的预处理流程：1）将语料库切分为几百 token 的文本块；2）用嵌入模型将文本块转为向量嵌入；3）存入向量数据库，按语义相似度检索。运行时，根据用户查询找到最相关的文本块，加入提示传给生成模型。

嵌入模型擅长语义匹配，但可能遗漏精确匹配。BM25（Best Matching 25）是基于词法匹配的排序函数，对包含唯一标识符或技术术语的查询特别有效。例如查询"Error code TS-999"，嵌入模型可能找到通用错误码内容，而 BM25 能精确匹配该字符串。

结合嵌入和 BM25 的 RAG 流程：1）切分文档；2）创建 TF-IDF 编码和语义嵌入；3）BM25 检索精确匹配的 Top 块；4）嵌入检索语义相似的 Top 块；5）用 rank fusion 合并去重；6）将 Top-K 块加入提示生成回答。

### 传统 RAG 的上下文困境

文档切分后，单个文本块可能缺乏足够上下文。例如 SEC 财报中，"公司收入较上季度增长3%"这个文本块，脱离原文后无法判断公司和时间，导致检索困难。

## Contextual Retrieval 详解

Contextual Retrieval 在嵌入和构建 BM25 索引前，为每个文本块 prepend 一段简短的上下文说明。以上述 SEC 文件为例，原始文本块"公司收入较上季度增长3%"被转换为"此文本块来自 ACME 公司 2023 年 Q2 的 SEC 文件；上季度收入为 3.14 亿美元。公司收入较上季度增长3%。"

其他方案（如添加通用文档摘要、假设性文档嵌入、基于摘要的索引）效果有限，与本文方案不同。

### 实现方式

使用 Claude 3 Haiku 为每个文本块生成上下文。提示模板要求模型根据整个文档，为指定文本块提供简短上下文，以改善检索效果。生成的上下文通常 50-100 token，prepend 到文本块后再进行嵌入和 BM25 索引。

### 利用 Prompt Caching 降低成本

借助 Claude 的 Prompt Caching，文档只需加载到缓存一次，无需为每个文本块重复传入。假设 800 token 的文本块、8k token 的文档、50 token 的上下文指令、100 token 的上下文输出，一次性成本为每百万文档 token $1.02。

### 性能提升

- Contextual Embeddings：检索失败率降低 **35%**（5.7% → 3.7%）
- Contextual Embeddings + Contextual BM25：检索失败率降低 **49%**（5.7% → 2.9%）

实验覆盖多种知识领域（代码库、小说、ArXiv 论文、科学论文），所有嵌入源组合中上下文化均提升性能。

### 实现注意事项

1. **文本块边界**：切分方式（大小、边界、重叠）影响检索性能
2. **嵌入模型**：Gemini 和 Voyage 嵌入效果最佳
3. **自定义上下文提示**：针对特定领域定制提示可能效果更好（如加入术语表）
4. **文本块数量**：传递 20 个文本块优于 10 个或 5 个，但信息过多可能干扰模型

## 叠加 Reranking 进一步提升

Reranking（重排序）是常用过滤技术，确保只有最相关的文本块传给模型。流程：1）初始检索获取 Top-150 候选块；2）将候选块和用户查询传入重排序模型；3）按相关性评分选取 Top-20；4）传入模型生成结果。

使用 Cohere Reranker 测试，Reranked Contextual Embedding + Contextual BM25 将检索失败率降低 **67%**（5.7% → 1.9%）。重排序会增加少量延迟和成本，需在性能与延迟/成本间权衡。

## 结论

大量实验的核心发现：1）嵌入+BM25 优于纯嵌入；2）Voyage 和 Gemini 嵌入最佳；3）Top-20 块优于 Top-10 或 Top-5；4）添加上下文大幅提升检索精度；5）重排序优于无重排序；6）**所有收益可叠加**：最佳组合为上下文嵌入 + 上下文 BM25 + 重排序 + Top-20 文本块。
