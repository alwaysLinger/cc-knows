# 通过 MCP 执行代码：构建更高效的智能体 — 详细摘要

## 背景

MCP（Model Context Protocol，模型上下文协议）是连接 AI 智能体与外部系统的开放标准，解决了传统"每对集成单独开发"的碎片化问题——开发者只需实现一次 MCP，即可接入整个集成生态。自2024年11月发布以来，社区已构建数千个 MCP 服务器，各大编程语言的 SDK 齐备，MCP 已成为连接智能体与工具/数据的事实标准。

然而，当智能体连接数十个 MCP 服务器、访问数百乃至数千工具时，传统方式暴露出严重的效率问题。

## 问题：过多的工具消耗导致智能体效率下降

### 1. 工具定义过载上下文窗口

传统 MCP 客户端将所有工具定义直接加载到上下文中。每个工具定义包含名称、描述、参数和返回值说明。当连接数千工具时，模型在读取用户请求之前就需处理数十万 token 的定义信息，增加了响应时间和成本。

### 2. 中间工具结果消耗额外 token

直接工具调用模式下，每个中间结果都必须流经模型。以"从 Google Drive 下载会议记录并附加到 Salesforce 线索"为例：模型先调用 `gdrive.getDocument` 获取完整文本（进入上下文），再调用 `salesforce.updateRecord` 将同一文本再次写入上下文。2小时销售会议的记录约5万 token，被处理两次。更大的文档甚至可能超出上下文窗口限制，导致工作流中断。此外，模型在工具调用间复制数据时也更容易出错。

## 解决方案：代码执行提升上下文效率

核心思路：将 MCP 服务器呈现为代码 API 而非直接工具调用，让智能体编写代码与之交互。具体实现方式是从已连接的 MCP 服务器生成一个文件树：

```
servers/
├── google-drive/
│   ├── getDocument.ts
│   └── index.ts
├── salesforce/
│   ├── updateRecord.ts
│   └── index.ts
```

每个工具对应一个 TypeScript 文件，包含类型定义和对 `callMCPTool` 的封装调用。智能体通过浏览文件系统发现工具，按需读取所需的定义文件。

前述示例变为简洁的代码：读取文档、将内容直接传给 Salesforce 更新——无需中间结果流经模型。Token 用量从15万降至2千，节省 **98.7%**。Cloudflare 发表的"Code Mode"研究得出了相同结论。

## 代码执行的优势

### 渐进式披露（Progressive Disclosure）

模型擅长导航文件系统。将工具呈现为文件系统上的代码，模型可按需读取定义，而非一次性全部加载。还可添加 `search_tools` 工具，支持按关键词搜索和细粒度控制（仅名称、名称+描述、完整定义），进一步节省上下文。

### 上下文高效的工具结果

面对大数据集，智能体可在代码中先过滤和转换再输出。例如获取1万行电子表格时，可在执行环境中过滤出待处理订单，仅将5条示例行输出到上下文——而非让1万行全部流经模型。类似模式也适用于聚合、多数据源连接、字段提取等操作。

### 更强大且上下文高效的控制流

循环、条件判断和错误处理可用熟悉的代码模式实现，无需在智能体循环中逐个链接工具调用。例如轮询 Slack 消息等待部署通知，用 while 循环比交替发送工具调用和休眠命令高效得多。条件树的直接执行还节省了"首 token 时间"延迟——无需等待模型评估 if 语句。

### 隐私保护操作

代码执行时，中间结果默认留在执行环境中，仅显式 log 或 return 的内容进入模型上下文。对于更敏感的工作负载，MCP 客户端可自动对 PII（个人身份信息）进行令牌化处理：智能体看到的是 `[EMAIL_1]`、`[PHONE_1]` 等占位符，真实数据在 MCP 工具调用时自动还原——数据从 Google Sheets 流向 Salesforce，但从不经过模型。这还可用于定义确定性的安全规则，控制数据的流向。

### 状态持久化与技能（Skills）

代码执行配合文件系统访问，智能体可将中间结果写入文件以跨操作维持状态、恢复工作进度。更重要的是，智能体可将有效的代码实现保存为可复用函数，配合 SKILL.md 文件形成结构化的"技能"。这与 Claude 的 Skills 概念紧密相关——模型逐步构建高阶能力工具箱，不断进化其高效工作所需的脚手架。

### 权衡

代码执行也引入了自身的复杂性：运行智能体生成的代码需要安全的执行环境，包括适当的沙箱、资源限制和监控。这些基础设施要求增加了运维开销和安全考量，直接工具调用则无需这些。应根据实际场景权衡收益与实现成本。

## 总结

MCP 为智能体连接众多工具和系统提供了基础协议，但连接过多时工具定义和结果会消耗过多 token。代码执行将软件工程中成熟的模式——按需加载、数据过滤、状态管理——应用于智能体，使其用熟悉的编程结构与 MCP 更高效地交互。作者鼓励实践者与 MCP 社区分享发现。
