# 长时间运行 Agent 的有效 Harness

## 简要提炼

Anthropic 工程团队针对长时间运行 AI Agent 的核心难题——跨多次上下文窗口保持一致进展——提出了双 Agent 架构方案。

**核心问题**：每次新会话像没有交接班的换班工程师，对之前工作毫无记忆。即使使用上下文压缩（context compaction），Agent 仍会陷入两种失败模式：一是"一次性实现"——试图一次构建整个应用，中途耗尽上下文，留给下个会话一堆半成品；二是"过早完成"——后续会话看到部分进展就错误地宣称项目已完工。

**解决方案**：引入两类 Agent。**初始化 Agent**（initializer agent）在首次运行时搭建环境，生成详尽的 JSON 特性列表，将用户需求拆解为数百条可验证的功能项，每条附带测试步骤，初始标记为未通过。**编码 Agent**（coding agent）则逐条推进功能实现，每次只专注于一个特性，完成后提交代码并更新进度日志。

**关键实践**：一是增量推进，强制 Agent 每次只做一个特性，杜绝过度承诺；二是严格测试，要求 Agent 使用 Puppeteer 等浏览器自动化工具做端到端验证并截图，而非自行宣称完成；三是规范交接，每轮会话开始时读取 git 日志和 `claude-progress.txt` 恢复上下文，结束时提交代码和更新进度文件，确保下次会话能快速上手。

**启示**：长时 Agent 的可靠性不在于更强的模型，而在于结构化的工作流——用外部持久化状态替代上下文记忆，用可验证的测试替代主观判断，用增量推进替代大跃进。这套方案目前聚焦 Web 应用开发，向科研、金融建模等领域的泛化仍是开放问题。
