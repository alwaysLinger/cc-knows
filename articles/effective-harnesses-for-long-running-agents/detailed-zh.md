# 长时间运行 Agent 的有效 Harness

## 详细摘要

### 问题背景

随着 AI Agent 能力增强，开发者越来越倾向于让其承担跨越数小时甚至数天的复杂任务。然而，跨多个上下文窗口保持一致进展仍是开放问题。核心困境在于：每次新会话都从零开始，对之前的工作毫无记忆，如同没有交接记录的换班工程师。

Claude Agent SDK 虽有上下文压缩（context compaction）机制防止窗口耗尽，但仅靠压缩远远不够。即便是 Opus 4.5，也无法仅凭高层级提示词在多个会话中完整构建一个生产级的 claude.ai 复制品。

### 两种关键失败模式

1. **一次性实现（One-shot implementation）**：Agent 试图一次构建整个应用，中途上下文耗尽，留下半成品且无文档，后续会话只能被迫调试这些无法理解的代码。
2. **过早完成（Premature completion）**：后续会话看到部分进展后，错误地判定项目已经完成。

### 双 Agent 架构方案

团队为 Claude Agent SDK 设计了双 Agent 方案：初始化 Agent（initializer agent）和编码 Agent（coding agent）。两者仅在初始提示词上不同，系统提示、工具和 harness 完全一致。

### 环境管理

#### 特性列表（Feature List）

为解决一次性实现和过早完成问题，初始化 Agent 会将用户的提示词展开为一份详尽的 JSON 特性列表。以 claude.ai 复制品为例，列表包含超过 200 条功能描述，例如："用户可以打开新聊天、输入查询、按回车键、看到 AI 回复"。所有特性初始标记为未通过（`passes: false`），每条都附有结构化的测试步骤。编码 Agent 只被允许更新 `passes` 字段，严禁修改测试定义本身。

#### 增量推进（Incremental Progress）

强制 Agent 每次只处理一个特性，有效消除了过度承诺的倾向。同时要求 Agent：（1）提交附带描述性信息的 git commit；（2）更新 `claude-progress.txt` 进度日志。这使得 Agent 可以回滚错误变更，也让后续会话能快速了解先前工作，无需猜测。

#### 测试（Testing）

不加约束时，Claude 经常在未经端到端验证的情况下就标记功能完成。解决方案是要求 Agent 使用 Puppeteer MCP 等浏览器自动化工具，像人类用户一样测试功能，包括截图验证。不过该方法仍有局限，例如 Puppeteer 无法检测原生浏览器警告弹窗。

### 会话启动流程

每个编码 Agent 会话都以标准化入职步骤开始：

1. 运行 `pwd` 确认工作目录
2. 读取 git 日志和 `claude-progress.txt` 了解近期工作
3. 读取特性列表，选择优先级最高的未完成特性

典型流程为：确认环境 → 回顾进度 → 启动开发服务器 → 验证基础功能 → 选取下一个任务开始工作。

### 失败模式与解决方案对照

| 问题 | 初始化 Agent 的作用 | 编码 Agent 的作用 |
|---|---|---|
| 过早宣称项目完成 | 创建结构化 JSON 特性列表 | 会话开始时读取列表，选取一个未完成特性 |
| 环境留有 bug 或无文档 | 创建初始 git 仓库和进度日志 | 读取进度/git 日志、运行基线测试、结束时提交并更新进度 |
| 未经测试就标记完成 | 设置特性列表 | 仅在端到端测试通过后才标记 |
| Agent 难以启动应用 | 编写 `init.sh` 脚本 | 会话开始时引用该脚本 |

### 未来方向

该方案验证了增量式跨会话编码工作流的可行性，但仍有两个开放问题：（1）专业化多 Agent 架构是否优于单一通用编码 Agent；（2）如何将这一面向 Web 应用的流程泛化到科学研究、金融建模等其他领域。
