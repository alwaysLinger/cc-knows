# Agent 稳定性的关键不在循环本身，而在循环外的工程系统

- 链接：https://mp.weixin.qq.com/s/cIQYl9Wr1Eov4ma-_bYh-w

## 核心论点

文章的核心判断是：Agent Loop 本身并不复杂，真正决定系统能否长期稳定运行的是 Harness、上下文工程、工具设计、记忆整合、评测、追踪和安全边界。更强模型能提高上限，但在代码、运维、工程执行这类可验证任务里，自动化验证和清晰约束往往比单纯换模型更关键。

作者反复强调分工：模型负责推理，外部系统负责状态、边界、反馈和回退。新增能力应通过工具、提示结构和状态外化接入，而不是把核心 loop 改成庞大的状态机。

## 技术/架构拆解

文章从最小 Agent Loop 出发：模型接收 messages 和 tools，若返回 tool use 就执行工具并把结果写回上下文，否则输出文本。这个循环对应感知、决策、行动、反馈四阶段。Workflow 与 Agent 的区别在控制权：前者由代码预定义路径，后者由 LLM 动态决定下一步。

系统工程部分分成多条主线。Harness 包含验收基线、执行边界、反馈信号和回退手段，目标是把任务推向“目标明确且可自动验证”的区域。上下文工程通过常驻层、按需加载、运行时注入、记忆层和系统层分离，避免 Context Rot。工具设计按 ACI 原则面向 Agent 目标，而不是照搬 API endpoint，并用结构化错误、参数示例和动态工具发现减少误调用。

记忆系统被拆成工作记忆、程序性记忆、情景记忆和语义记忆；长任务通过 `feature-list.json`、进度文件和 git commit 外化状态；多 Agent 通过任务图、JSONL inbox、worktree 和协议隔离协作；评测则区分 Pass@k 与 Pass^k、transcript 与 outcome、agent harness 与 evaluation harness。OpenClaw 的五层架构把这些原则落到 Gateway、Channel、Agent Loop、工具集、上下文与记忆层中。

## 作者论证路径

作者先用不到 20 行的代码展示 Agent Loop 的朴素本质，降低神秘感；然后比较 Workflow 和 Agent，说明完整自主并不总是必要。接着把重点转向 loop 外围：Harness 决定任务是否能收敛，上下文决定模型看见什么，工具决定模型能做什么，记忆决定跨会话是否一致。

中段进一步讨论放权后的问题：跨 session 续跑、任务状态外化、后台 I/O、多 Agent 协议、评测和 Trace。最后用 OpenClaw 串联这些原则，展示一个可运行 Agent 系统如何把消息总线、分层提示、heartbeat、任务恢复、安全边界和 provider fallback 组合起来。

## 对开发者认知的改变

文章会改变几个常见直觉。第一，Agent 能力提升不主要来自更复杂的 loop，而来自更好的上下文、工具和反馈系统。第二，工具调用失败时不应先怀疑模型，而应先检查工具描述、粒度、返回值和错误格式。第三，多 Agent 的价值不是“多开几个模型”，而是用隔离工件和结构化协议减少人的持续参与。

更重要的是，评测系统本身也会出错。分数下降可能来自容器资源不足、评分器 bug 或测试脱离生产，而不是 Agent 退化。因此工程团队需要先验证评测信号，再调整模型、prompt 或工具。

## 潜在局限或适用边界

这些方法最适合目标清晰、结果可验证、工具边界可定义的任务，比如 Coding Agent、运维排障、数据处理和工程自动化。开放式研究、产品讨论、谈判协商等弱验证任务仍然依赖模型本身的推理上限和人的判断。

文章给出的 OpenClaw 式架构需要较强基础设施：持久化会话、事件流、Trace、评测套件、权限和审计。如果团队规模小或任务低频，完整搭建可能成本过高，应优先从单渠道、单任务、可执行验收和安全边界开始。

## 实践中的指导意义

构建 Agent 时应按顺序推进：先让一条最小链路跑通，再补安全边界、工作空间隔离和审计；随后把项目知识拆成常驻索引、Skills 和文档；工具按任务目标重做描述和错误返回；从第一个真实失败案例开始建评测；最后再考虑多 Agent、动态工具发现和更复杂的调度。

调试时也要改变顺序：工具选错先查工具定义，上下文漂移先查分层和压缩策略，长任务失败先查状态是否外化，分数下降先查评测基础设施。这样才能把 Agent 从“会演示的 Demo”推进到“可长期运行的工程系统”。
