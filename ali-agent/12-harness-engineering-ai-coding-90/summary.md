# 把 AI Coding 变成可控交付系统

## 文章信息

- 原文标题：Harness Engineering：耗时一周，我是如何将应用的AI Coding率提升至90%的
- 链接：https://mp.weixin.qq.com/s/rlIyIIZOXFObNIXbPI7gDg
- 公众号：阿里云开发者
- 发布时间：2026年5月7日 08:32

## 核心论点

文章的核心不是证明 AI 会写代码，而是证明“让 AI 写出的代码可信”需要一套外部工程系统。作者把 AI Coding 的演进划成 Prompt Engineering、Context Engineering、Harness Engineering 三阶段：Prompt 解决单次表达，Context 解决给 Agent 看什么，Harness 则解决跨会话、跨角色、跨阶段的约束、反馈和流程控制。

作者认为，大模型原始能力已经足够强，真正阻碍企业级落地的是可靠性鸿沟：Agent 能生成语法正确、风格统一的代码，却容易在业务语义、隐式约束、验证闭环上出错。因此 Harness 的目标不是让模型“更聪明”，而是让错误可发现、可拦截、可修复，并把每次失败沉淀成下一次不会再犯的机制。

## 技术/架构拆解

作者把 Harness 落地为四类资产：Rules、Skills、Wiki、Changes。Rules 固化稳定约束，例如工程结构、编码规范、流程规范；Skills 把需求分析、编码、评审、单测、CI、部署等工作做成可复用 SOP；Wiki 承载业务链路、数据模型和领域知识；Changes 记录每个需求从分析到上线的审计轨迹。

关键编排中心是 Application Owner Agent。它不是堆满信息的百科全书，而是一个 Index & Map：在 400 行左右的定义文件中描述角色、项目背景、配置索引、七项职责、十阶段调度和硬性约束。Agent 在不同阶段按需加载对应 Skill 和知识，避免上下文窗口被无关信息污染。

执行层采用 10 阶段流水线：需求分析、需求评审、编码实现、编码评审、单元测试编写、单元测试评审、代码推送、CI 验证、部署验证、用户确认。每个阶段都有入口条件、产物路径、质量门禁和失败回退路径。比如 CI 失败时按失败类型回退到测试或编码阶段，而不是从头重来。

质量控制上，文章强调机械化门禁：不能只写“检查 CI 是否通过”，而要明确 `status == SUCCESS && total_tests > 0 && passed == total`。不能只要求“生成评审报告”，还要验证文件路径存在且包含必填章节。凡是不能被程序验证的约束，在 Agent 执行中都可能漂移。

## 作者论证路径

文章先借 Anthropic、OpenAI 的实践说明 Agent 的四类失败模式：一步到位、过早宣布胜利、过早标记完成、冷启动困难。随后把问题落到企业级 Java 存量系统：十万行代码、RPC、流程编排、配置中心、缓存、数据库中间件混杂，隐性知识大量散落在团队经验里。

在问题建立后，作者给出自己的 `.harness/` 目录实践，从 Agent 角色定义、分层上下文、十阶段流程、Skill 体系、变更审计逐层展开。最后用 Dry Run 发现的缺陷、Agent-to-Agent Review 的拦截案例、AI 代码率从 24.86% 到 90.54% 的数据，证明 Harness 的价值在质量可控前提下提升了 AI 产出占比。

## 对开发者认知的改变

普通开发者容易把 AI 编程理解为“写好提示词”。这篇文章把焦点移到环境设计：文档、规范、测试、CI、部署、审计都必须变成 Agent 可读取、可执行、可验证的系统资产。开发者的工作从直接写代码，转向设计 Agent 的工作环境。

文章也改变了对 Bug 的处理方式。传统做法是修当前代码；Harness 视角则要求修系统：为什么 Agent 会犯这个错？需要增加规则、评审清单、测试门禁还是流程回退？每个错误都应该转化成 Harness 的一次迭代。

另一个重要认知是：文档不再只是给新人看的参考资料，而是 Agent 的认知边界。没有进入仓库、不能被索引、不能按需加载的知识，对 Agent 来说就等于不存在。

## 潜在局限或适用边界

这套体系前期投入明显，作者构建 Harness 约耗时一周，且后续还要持续维护 Rules、Skills、评审模板和变更记录。它更适合长期维护、业务复杂、返工成本高的存量系统；对一次性脚本、探索性原型或极小需求，完整 10 阶段流程可能过重。

文章展示的 AI 代码率提升很有启发，但它仍是特定项目、特定团队、特定工具链下的数据。AI 行占比不等于整体交付效率，是否真正提效还要结合需求周期、返工率、线上缺陷率、Review 时间等指标判断。

此外，Harness 质量取决于门禁是否真的机械化、文档是否持续更新、评审 Agent 是否有独立视角。如果规则写得模糊、Wiki 过期、测试只做形式覆盖，Harness 会变成新的仪式化负担。

## 实践中的指导意义

落地时可以从最小 Harness 开始：先把项目不可违背的规则写成 `rules/`，把高频需求流程写成 1-2 个 `skills/`，为每个需求建立 `changes/` 审计目录，再把 CI 和测试门禁改成可程序检查的条件。不要一开始追求完整平台，先让一次真实需求跑通闭环。

团队应优先把“历史踩坑”转成规则。例如价格字段类型、外部调用超时、配置中心约束、流程编排边界、测试用例数非零等，都应从口头经验变成 Agent 必须遵守的文件或脚本。每次 Agent 犯错后，不只修代码，还要追问 Harness 少了哪条约束。

对复杂任务，应把执行 Agent 和评判 Agent 分开：编码者负责产出，评审者按独立清单检查计划、实现、测试和风险。人类 Review 的重点也应前移，从逐行挑错转为确认 Spec、门禁和例外处理是否可靠。
