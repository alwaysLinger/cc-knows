# AI 编程的工作台迁移：从单 Agent 对话到可观测编排

## 多角度总结

### 核心论点

文章认为，AI Coding 的主要矛盾正在从“人和一个 Agent 如何对话”转向“人如何同时管理一组 Agent”。当模型和 Coding Agent 能承担更长任务链路后，人的时间不应继续消耗在等待单个 Agent 返回、逐轮追问和手工检查里，而应转向拆解任务、定义边界、观察执行、Review 结果和做关键决策。

Mexus 是作者对这种新范式的设计实践。它不是另一个 AI IDE，而是面向多 Agent 并行开发的 WebUI 控制台：左侧管理多个 CLI Agent Pane，中间展示 Agent 活动和代码 Diff，右侧展示文件树。它试图把开发者的主操作面从代码编辑器迁移到 Review 和编排界面。

### 技术/架构拆解

Mexus 的第一层设计是多 Agent 执行容器。每个 Pane 可以绑定自己的工作目录和任务说明，支持 Claude Code、Codex、OpenCode、Kimi Code CLI 等不同 Agent。不同任务可以分配给不同模型和工具，核心模块用强 Agent，测试、文档、低风险任务用更轻量的 Agent，从而在能力、成本和限额之间做组合。

第二层是以 Spec 为起点的编排。系统不会直接把用户需求拆给多个 Agent，而是先由规划 Agent 生成结构化 spec，包含目标、范围、验收标准、影响范围和任务拆解建议。用户批准 spec 后，再生成执行计划，明确每个 Agent 的任务、可修改路径、依赖关系和执行模式。这里至少有两个关键人工确认点：批准 spec，批准执行计划。

第三层是共享工作区下的协作机制。Mexus 默认共享工作区，而不是默认 git worktree 隔离。为了降低共享风险，每个 Agent 会拿到 `allowedPaths`，系统维护轻量文件 claim，标记谁正在编辑哪些文件；Observer Agent 持续观察冲突、越界修改和停滞任务，轻量问题可提醒 Agent 避让，重大风险上报给人。claim 不是强锁，而是把协作事实暴露出来。

第四层是可观测性。活动面板提供 Agent 仪表盘、模块视图、依赖视图和冲突面板，帮助人快速判断“现在有没有问题、问题在哪、是否需要介入”。更关键的是，这些数据不是只给 UI 使用，而是统一沉淀为结构化环境事实层，未来 Observer Agent 也消费同一份状态做运行时协调。

第五层是 Review 中心化。多 Agent 并行时，Review 不再只是最后验收，而是持续控制执行方向的主界面。开发者可以在 Diff 中直接给对应 Pane 反馈，让 Agent 继续修改，形成“观察 -> Review -> 反馈 -> 再执行”的闭环。

### 作者论证路径

作者先描述单 Agent 协作的低效：人被困在等待、查看、确认、继续的链路里。随后说明自己尝试多聊天窗口并行后，发现现有 AI IDE 和 OpenCode Web 模式接近但仍偏单 Agent 工作流，于是提出 Mexus。

接着文章围绕四个问题展开：如何让多个 Agent 有序协作，如何观测多个 Agent 的工作过程，如何让 Review 成为工作流中心，如何把不同 Agent 放到同一个控制台管理。之后回应“为什么重复造轮子”和“为什么默认共享工作区而不是 worktree”，最后回到 Harness Engineering：人类不再主要写代码，而是定义环境、意图、约束和反馈循环。

### 对开发者认知的改变

文章把开发者角色从“AI 的聊天操作者”提升为“Agent 工作环境的设计者”。当 Agent 能执行越来越多任务时，人类的价值不是继续逐行写代码，而是判断什么值得生产、怎样拆分任务、哪些边界不能越过、什么时候介入。

第二个认知变化是：Review 会从终点动作变成主操作面。过去 Review 是 Agent 做完之后的验收；多 Agent 并行后，Review 是持续调度和质量控制的界面。

第三个认知变化是：可观测性不只是给人看的仪表盘，也是未来 Agent 自协调的基础设施。如果 UI 和 Observer Agent 使用同一套事实层，人和 Agent 才能基于同一状态做判断，避免各看各的。

### 潜在局限或适用边界

Mexus 的一些编排能力仍是设计中或未完全落地，尤其是 Observer Agent 的运行时仲裁、冲突协调和冻结控制。共享工作区虽然效率高，但软边界和 claim 不能替代强隔离；在高风险重构、实验性改动、跨模块大迁移中，git worktree 或独立分支仍更稳妥。

这种模式也对用户提出更高要求：必须能写清 spec、拆任务、判断依赖、理解 Diff 和冲突信号。对很小的单点修改，多 Agent 控制台可能比单 Agent 对话更重。企业场景还需要补齐权限、审计、任务归档、密钥隔离和团队协作等能力。

### 实践中的指导意义

要实践多 Agent AI Coding，不应先追求“同时开很多窗口”，而应先建立编排协议。可从四件事开始：所有任务先写 spec 和验收标准；每个 Agent 明确 `allowedPaths` 和成功条件；执行期间记录文件 claim、模块归属和依赖影响；把 Review 反馈直接路由回对应 Agent。

默认共享还是默认隔离，不应二选一。常规协作、接口联调和需要实时互相感知的任务可以共享工作区，但必须有路径边界、冲突观测和人工接管；高风险、探索性和需要独立验证的任务放到 worktree。真正的目标不是让 Agent 多跑，而是让并行执行过程可观察、可约束、可修正。
