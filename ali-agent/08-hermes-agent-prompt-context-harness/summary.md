## 多角度总结

### 核心论点

文章把 Hermes 的突破概括为“自进化”：它不只是能持久运行，还能通过外部 Skill 沉淀和内部 RL 训练两条路径持续变强。动态 Skill 解决即时经验复用，RL 训练把高质量轨迹和奖励反馈内化到模型权重中。Prompt、Context、Harness 三层设计则为这套自进化提供兼容、压缩、记忆、安全和运行保障。

### 技术/架构拆解

自进化第一层是动态 Skill：主 Agent 响应后，后台 review agent 对记忆、技能和综合改进做审查，必要时生成或更新 Skill。第二层是 RL 闭环：通过 `batch_runner.py`、`mini_swe_runner.py` 收集或合成 Agent 轨迹，转换为 ShareGPT 格式，过滤不完整推理，再用 `trajectory_compressor.py` 保护头尾、摘要中间，压缩到训练可用规模。`rl_cli.py` 将训练拆成环境发现、配置、测试、训练和评估。GRPO 通过同题多答案和规则奖励学习高分行为，奖励函数可借助工具做真实验证，而不只是文本匹配。

### 作者论证路径

作者先对比 OpenClaw/Claude Code：它们能执行长任务，但不会自动沉淀执行经验。随后提出 Hermes 的“内外双路径”：Skill 像记笔记，RL 像练内功。中段展开轨迹格式、批量合成、SWE 数据生成、轨迹压缩、GRPO 和奖励函数。之后回到三层工程框架：Prompt 层做模型差异补丁和生态迁移兼容；Context 层用比例阈值压缩、文件记忆、SQLite 对话持久化、外部记忆服务和 `@` 注入；Harness 层用 Hook、错误分类、自愈、子 Agent 隔离、插件和安全护栏保证生产可用。

### 对开发者认知的改变

文章改变了“自进化等于自动拿用户聊天记录训练”的误解。Hermes 的 RL 主要是蒸馏和领域强化：用强 Teacher 模型、Benchmark 或人工筛选数据训练更便宜、更快、更合规的小模型，而不是直接吃用户私密对话。另一个认知变化是，多模型 Agent 不能假设所有模型都会主动用工具；Prompt 需要按 Claude、GPT/Codex、Gemini/Gemma 的行为差异动态补丁。上下文也不只是压缩，还包括主动 `@file`、`@diff`、`@url` 预挂载，减少工具回合和 token 浪费。

### 潜在局限或适用边界

动态 Skill 适合快速沉淀可解释经验，但无法改变基座模型能力；RL 能内化能力，但需要数据质量、奖励设计、算力、评估和隐私治理。直接用用户历史对话训练存在敏感信息和质量噪声风险，文章也明确更推荐人工导入、Teacher 参考和质量把关。Hermes 的多层能力适合长期运行、任务重复、领域明确的团队；若只是轻量问答或短期编码辅助，完整 RL 闭环和插件生态可能过重。

### 实践中的指导意义

要复用 Hermes 思路，可以先落第一层：任务结束后自动抽取轨迹，生成可审计的 Skill，并允许后续 patch。等任务模式稳定后，再挑选高质量轨迹或 Benchmark，设计 3-5 个奖励函数，用小规模 RL 验证是否真的提升。Prompt 层要为不同模型写工具使用补丁；Context 层要从绝对 token 阈值改成按窗口比例触发，并区分运行时压缩和训练数据压缩；Harness 层则必须先做错误分类、子 Agent 权限隔离、安全扫描和 Hook 扩展点。
