# AGENTS.md 是 AI Coding 的项目地图，不是越写越厚的手册

## 核心论点

文章认为 AGENTS.md 的价值不是“多写一份说明”，而是把项目中原本散落在人脑、聊天记录和隐性约定里的知识，变成 AI Coding Agent 打开仓库即可读取的工作上下文。它应该用最小上下文成本提供最大项目理解，让 Agent 能做到“打开即理解，改完即验证”。

核心原则是“地图，而非手册”：AGENTS.md 只放项目全貌和硬性规则，详细架构、组件模式、参考项目、验证流程放到 `docs/`、脚本和源码中，由 Agent 按需读取。

## 技术/架构拆解

文章把 AGENTS.md 放进一套完整 harness 中，而不是孤立文件。仓库层面，通过脚本聚合或 monorepo 解决前后端上下文割裂；环境层面，把本地配置统一到 `~/.<project>_env`，再用启动脚本封装构建、重启和健康检查；验证层面，定义稳定的 curl 规范、临时文件传递、token 获取模板和日志排查路径。

规则执行层面，作者用分层依赖检查脚本把“禁止跨层依赖”变成可执行约束，并在错误信息里写清 WHAT、WHY、HOW，让人和 AI 都能直接修。上下文补充层面，通过 `reference-projects/` 引入闭源组件和相关项目源码，再用 `docs/design-docs/ref-*.md` 做导航，形成“文档是地图，源码是事实”的结构。

最终模板包括项目概述、快速命令、后端架构、前端架构、关键约定、本地开发及验证流程、质量检查、参考项目约定和文档导航，建议控制在 200 行以下。

## 作者论证路径

作者先解释 AGENTS.md 的标准化背景：从 CLAUDE.md、工具私有规则到 AGENTS.md 成为事实标准。然后从亲身项目痛点切入，包括前后端仓库分裂、私域组件不可见、项目规矩只在人脑里、Agent 不会启动和自测。

接下来文章逐项给出改造实践：仓库聚合、统一环境配置、验证闭环、自动化检查、引入参考项目。最后把这些经验压缩成 AGENTS.md 模板和实施建议，强调从 `/init` 或 harness-creator 起步，再用 bad case 持续迭代。

## 对开发者认知的改变

很多开发者以为让 AI 更懂项目的方法是把文档写得更全，但文章给出的方向相反：把 AGENTS.md 写短，把细节放到可导航的文档、脚本和源码里。AI 不需要一次读完所有内容，它需要知道关键规则和去哪里找细节。

另一个重要变化是：AI Coding 的效率瓶颈不只是模型是否会写代码，而是项目是否对 Agent 友好。能否启动、能否验证、能否查源码、能否理解私域组件、能否被自动检查纠错，决定了 Agent 能不能独立完成闭环。

## 潜在局限或适用边界

AGENTS.md 对已有规范、构建脚本和自动化检查较成熟的项目收益最大。若项目本身没有统一启动方式、测试缺失、架构边界混乱，单独写 AGENTS.md 只能缓解信息不对称，不能替代工程治理。

将参考项目源码引入仓库也有成本：仓库体积、submodule 管理、权限合规、CI/CD 干扰都需要处理。对于小项目或公开组件占主导的项目，维护少量文档链接可能比引入大量源码更划算。

## 实践中的指导意义

落地 AGENTS.md 可以从最小版本开始：前 10 行讲清项目是什么、技术栈和仓库结构；随后列出构建、启动、检查、测试命令；再写 5 到 10 条违反就会出问题的硬规则；最后给出文档导航和验证流程。超过 200 行的细节应拆到 `docs/`。

真正的效果来自配套设施：启动脚本让 Agent 能运行项目，Makefile 让检查命令稳定，lint 脚本让规则可执行，curl 或浏览器验证让任务闭环，参考项目和源码让私域知识可读取。每遇到一个 AI bad case，就判断该补全局规则、模块文档、脚本检查还是参考源码；这样 AGENTS.md 会从一次性文档变成持续进化的团队知识入口。
