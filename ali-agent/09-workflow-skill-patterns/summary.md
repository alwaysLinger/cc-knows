# 把 Skill 写成可路由、可执行、可验证的工作流单元

## 文章信息

- 原文标题：工作流的 Skill 怎么写？从 7 个顶级 Skill 中提炼的模式与最佳实践
- 链接：https://mp.weixin.qq.com/s/aoNwyY5ZkCRMkZirn1rElQ
- 公众号：阿里云开发者
- 发布时间：2026年4月27日 08:54

## 核心论点

文章把 Skill 从“给模型看的说明文档”重新定义为一种轻量的工作流组件：它不创造新工具，而是把稳定的操作知识、判断条件、流程约束和退出标准注入上下文，让 LLM 用已有工具完成任务。写好 Skill 的关键不是正文越全越好，而是让模型在正确时机加载、按正确流程行动、在关键边界上不偷懒。

最核心的判断是：`description` 决定 Skill 能否被正确触发，正文结构决定触发后能否稳定执行。一个高质量 Skill 应该像“路由条件 + 操作协议 + 验证清单”的组合，而不是泛泛的知识库。

## 技术/架构拆解

文章将 Skill 拆成三层：frontmatter 是低成本索引，负责被模型扫描和路由；`SKILL.md` 正文承载核心指令、流程、决策树和验证条件；`references/`、`resources/`、`scripts/` 等目录承载大体量细节，按需读取。这个结构本质上是上下文工程里的渐进式披露。

作者归纳了 5 种核心模式：线性流程适合部署、安装、迁移；决策树适合平台选型和问题诊断；循环迭代适合 TDD、审查、设计评审；接力棒循环用文件保存跨 session 状态；多阶段加检查点适合跨多天的复杂流程。额外的“思维框架”模式更像分析协议，用量化阈值、非目标约束和反幻觉规则控制模型如何思考。

几个机制尤其可复用：用 Good/Bad 对比降低误解，用借口反驳表提前封堵模型的自我合理化，用负面指令和安全默认值约束危险动作，用 checklist 作为退出条件，用文件协议把长期状态外化。

## 作者论证路径

文章先解释 Skill 的运行机制：模型通过 frontmatter 判断是否加载，加载后把 `SKILL.md` 作为上下文使用。随后聚焦 description，因为它是路由入口，也是常见失败点。再通过 7 个顶级 Skill 样本归纳出 5 类工作流模式和 1 类思维模式，分别说明适用场景、结构模板和有效技巧。

最后，作者从模式归纳转向可操作模板：给出通用写作技巧、知识组织的 token 预算、模式选择决策树和最小可用模板。这使文章不是停留在“有哪些好 Skill”，而是把经验压缩成可直接复用的设计方法。

## 对开发者认知的改变

普通开发者容易把 Skill 当成更长的 prompt，但文章强调它更接近一种可加载的工程协议。Skill 的重点不是展示知识，而是让 Agent 在任务中形成稳定行为：什么时候加载、加载后先做什么、怎么验证、遇到不确定性如何降级。

另一个认知变化是：Skill 的“短”不是偷工减料，而是架构能力。主文件应只放核心规则和导航，细节拆到 supporting files。越是大型知识域，越需要决策树和索引，而不是把所有内容塞进正文。

## 潜在局限或适用边界

文章主要面向具备文件系统和工具调用能力的 Coding Agent。如果运行环境不能让模型读取引用文件、执行脚本或保存外部状态，那么接力棒循环、多阶段编排等模式会打折。

Skill 也不适合承载所有知识。低频、一次性、变动快的内容放文档可能更合适；有强副作用的操作需要配合权限、确认和审计机制；跨很多任务的“万能 Skill”会削弱路由精度，反而让模型频繁误触发。

## 实践中的指导意义

写 Skill 时可以先问 4 个问题：触发条件是什么，任务是否有固定流程，是否需要循环验证，状态是否要跨 session 保存。答案基本决定了该用线性、决策树、迭代、接力棒还是多阶段模式。

落地时优先把 `description` 写成路由条件，而不是功能介绍；正文控制在核心流程和关键约束内；复杂细节放到 `references/`；危险操作设置安全默认值和明确禁止项；最后用 checklist 定义完成条件。这样写出的 Skill 才能从“模型可能会参考的文档”变成“模型大概率会遵守的工作流”。
