# 从写代码到控交付：Harness 才是企业 Agent 的分水岭

## 多角度总结

### 核心论点

文章认为，企业环境里 Agent 能否从玩具变成生产力，关键不在 Prompt 多漂亮，而在 Harness 是否足够扎实。Harness 是把非确定性模型接入确定性工程流水线的控制面，覆盖真相源、执行边界、能力路由、日志观测、验证回归和交接恢复。工程师的角色也因此从亲手实现每一行代码，迁移到定义目标、卡住边界、掌控节奏和验收结果。

### 技术/架构拆解

作者用两个轴定义 Agent 架构：执行流是静态预设还是动态自主，状态上下文是隐式内存还是显式外部。Harness 位于“模型给意图、外部系统管状态和校验”的象限。好的 Harness 包括最小真相源 Spec/Handoff、Evaluator 沙盒、Checkpoint 审批、Capability 路由、前置验证、日志回传和回归测试。Aegis 案例中，一个 Capability 被拆成专属 Prompt、确定性 Python 脚本和 Validator，让模型只做轻量路由，而不是在巨型 Prompt 中猜分支。

### 作者论证路径

文章先区分传统软件工程和 Harness Engineering：前者管确定性代码，后者管非确定性模型。随后用象限图界定边界，再指出“软约束 Prompt”“工具军火库”“暴力死循环重试”“重型文档流”等伪 Harness 或劣质 Harness。接着通过 Aegis 项目展示从读文档、目标收敛、Spec/Handoff、Capability、真实接口排障到测试前置的完整过程。附录进一步给出 8 阶段 SOP、三层目标、偏航信号和可直接复用的提示句式。

### 对开发者认知的改变

开发者常把 Agent 协作理解成“写更长 Prompt”或“让模型更自由”，但文章强调真正的能力来自把自由度放进轨道。模型不是资深同事，而是高速但不稳定的执行者；你可以少写代码，但不能放弃技术判断。真正的 Harness 使用者要知道什么时候放手、什么时候下潜：架构主线被污染、阶段目标漂移、运行日志暴露系统性异常、模型混淆阶段完成和全局完成时，都必须接管。

### 潜在局限或适用边界

Harness 的投入适合长链路、多人维护、可回归验证、需要跨天交接的企业工程。对一次性翻译、情感分类、短任务 Demo，重型控制面可能过度。文章强调的 SDD、Spec、Handoff 和 Checkpoint 也需要团队纪律，否则会退化成无人维护的文档流程。另一个边界是模型仍需强技术负责人验收，Harness 不能替代架构判断、权限治理和生产变更制度。

### 实践中的指导意义

团队从 0 到 1 落地时，应先搭轻量真相源，再做执行边界和能力目录，随后接入测试、日志、回归、Handoff，最后逐步释放自动化自由度。单轮协作不要把大需求整包丢给模型，而是要求它先复述目标、压成最小 spec、做 checkpoint、说明风险和验证方式，确认后再执行。验收时只接受测试、日志、接口回包和现场现象，而不是“我觉得好了”。

## 金句摘抄

- “传统软件工程管的是「确定性」”
- “Prompt 是指令，Harness 是约束。”
- “阶段完成不等于全局完成。”
