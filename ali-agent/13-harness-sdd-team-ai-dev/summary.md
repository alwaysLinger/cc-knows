# 从出码率幻觉到团队级 AI 研发闭环

## 文章信息

- 原文标题：告别“氛围编程”：基于 Harness 治理和 SDD 的团队级 AI 研发范式演进与实践
- 链接：https://mp.weixin.qq.com/s/-_IBJFuXpvoqMJxL9oaEJQ
- 公众号：阿里云开发者
- 发布时间：2026年5月7日 18:00

## 核心论点

文章最重要的判断是：AI 出码率提升不必然带来研发提效。作者所在团队从早期 53% AI 出码率提升到 80%-90%，但交付周期和开发者工作量没有同步下降，说明只优化“写代码”这个局部环节，会被需求澄清、方案设计、评审、联调、测试、上线等全链路成本抵消。

真正的解法是把 AI Coding 从个人技巧升级为团队级工程能力：用 SDD 把模糊需求转成清晰规范，用 Harness 把 AI 的执行过程放进知识、约束、反馈和人工监督构成的受控系统里。AI 不再是随口提示就生成代码的黑盒，而是按 Spec、计划、角色和门禁推进交付的研发基础设施。

## 技术/架构拆解

SDD 负责“做什么”。流程包括 Specify、Plan、Implement、Validate：先由人和 AI 共同澄清需求，形成包含数据模型、接口规范、验收标准的结构化 Spec；再由 AI 把 Spec 编译成技术方案和任务拆解；随后由 Agent 执行；最后用测试和人工 Review 验证结果。

Harness 负责“如何可控地做”。文章拆成四个支柱：上下文工程维护单一事实来源，架构约束用物理或语法手段防止越层访问，反馈回路让 Agent 在测试失败后读取日志并自修复，熵管理则把人类修复经验固化成新规则；人类监督负责模糊业务边界和最终决策。

实践上，作者基于 Qoder 组织了项目层、技术层、资产层三类知识库，并用顶层 `README.md` 作为索引。需求处理使用 Quest Spec 模式生成 `design.md`，通过 HITL 补齐隐性知识。执行阶段使用 Experts Mode，把大型任务拆给不同专家 Agent。部署阶段通过 Aone MCP 打通 CI/CD，使链路从 PRD 延伸到部署。

## 作者论证路径

作者先回顾团队去年基于 Prompt、Context、Repo Wiki、Memory、Rules 达到 53% 出码率的实践，再指出半年后虽然出码率可达 80%-90%，但提效并不明显。接着归纳三个根因：研发是全链路过程，存量应用中 Vibe Coding 风险很高，大型复杂需求超出单次 AI 对话能力边界。

在问题之后，作者引入 SDD 和 Harness 两个概念，并用 Qoder 的端到端实践拆解从知识库、PRD、专家团执行到部署的完整流程。最后给出未来方向：更智能的 Spec 生成、更强的 Agent Teams、更完善的知识管理。

## 对开发者认知的改变

文章首先打破“出码率即效率”的直觉。AI 写了更多代码，可能反而制造更多 Code Review、调试和返工成本。衡量 AI 研发的指标应从代码行数转向需求交付周期、返工率、缺陷率和跨环节等待时间。

其次，它把“氛围编程”的适用边界讲清楚：新项目、小脚本、原型可能可行，但存量系统有历史包袱、隐式依赖和线上风险，必须先有规范和验收标准。AI 不是不需要约束，而是越强越需要约束。

最后，开发者角色发生变化：不是被 AI 替代的编码者，而是需求澄清者、规范审核者、计划验收者和 Harness 维护者。开发者要学会定义边界、写可测试验收标准、审查 Agent 计划，而不是只学习更花哨的提示词。

## 潜在局限或适用边界

文章中的实践强依赖 Qoder、Quest、Experts Mode、Aone MCP 等工具生态，在其他团队落地时需要替换成对应的 Agent、CI/CD 和知识管理工具。其方法论可迁移，但演示链路不一定可直接复制。

SDD 与 Harness 都依赖高质量知识库。如果 README 索引过期、技术层文档不完整、资产层示例不可信，AI 会在错误上下文中稳定地产生错误结果。知识维护成本会成为长期负担。

此外，HITL 不是可省略环节。需求澄清、计划审核、部署确认、结果验收都需要人介入。对于业务边界含糊、组织协作薄弱或缺少自动化测试的团队，仅引入 AI 工具无法直接得到全链路提效。

## 实践中的指导意义

团队要先把指标从“AI 写了多少代码”改成“需求从提出到上线缩短了多少、返工减少了多少、Review 压力是否下降”。如果只看出码率，很容易得到虚假的繁荣。

落地可以从三个动作开始：建立顶层知识库索引，让 AI 知道该去哪里找项目、技术和资产信息；在每个较大需求前写 Spec，明确用户故事、接口、数据模型和可测试验收标准；把执行拆给不同角色 Agent，并在关键节点保留人工确认。

对于存量应用，任何 AI 生成都应进入工程治理轨道：先 Spec，后计划，再实现，最后测试和部署验证。越是核心接口、跨模块重构、前后端联动需求，越不能靠一次对话解决。
